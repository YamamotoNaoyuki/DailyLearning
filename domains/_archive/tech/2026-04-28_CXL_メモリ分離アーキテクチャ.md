# CXL (Compute Express Link) — メモリ・コンピュート分離の物理層
**日付**: 2026-04-28
**分野**: tech
**タグ**: #CXL #memory_pooling #disaggregation #PCIe #cache_coherence #AI_infrastructure

## 学んだこと

### 1. CXLとは何か — PCIeを「キャッシュコヒーレント」にする

CXL (Compute Express Link) は **PCIe の物理層 (PHY) を共有しつつ、その上に新しい3つのプロトコルを乗せた** 高速インターコネクト規格。2019年に Intel 主導で発足、現在 (2026年) は CXL 4.0 (2025年11月リリース) が最新で、対応ハードウェアは Sapphire Rapids/Granite Rapids 世代以降の Xeon、AMD Genoa/Turin の EPYC、ARM Neoverse v2 系で実装されている。3 つのプロトコルは:

- **CXL.io** — PCIe ブロックI/Oをそのまま流用 (デバイス発見、設定、DMA)
- **CXL.cache** — デバイスがホストのキャッシュにアクセスし**コヒーレントに参加**するプロトコル
- **CXL.mem** — ホストCPUがデバイス側メモリ (DDR や HBM) を**直接ロード/ストア命令で**読み書きするプロトコル

PCIe 5.0 物理層上の CXL 2.0 は 32 GT/s/lane だったが、**CXL 3.0 で PCIe 6.0 PHY に乗り換えて 64 GT/s/lane**、CXL 4.0 では **128 GT/s/lane** に倍増。x16 構成で双方向 256 GB/s に近づき、HBM3 の 819 GB/s には及ばないが DDR5 の 76 GB/s/channel を大きく超える。重要なのは速度より **「一貫性のある共有メモリ」がCPU の外に出た** ことそのものだ。

### 2. デバイスタイプ Type 1/2/3 — 何を共有するか

CXL ではデバイスを 3 つのタイプに分類する。これが**システム設計の意味論を決定する**。

- **Type 1**: アクセラレータ (NIC, GPU の一部) — `CXL.io + CXL.cache`。**自前メモリは持たないがホストのキャッシュ階層に参加**。Smart NIC が CPU と同じメモリビューでパケット解析するなど。
- **Type 2**: アクセラレータ + 固有メモリ (GPU, FPGA) — `CXL.io + CXL.cache + CXL.mem`。**両側のメモリが両側からコヒーレントに見える**。GPU の HBM が CPU から直接 ld/st 可能で、PCIe DMA + 明示同期の煩雑さが消える。
- **Type 3**: メモリ拡張デバイス (CXL Memory Module) — `CXL.io + CXL.mem`。**自前のキャッシュは持たない、純粋な「外付けRAM」**。DDR5 DIMM スロットの代わりに CXL ケーブルで TB 級メモリを増設する。

Type 3 が **メモリ・ディスアグリゲーション (memory disaggregation)** の物理基盤で、現在の CXL ハイパースケール展開の主役。

### 3. メモリプーリングと共有 — CXL 2.0 → 3.0 の本質的進化

**CXL 2.0** で導入された **memory pooling** は、複数ホストが共通の Type 3 デバイス群に接続し、**各ホストに排他的なパーティションが割り当てられる**仕組み。これだけでも巨大な意味がある: あるサーバが 256GB しか使っていなくて 1.5TB が遊んでいても、別のサーバの突発負荷に動的に再割当てできる。**Meta の Pond プロジェクト** (2023 ASPLOS) は実環境で **DRAM コストの7%削減** を実証した。

**CXL 3.0** は **memory sharing** を追加: **複数ホストが同じメモリ領域を同時に**読み書きでき、CXL fabric がコヒーレンスを維持する。これは MPI の共有メモリやハードウェア DSM (Distributed Shared Memory) の現代的復活で、**従来 RDMA + ソフトウェアコヒーレンスでやっていた分散共有メモリがハードウェア層に降りた**ことを意味する。

加えて 3.0 は **multi-level switch cascade** をサポートし、単一ファブリックで **最大 4096 デバイス/ポート**接続可能 (2.0 は16エンドポイント)。spine-leaf トポロジーでデータセンター規模のメモリ網を構成できる。

### 4. なぜ AI が CXL を必要とするか — メモリの壁

LLM 学習・推論の現場では「**メモリの壁 (memory wall)**」が深刻化している。具体的には:

- **モデルパラメータ**: 70B モデルで FP16 換算 140GB、これは H100 の 80GB HBM 単独では収まらない
- **KV キャッシュ**: 長コンテキスト推論ではバッチ全体で数百 GB に膨らむ
- **オプティマイザ状態**: AdamW では学習時にパラメータの3倍のメモリが必要

H100 の HBM3 は 80GB で頭打ち、HBM4 でも 192GB 程度。一方で **CXL Type 3 で TB 級 DDR5 をホストに接続**し、HBM をホットセット用、CXL DRAM をコールドセット用と階層化することで、**「全体のサイズ」と「アクセス速度」のトレードオフを物理的に設計可能**になる。

例えば **vLLM や DeepSpeed-FastGen の KV キャッシュオフロード**は、まず GPU HBM、次にホスト DRAM、次に CXL プールという 3 階層を想定している。**CXL は HBM の代替ではなく、HBM の「下に伸びる」階層**として位置付けられる (latency 250ns vs HBM 100ns、DDR5 80ns)。

### 5. レイテンシとプログラミングモデルの現実

CXL.mem のレイテンシは **ローカル DDR5 の 1.5–3倍** (約 150–300 ns vs 80 ns)。これは NUMA 遠リモートノードと同等で、**OS の NUMA サブシステムから「もう一段遠いノード」として見える**ように設計されている (Linux 5.18 以降)。具体的には:

- CXL メモリは **`numactl --hardware`** で別ノードとして表示
- **`mbind()` / `migrate_pages()`** で明示的に配置可能
- **DAMON (Data Access MONitor)** などのカーネル機能でホットページを自動的にローカル DRAM に migration、コールドページを CXL に push 可能

つまり**プログラマから見れば「遠い NUMA ノード」**であり、新しい API は不要だが、**ワーキングセット解析と階層配置がアプリケーション設計の中心問題に昇格**した。

## 気づき・洞察

### 「ハードウェアが分散システム化する」という転換
CXL 3.0 で複数ホストが共有メモリを同時に読み書きする事象は、**「データセンター内の複数物理サーバが、論理的には一つのキャッシュコヒーレントマシンになる」**ことを意味する。これは過去 30 年間 NUMA で「単一 OS イメージの中の不均一性」だった話を、**シャシーの境界を越えて**実装し直していることに等しい。1990年代の DSM (Distributed Shared Memory) 研究は性能で破綻したが、PCIe 6.0 の物理層と CXL fabric のコヒーレンスエンジンによって**ハードウェア層に降ろすことで再生した**。

### コンピュート/メモリの分離は「役割の分離」を強制する
従来サーバはCPUとメモリが**1:1で物理筐体に固定**されていたため、メモリ過剰のサーバとCPU過剰のサーバが混在しても再配分できなかった。CXL でこの拘束が外れると、**「CPU シャシー」「メモリシャシー」「アクセラレータシャシー」**の3 種類を独立して購入・更新する世界が来る。これは **クラウドの "right-sizing"** がハードウェアレベルで実現することを意味し、ハイパースケーラの BOM 構造を根本から変える。Meta が 7% の DRAM コスト削減を実証したのは始まりに過ぎない。

### コヒーレンスの「物理半径」が拡大する
キャッシュコヒーレンスは長らく **「同一マザーボード内」**の概念で、それを越えるとソフトウェア層で扱うしかなかった (RDMA + 明示同期、MPI、Spark など)。CXL 3.0 はこの**物理半径を「ラック内」に拡張**し、CXL 4.0 では「複数ラック」への拡張も視野に入る。コヒーレンスの拡大は **「分散合意」を必要とする領域を縮小する**——Raft や Paxos が必要だった範囲の一部が、ハードウェア coherence で代替可能になる。これは分散システム工学の前提条件を変える。

### レイテンシの段差が「アルゴリズムの設計空間」を作る
CPU レジスタ (1ns) → L1 (1ns) → L2 (3ns) → L3 (10ns) → DDR5 (80ns) → CXL.mem (200ns) → SSD (10μs) → HDD (10ms) という階層は、**桁が変わるごとに別のアルゴリズムが最適**になる。CXL の追加は **80ns と 10μs の間に「200ns 層」を挿入**したことを意味し、**従来「メモリかストレージか」だったデータ構造設計が「ローカル DRAM か CXL か」**という新軸を獲得した。LSM ツリーが SSD の登場で発明されたように、CXL のために設計された新しいデータ構造 (例: tiered memory-aware B-tree) が今後登場するはず。

## 他分野との接続

- **cs (LSMツリー、CPUキャッシュ、NUMA、MVCC)**: CPU キャッシュ階層と仮想メモリの設計判断が **CXL でメインメモリ自体に階層化**として再現される。MVCC の snapshot は CXL 共有メモリ上で実装すると分散版 MVCC になる
- **cs (Raftコンセンサス)**: ハードウェアコヒーレンスの物理半径拡大は、Raft が担当していた一部の合意問題を消す。**「合意のソフトウェア・ハードウェア境界」が再交渉**される
- **tech (Confidential Computing TEE)**: CXL でメモリが筐体外に出ると、**信頼境界の物理的設計**が変わる。TEE は CXL 越しのメモリも保護する必要があり、CXL 3.0 ではリンク暗号化 (IDE: Integrity and Data Encryption) が標準
- **tech (Durable Objects エッジアクター)**: 共有状態を物理的に近接させる Durable Objects と、メモリを物理的に分離する CXL は**逆方向の解**——状態の局所化 vs 計算の局所化
- **anatomy (筋膜のテンセグリティ)**: ハードウェアの「機能分離して通信で繋ぐ」設計は、筋膜が**機能ユニットを物理的に分離しつつ張力で繋ぐ**生体構造と類比的——分離と接続の両立
- **bread (湯種法と段階分離)**: 製パンで湯種を別工程で作って本生地に投入するように、CXL はメモリを「別工程で作る」アプローチ
