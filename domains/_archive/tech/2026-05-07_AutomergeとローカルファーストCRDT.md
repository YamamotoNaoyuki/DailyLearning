# Automergeとローカルファースト・CRDT — 「サーバ無しで同期する」データ構造の設計

**日付**: 2026-05-07
**分野**: tech
**タグ**: #CRDT #ローカルファースト #Automerge #分散システム #リアルタイムコラボ

## 学んだこと

### 「ローカルファースト」という設計思想

Ink & Switch が 2019 年に提唱した *Local-First Software* は、Google Docs 的な「クラウド常時接続前提のコラボ」へのアンチテーゼとして登場した。クラウドモデルは UX としては素晴らしいが、「ネットワークが切れた瞬間に何もできない」「サービス終了でデータが蒸発する」「ユーザーがデータを所有していない」という構造的弱点を抱える。ローカルファーストの 7 つの理想は: ①ローカルでの即応性、②マルチデバイス同期、③ネットワーク必須でない、④長期的な所有権（ベンダーロックイン排除）、⑤シームレスなコラボ、⑥セキュリティ・プライバシ、⑦ユーザーがデータの主権を持つ。これらを技術的に成立させる中核要素が CRDT (Conflict-free Replicated Data Type) であり、その商用品質の実装が Automerge である。

### Automerge の正体 — JSON ライクな CRDT

Automerge は「同時編集可能で自動マージできる JSON 風データ構造」を提供する Rust 製ライブラリで、JS には WebAssembly を介してバインドされる。内部的には複数の CRDT (RGA テキスト、LWW Map、最大値レジスタ等) を組み合わせて JSON のオブジェクト・配列・テキストをモデル化する。各操作は「アクター ID + Lamport タイムスタンプ」で順序付けられ、操作ログを互いに交換することで全レプリカが同じ最終状態に収束する (Strong Eventual Consistency)。重要なのは、衝突解決のルールが**データ型ごとに事前定義されている**こと: テキストは挿入位置を保つ RGA、マップは Last-Writer-Wins、リストは因果順序付き挿入。つまりアプリ開発者は「どう衝突解決するか」を毎回考えなくていい。

### 操作ログの圧縮と効率

CRDT の最大の弱点は「全操作履歴を保持するためデータ膨張する」こと。Automerge は独自の **column-oriented binary encoding** でこれを解決している: 同じカラム (操作タイプ、タイムスタンプ、参照位置等) をまとめて圧縮することで、JSON のテキスト編集ログが元テキストと同等オーダーのサイズに収まる。さらに同期プロトコルは **Bloom filter ベースの差分検出** を採用し、ピア間で「自分が知らない操作だけ」を効率的に転送する。結果として、数千万操作の文書でも実用的な速度で同期できる。

### automerge-repo — 「電池付き」ツールキット

Automerge 単体は純粋な CRDT データ構造で、ネットワークやストレージは関知しない。これを実用的な「リポジトリ抽象」にラップしたのが **automerge-repo** で、以下を統合する:
- **NetworkAdapter**: WebSocket、WebRTC、BroadcastChannel など、任意のトランスポートを差し替え可能
- **StorageAdapter**: IndexedDB、Filesystem、SQLite など永続化先を選択
- **DocHandle**: ドキュメントの reactive な参照、変更購読

サーバ実装は「ネットワーク+ストレージアダプタを別物にした同じコード」が動く。つまりサーバは「リレー兼バックアップ」の役割しか持たず、真実の源 (source of truth) はあくまで各ピア。これがローカルファーストの実装上の核心: サーバが落ちてもアプリは動く。

### 競合する設計判断 — Yjs との対比

CRDT 実装には Automerge の他に **Yjs** がある。Yjs は性能を最優先し操作ログを積極的に GC する設計、Automerge は「全履歴を保つことで時間軸スクラブやブランチ・マージが可能」という設計。Figma のようなリアルタイム描画は Yjs が向き、Notion のような「編集履歴を辿りたい」用途は Automerge が向く。これは「メモリ vs 機能」のトレードオフ判断。

## 気づき・洞察

ローカルファーストは技術問題というより**経済構造に対する反論**である。クラウド SaaS の経済性は「データを人質にとった月額課金」に依存するが、CRDT が真に普及するとユーザーは自分のデータを抱えたまま別アプリに乗り換えられる。これは Notion/Airtable/Figma の堀を破壊する可能性を持つ。だから大手 SaaS 企業はこの技術を採用したくない経済的インセンティブがある。一方で Linear や Reflect のようなスタートアップは「ローカルファーストでオフラインも速い」を差別化要因として戦っている。技術選択がビジネスモデル選択と直結している珍しい事例。

もう一つの洞察は、CRDT は「分散合意問題を回避するエレガントな方法」だということ。Paxos/Raft は「全員で一つの真実に合意する」アプローチだが、CRDT は「衝突しない代数構造を使えば合意は不要」という反転発想。両者は対立ではなく相補的: 強整合が必要な金融・在庫管理は Paxos、人間の編集は CRDT、というレイヤ分離が現実解。

そして昨日の HTMX (ハイパーメディア駆動) との対比も興味深い。HTMX はサーバを真実の源として「シンプルさ」を取り戻す方向、Automerge はサーバを補助役にして「分散性とオフライン性」を取り戻す方向。両者ともに React + REST + GraphQL の複雑性への異なる対案であり、「Web の複雑性疲れ」という時代精神を示している。

## 他分野との接続

CS 分野で扱った **Lamport 論理時計・HLC** は CRDT の基礎をなす。各操作にタイムスタンプを付ける必要があるが、物理時計は信頼できないので論理時計が必須。Automerge の操作 ID はまさに `(actor_id, counter)` という Lamport クロックそのもの。さらに **永続データ構造 (Path Copying)** との接続も深い: CRDT は本質的に「全履歴を持つ永続的木構造」であり、Path Copying と同じ原理 (構造共有) でメモリ効率を実現している。

音楽分野で扱った **アルヴォ・ペルトのティンティナブリ様式** との意外な共鳴もある。ペルトは「制約された音響素材内で、分岐と統合が自動的に成立する代数的構造」を作曲した。CRDT も「制約された操作集合内で、分岐と統合が自動的に成立する」構造であり、両者ともに **マージの可換性 (commutativity)** という同じ数学的性質を中核に置いている。芸術と分散システムが同じ抽象に到達した。
