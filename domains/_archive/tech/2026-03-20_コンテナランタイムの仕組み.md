# コンテナランタイムの仕組み — OCI、runc、containerd、そしてその先
**日付**: 2026-03-20
**分野**: テクノロジー
**タグ**: #コンテナ #OCI #runc #containerd #gVisor #KataContainers #Firecracker #セキュリティ #eBPF #WebAssembly

## 学んだこと

### 1. OCI（Open Container Initiative）— コンテナの「標準規格」

2015年、DockerとCoreOSの競争が激化する中で、Linux Foundationの下にOCIが設立された。「コンテナとは何か」を業界共通で定義するための取り組みであり、現在3つの仕様を策定している（いずれも2025年11月に最新版公開）。

**Runtime Specification（runtime-spec）**
「ファイルシステムバンドル」をどう実行するかを定義する。具体的には`config.json`の構造を規定し、コンテナのライフサイクル操作（create → start → stop → delete）を標準化する。Linuxの場合、名前空間・cgroups・seccomp・マウントポイント・ケイパビリティなどの設定がここに含まれる。

**Image Specification（image-spec）**
コンテナイメージのフォーマットを標準化する。レイヤー構造（各レイヤーはtar.gz）、マニフェスト（JSON形式でレイヤーの参照とメタデータを記述）、イメージインデックス（マルチアーキテクチャ対応）、設定（エントリポイント、環境変数、ワーキングディレクトリ等）を定義する。コンテンツアドレッサブル——SHA256ダイジェストで全てのコンテンツを一意に識別する。

**Distribution Specification（distribution-spec）**
コンテナイメージのpush/pullプロトコルを標準化する。レジストリAPIのエンドポイント、blobストレージ、マニフェスト取得の手順を定義する。Docker Registry V2 APIが事実上のベースとなっている。

この3仕様が「コンテナのUSB-C」として機能し、異なるランタイム・レジストリ・ツール間の相互運用性を保証している。

### 2. コンテナランタイムの階層構造

コンテナランタイムは「高レベル」と「低レベル」の2層に分かれる。これはOSにおけるシェルとカーネルの関係に似ている。

**高レベルランタイム（CRI準拠）**
Kubernetesのkubeletと対話し、イメージの管理・コンテナのライフサイクル管理・ネットワーク設定などを担当する。

- **containerd**: Docker社が2016年に開発し、2017年にCNCFに寄贈。現在CNCF卒業プロジェクト。Kubernetes・Docker Engine・クラウドプロバイダーで最も広く使われている。2025年にcontainerd 2.0がリリースされ、Transfer Service・Sandbox Service・NRI・CDIがデフォルト有効に。containerd 2.3（2026年4月）以降は4ヶ月サイクルでリリース
- **CRI-O**: Red Hat主導。Kubernetesに特化した軽量ランタイム。PodmanやBuildahと同じcontainersエコシステムに属する

**低レベルランタイム（OCI準拠）**
実際にLinuxカーネルの機能を使ってコンテナプロセスを隔離・実行する。

- **runc**: OCIリファレンス実装。Go製。Docker社が`libcontainer`を切り出して開発。最も広く使われている
- **crun**: Red Hat主導。C製。runcより高速かつ省メモリ。RHEL 9以降のデフォルトランタイム。Go特有の`fork/exec`問題（Goランタイムのスレッドモデルとclone(2)の相性）を回避
- **youki**: Rust製のOCIランタイム。コンテナ作成〜削除が198.4ms（runcの352.3msに対して約44%高速）。Rustの所有権モデルがシステムコール操作の安全性と親和的。CNCF Sandbox申請中

### 3. runcの内部動作 — Linuxカーネル機能の統合

runcの核心は、複数のLinuxカーネル機能を組み合わせてプロセスの隔離環境を構築することにある。

**名前空間（Namespaces）— 何が見えるかの制御**
OCI runtime-specは8つの名前空間を定義している：

| 名前空間 | 隔離対象 | 効果 |
|---------|---------|------|
| **pid** | プロセスID | コンテナ内のプロセスは他のコンテナ・ホストのプロセスが見えない |
| **net** | ネットワークスタック | 独立したネットワークインターフェース・ルーティングテーブル・iptables |
| **mnt** | マウントテーブル | 独立したファイルシステムツリー |
| **uts** | ホスト名・ドメイン名 | コンテナ固有のホスト名を設定可能 |
| **ipc** | System V IPC・POSIXメッセージキュー | プロセス間通信をコンテナ内に限定 |
| **user** | UID/GID | コンテナ内のroot（UID 0）をホスト上の非特権ユーザーにマッピング |
| **cgroup** | cgroupの可視性 | コンテナから見えるcgroupの階層を隔離 |
| **time** | システムクロック | 独立したクロック管理（比較的新しい追加） |

各名前空間は`path`で既存の名前空間を参照するか、新規に作成するかを選択できる。これによりPodのようなコンテナ間の名前空間共有が可能になる。

**cgroups — どれだけ使えるかの制御**
リソースの割り当てと制限を担当する。cgroup v2が主流となり、以下のリソースコントローラーを管理する：

- メモリ: ハードリミット・ソフトリミット・スワップ制限・カーネルメモリ
- CPU: shares（相対ウェイト）・quota（絶対制限）・burst（バースト許可）
- Block I/O: デバイスごとの帯域幅スロットリング
- PID: プロセス数の上限（fork bomb防止）
- RDMA: リモートDMAリソースの管理
- デバイスアクセス: allowlistでデバイスタイプ・メジャー/マイナー番号・権限（read/write/mknod）を制御

**seccomp — 何ができるかの制御**
Secure Computing Modeにより、コンテナプロセスが発行できるシステムコールをフィルタリングする。BPFフィルターで許可/拒否/トレースを設定し、カーネルの攻撃対象面を大幅に削減する。containerd 2.0では`io_uring_*`システムコールがデフォルトseccompプロファイルから削除された（セキュリティ上の理由）。

**pivot_root — ファイルシステムの切り替え**
runcはchrootではなく`pivot_root(2)`を使用する。これは現在のルートファイルシステムを新しいルートに「スワップ」する操作で、chrootよりも強力な隔離を提供する。古いルートはアンマウントされるため、ホストファイルシステムへのエスケープが困難になる。

**ケイパビリティ（Capabilities）**
Linux capabilitiesにより、root権限を細分化して必要最小限の特権のみをコンテナに付与する。`CAP_NET_BIND_SERVICE`（1024番以下のポートバインド）や`CAP_SYS_PTRACE`（プロセストレース）など、約40の個別ケイパビリティがある。

### 4. containerdアーキテクチャ — スマートクライアント設計

containerdは「スマートクライアントアーキテクチャ」を採用している。デーモン自体はコンテナを直接起動せず、上位のマネージャーとして各コンポーネントを調整する。

**Shimモデル — プロセス分離の核心**
containerdはコンテナプロセスを直接管理しない。代わりに「shim」と呼ばれる中間プロセスを介する。shimの役割は：

1. ファイルシステムのrootfsへのマウント
2. 低レベルランタイム（runc等）の呼び出し
3. コンテナのstdio管理
4. コンテナプロセスの監視（containerdデーモンが再起動してもコンテナは生き続ける）

Runtime v2 shimはファーストクラスのAPIを提供し、ランタイム作者がcontainerdと統合できる。containerd 2.0ではRuntime V1/Runc V1 shimが完全に削除され、`io.containerd.runc.v2`への移行が必須となった。

**Snapshotter — レイヤーの実体化**
イメージのレイヤーをディスク上のファイルシステムに展開する。デフォルトはoverlayfs。コンテンツストアからレイヤーを取り出し、スナップショッターがそれを重ね合わせてコンテナのrootfsを構築する。リモートスナップショッター（例：stargz、nydus）を使えば、イメージ全体をダウンロードする前にコンテナを起動できる（lazy pulling）。

**Content Store — コンテンツアドレッサブルストレージ**
コンテナイメージとその全コンテンツをローカルに保持する。OCIレジストリからのダウンロードが主要な取り込み経路だが、直接ロードも可能。SHA256ダイジェストによるコンテンツアドレッシングで重複排除を実現。

**Transfer Service（containerd 2.0で安定化）**
任意のソースと宛先の間でアーティファクトオブジェクトを転送するシンプルなインターフェース。pull/push/import/exportを統一的に扱う。

**Sandbox Service（containerd 2.0で安定化）**
Pod・VMなどのマルチコンテナ環境のshim管理を拡張。Update APIで作成後の仕様・ランタイム・拡張・ラベルの変更が可能。CRIプラグインはレガシーCRIサーバーからSandbox Controller実装に移行した。

**NRI（Node Resource Interface）とCDI（Container Device Interface）**
containerd 2.0でデフォルト有効。NRIはドメイン固有のランタイムロジックをプラグインとして注入でき、CDIはベンダーデバイス（GPU等）のアクセス仕様を標準化する。

### 5. Dockerの進化 — モノリスから分散へ

Dockerの歴史はコンテナ技術の標準化の歴史そのものである。

**Phase 1: モノリシックDocker Engine**
初期のDocker Engineは一枚岩で、イメージビルド・レジストリ通信・コンテナ実行のすべてを単一バイナリで担っていた。

**Phase 2: containerdの抽出（2016-2017）**
Docker社はコンテナランタイム部分を`containerd`として切り出し、2017年にCNCFに寄贈した。同時に低レベルランタイムを`runc`としてOCIに寄贈。Docker Engineはcontainerdの「クライアント」となった。

**Phase 3: Kubernetes CRI統合**
KubernetesがCRI（Container Runtime Interface）を導入したことで、kubeletがコンテナランタイムとプラグイン方式で対話できるようになった。

- **containerd 1.0**: cri-containerdという外部デーモンがkubeletとcontainerdの間を仲介。gRPCホップが追加されるオーバーヘッドがあった
- **containerd 1.1**: CRI機能がcontainerdのプラグインとしてビルトイン化。直接関数呼び出しで効率向上
- **Kubernetes 1.24（2022年）**: dockershim完全削除。Docker Engineを直接Kubernetesのランタイムとして使う方法は廃止。containerdまたはCRI-Oを直接使用する形に

この進化は「コンポーネントの分離と標準化が相互運用性を生む」というクラウドネイティブの基本原則を体現している。

### 6. 代替ランタイム — 隔離の強化

標準的なruncはLinux名前空間とcgroupsで隔離を提供するが、カーネルはホストと共有している。これは攻撃対象面としてのカーネルがマルチテナント環境ではリスクとなる。

**gVisor（Google、runsc）— ユーザー空間カーネル**
- **アーキテクチャ**: Sentry（ユーザー空間Linuxカーネル再実装、Go製）+ Gofer（ファイルシステムプロキシ、9Pプロトコルで通信）+ runsc（OCI互換ラッパー）
- **動作原理**: アプリケーションのシステムコールをSystrap（seccomp `SECCOMP_RET_TRAP`）で傍受し、Sentryがユーザー空間で処理。ホストカーネルへの直接システムコールを遮断
- **Sentryの自己隔離**: Sentry自身もseccompフィルター・名前空間・cgroups・pivot_rootで制限されている。`exec(2)`や`connect(2)`すら発行できない
- **メモリ安全性**: GoのメモリセーフティがLinuxカーネル（C製）のメモリ安全性問題を本質的に回避
- **性能**: コンテナ起動はミリ秒単位。ハードウェア仮想化不要でVMと同等のリソース効率
- **用途**: Google Cloud Run、GKE Sandboxで本番運用。2025年にはGPU/CUDAサポート追加でAI/MLワークロードの隔離にも対応
- **Falco統合**: ランタイムモニタリングエンジンとの統合が可能（ただし2026年にFalcoのgVisorサポートはレガシー化の方向）

**Kata Containers（OpenInfra Foundation）— 軽量VM**
- **アーキテクチャ**: 各コンテナ（またはPod）を軽量VMで実行。ハイパーバイザー（QEMU/Cloud-Hypervisor/Firecracker）でハードウェアレベルの隔離を提供
- **OCI互換**: containerd shim経由でKubernetesと透過的に統合
- **柔軟性**: 単一Kubernetesクラスター内でrunc（通常ワークロード）とKata+QEMU（高セキュリティ）とKata+Firecracker（高速起動）を混在運用可能
- **Agent Sandbox統合**: AIエージェントのワークロード隔離にKata Containersを活用する動きが2025年に活発化

**Firecracker（AWS）— マイクロVM**
- **設計**: Rust製のVMM（Virtual Machine Monitor）。Linux KVMを利用。最小限のデバイスモデル（virtio-net, virtio-block, virtio-vsock, シリアルコンソール, キーボードコントローラーの5デバイスのみ）
- **性能**: 起動125ms未満、メモリフットプリント5MiB未満、ホストあたり毎秒150マイクロVM作成可能
- **Jailer**: VMM自体を隔離する「第2の防衛線」。仮想化バリアが破られた場合のフォールバック
- **用途**: AWS Lambda・AWS Fargateの基盤技術
- **Kata統合**: KataのバックエンドハイパーバイザーとしてFirecrackerを選択可能

**隔離強度の比較**:
```
弱 ←——————————————————————————————→ 強
runc(名前空間)  < gVisor(ユーザー空間カーネル) < Kata/Firecracker(ハードウェア仮想化)
高速・軽量         中間                            最も強力な隔離
```

### 7. セキュリティモデル — ルートレスコンテナとユーザー名前空間

**ルートレスコンテナ**
非特権ユーザーがコンテナの作成・実行・管理を行える仕組み。Dockerデーモンとコンテナプロセスがすべてユーザー名前空間内で動作する。

**ユーザー名前空間の仕組み**
- コンテナ内のroot（UID 0）をホスト上の非特権ユーザー（例: UID 100000）にマッピング
- コンテナ内では`CAP_SYS_ADMIN`や`CAP_NET_ADMIN`などのケイパビリティを持つが、これらはユーザー名前空間内でのみ有効。ホスト上の他ユーザーのファイルやプロセスには一切干渉できない
- カーネルがユーザー名前空間付きコンテナに対して追加の制限を適用するため、コンテナブレイクアウトに対する最も効果的なハードニングメカニズムの一つ

**Kubernetes 1.30でのユーザー名前空間サポート**
2024年にKubernetes 1.30でユーザー名前空間サポートが導入された。ホストレベルでは非特権でありながら、コンテナ内ではroot権限を持つように見えるPodの実行が可能に。

**ルートレス時のネットワーク**
ルートレスコンテナはネットワーク名前空間の作成にroot権限が必要という問題がある。`slirp4netns`がユーザーモードネットワーキングを提供してこれを解決する。`RootlessKit`が名前空間設定のユーティリティとして機能する。

**containerd 2.0のCRI強化**
ユーザー名前空間サポート、再帰的読み取り専用マウント、非特権ポート/ICMPアクセスがデフォルト有効。runc 1.2.0以降が必要。

**CVE-2025-31133 / CVE-2025-52565**
2025年にruncに「masked path」の悪用によるコンテナエスケープ脆弱性が報告された。マウントレースコンディションを突く攻撃で、`/dev/console`マウントに関連する脆弱性。これはコンテナセキュリティが依然として活発な攻防の場であることを示している。

### 8. 過去のトピックとの接続 — eBPFとWebAssembly

**eBPFによるコンテナオブザーバビリティ（2026-03-18学習済み）**
コンテナランタイムの「外側」からカーネルレベルで観察する技術として、eBPFは不可欠な存在になっている。

- **Falco**: eBPFでsyscallを監視し、コンテナの異常行動をリアルタイム検出。2026年にはmodern_bpfドライバー（BTF/CO-RE対応）に統一される方向
- **Cilium/Hubble**: eBPFベースのネットワーク可視化。コンテナ間通信のフローをカーネルレベルで追跡
- **Tetragon**: カーネル内でランタイムポリシーをエンフォースメント。「検出」ではなく「防止」をeBPFで実現
- **Grafana Beyla/OBI**: eBPFによるゼロ計装オブザーバビリティ。コンテナに一切変更を加えずにHTTP/gRPCトレースを自動収集

eBPFとコンテナランタイムの関係は「演奏者（コンテナ）と音響エンジニア（eBPF）」のようなものだ——演奏者は自分の演奏に集中し、音響エンジニアが全体のサウンドを監視・最適化する。

**WebAssemblyとコンテナの共存（2026-03-17学習済み）**
containerdのサブプロジェクト`runwasi`が、WasmワークロードをKubernetesで実行するためのcontainerd shimを提供している。

- **runwasi**: containerd-wasm-shimの基盤ライブラリ。Wasmtime・WasmEdgeなどのWasmランタイムを直接呼び出し、runcのような低レベルランタイムを介さない
- **Docker+Wasm**: containerd shimを通じて、LinuxコンテナとWasmモジュールのハイブリッドデプロイが可能
- **SpinKube**: CNCF SandboxプロジェクトとしてKubernetes上でWasmワークロードを実行するフレームワーク

「置換」ではなく「共存」が方向性。コンテナのshimアーキテクチャが持つ拡張性が、Wasmという全く異なる実行モデルの統合を可能にしている。

## 気づき・洞察

**「標準化の連鎖」が生態系を形成する**
OCI → CRI → containerd shimという標準化の連鎖が、代替ランタイム（gVisor、Kata、Wasm）の百花繚乱を可能にしている。皮肉なことに、「何を標準化するか」を決めたことが「それ以外の部分での多様なイノベーション」を促進した。これはMCP/A2Aのプロトコル標準化が多様なエージェント実装を可能にする構図と完全にパラレルだ。

**「隔離の強度はトレードオフ」という基本原理**
runc → gVisor → Kata/Firecrackerと隔離が強くなるにつれ、起動時間・メモリオーバーヘッド・複雑性が増す。完璧な隔離は存在せず、ワークロードの信頼レベルに応じた選択が必要。同一Kubernetesクラスター内でruncとKataを混在運用する「ランタイムクラス」のアプローチは、この本質的トレードオフへの実践的回答。

**「モノリスの分解」は技術進化の普遍的パターン**
Docker Engineの進化（モノリス→containerd抽出→CRI標準化→dockershim削除）は、WebフレームワークやAPIゲートウェイの進化と同じパターンを辿っている。コンポーネントの分離と標準インターフェースの確立が、エコシステムの拡大を可能にする。

**runcの2025年脆弱性が教えること**
CVE-2025-31133/52565は、名前空間とseccompという強力な隔離メカニズムがあっても、レースコンディションのような微妙な問題がエスケープを可能にすることを示している。ユーザー名前空間とルートレスコンテナは「runc自体が非特権であること」により、バグの影響範囲を限定するという「深層防御」のアプローチ。

## 他分野との接続

- **解剖学**: コンテナランタイムの階層構造（高レベル/低レベル）は、神経系の階層構造（大脳皮質の意思決定 → 脊髄の反射）に類似。containerdが「戦略的判断」、runcが「実際の筋肉運動」を担当する
- **音楽**: オーケストラにおける指揮者（containerd）と各奏者（コンテナプロセス）の関係。shimは「コンサートマスター」——指揮者と奏者の間を仲介し、指揮者が不在でも奏者が演奏を続けられるようにする
- **ピアノ**: seccompのシステムコールフィルタリングは、ピアノのソステヌートペダルのような選択的制御——全ての音を響かせるのではなく、必要な音だけを選択的に持続させる
- **ゴルフ**: ランタイム選択はクラブ選択に似ている。runc（ドライバー：高速・軽量）、gVisor（アイアン：バランス型）、Kata（ウェッジ：精密制御）——コースの状況（セキュリティ要件）に応じて使い分ける
- **パン**: コンテナイメージのレイヤー構造はミルフィーユ生地の層構造に似ている。各レイヤーは独立して作成されるが、積み重ねることで完成した製品となる。コンテンツアドレッサブルな重複排除は、サワードウスターターの「同じ種から異なるパン」に通じる
