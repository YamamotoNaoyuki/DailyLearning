# eBPF（extended Berkeley Packet Filter）— カーネルのプログラマビリティ革命
**日付**: 2026-03-18
**分野**: テクノロジー
**タグ**: #eBPF #カーネル #オブザーバビリティ #ネットワーキング #セキュリティ #Cilium #クラウドネイティブ

## 学んだこと

### 1. eBPFとは何か — カーネル内プログラマブル仮想マシン

eBPF（extended Berkeley Packet Filter）は、Linuxカーネル内に埋め込まれた**プログラマブル仮想マシン**。カーネルのソースコードを変更したりカーネルモジュールをロードしたりすることなく、カーネル空間でサンドボックス化されたプログラムを実行できる。

元々はBSD（1992年）のパケットフィルタリング機構だったBPFが、2014年にAlexei Starovoitovによって汎用カーネルプログラミング基盤に拡張された。「extended」が付くが、もはや原型のパケットフィルタとは別物であり、eBPF.ioは「eBPFはもはや何かの略称ではない」と公式に述べている。

**なぜeBPFが革命的か**: 従来、カーネルの動作を変更するには（1）カーネルソース修正→リビルド、または（2）カーネルモジュール（LKM）のロードが必要だった。前者はアップストリーム反映に年単位かかり、後者はクラッシュリスクが高い。eBPFは**安全性を保証しながらカーネルを動的に拡張する第3の道**を開いた。

### 2. アーキテクチャ — 検証・JIT・フック

eBPFプログラムの実行パイプラインは3段階:

```
[ユーザ空間]                    [カーネル空間]
C/Rust → Clang/LLVM → BPFバイトコード → Verifier → JIT Compiler → ネイティブコード
                                            ↓                         ↓
                                        安全性検証              フックポイントにアタッチ
                                        （静的解析）           （kprobe, tracepoint, XDP, tc, LSM...）
```

#### Verifier（検証器）

eBPFの安全性の根幹。オープンソース界で最も洗練された静的解析エンジンの一つ。ロード時にバイトコードを全パス探索し、以下を保証する:

- **メモリ安全性**: スタック範囲外アクセス、不正なポインタ演算、未初期化メモリ読み取りを禁止
- **終了保証**: 無限ループの禁止（バックエッジの静的解析）。eBPFは**チューリング不完全**（意図的な設計）
- **ヘルパー関数引数検証**: カーネルヘルパー呼び出し時の引数型・範囲チェック
- **レジスタ状態追跡**: 全レジスタのスカラー値範囲を追跡し、危険な操作を事前排除

2024-2025年の進展として、**スカラー値追跡の改善**（スタック操作跨ぎの範囲解析精度向上）や、検証器自体の正当性検証研究（状態埋め込み技術）が進行中。eBPF Foundationは検証器改善に25万ドルの研究助成金を投入。

#### JITコンパイラ

検証通過後、BPFバイトコードはJIT（Just-In-Time）コンパイラによって**ネイティブ機械語に変換**される。これにより:

- インタプリタオーバーヘッドの排除
- **ネイティブ並みの実行速度**（カーネルモジュールと同等性能）
- x86-64、arm64、riscv64等の主要アーキテクチャに対応

2025年のeBPF Foundationの資金は、x86-64/arm64/riscv64のJITコンパイラの**セキュリティ監査**（命令エンコーディング、レジスタ割り当て、即値処理の正当性検証）と、JITコンパイル済みプログラムへの**KASAN（Kernel Address Sanitizer）有効化**に投じられた。

#### フックポイント

eBPFプログラムは多様なカーネルイベントにアタッチできる:

| フック | 用途 | 例 |
|--------|------|-----|
| **kprobe/kretprobe** | カーネル関数の入口/出口 | 関数呼び出しの計測 |
| **tracepoint** | 静的トレースポイント | syscall監視 |
| **XDP** | NICドライバ直後 | 超高速パケット処理 |
| **tc (traffic control)** | パケットスケジューリング | トラフィック制御 |
| **BPF LSM** | Linux Security Module | セキュリティポリシー適用 |
| **sched_ext** | CPUスケジューラ | カスタムスケジューリング |
| **HID BPF** | 入力デバイス | キーリマッピング |

#### BPFマップ — カーネル・ユーザ空間の共有データ構造

eBPFプログラムはスタックが512バイトに制限され、ヒープ割当が禁止されている。データの永続化や共有にはBPFマップを使う:

- **Hash Map / Array Map**: キーバリューストア
- **Ring Buffer**: 高効率なイベントストリーミング（perf array bufferより大幅に低オーバーヘッド — 64コアノードで7% vs 50%の差）
- **Per-CPU Map**: CPUごとの独立データ（ロックフリー）
- **LPM Trie**: 最長プレフィックスマッチ（ルーティング用）

### 3. カーネル新機能（2024-2025）

**BPF Token**（Linux 6.9）: 非特権eBPF使用を可能にする委任権限メカニズム。ユーザー名前空間とBPFファイルシステムインスタンスに紐付き、特権デーモンがコンテナに完全なroot権限なしでeBPF機能を制御付きで付与できる。

**BPF Arena**: カーネルeBPFプログラムとユーザ空間の共有メモリ領域。マップやperfイベントに頼らず、高スループットデータ交換を効率的に行える。

**BPF Exceptions**: 検証器が形式的に証明できない前提条件について、ランタイムで条件を定義し即座に終了させる仕組み。検証器の柔軟性を拡張。

**sched_ext**（Linux 6.12）: eBPFによるカスタムCPUスケジューリングポリシー。カーネルの再ビルドやフォークなしに、本番環境でスケジューリングアルゴリズムをテスト可能。

**HID BPF拡張**（Linux 6.11）: 入力デバイスフックの追加。カーネルレベルでカスタムイベントフィルタリングやリマッピングが可能に。

### 4. ユースケース① — オブザーバビリティ

eBPFのオブザーバビリティにおける最大の特徴は**ゼロ計装（zero-instrumentation）**。アプリケーションコードの変更なしに、カーネルレベルでメトリクス・トレース・ログを収集できる。

**bpftrace**: DTraceやSystemTapの後継となる高水準トレーシング言語。awk/Cに着想を得た構文で、ワンライナーでカーネルイベントを計測:

```
# プロセスごとのsyscall回数
bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }'

# read syscallのレイテンシ分布
bpftrace -e 'kprobe:vfs_read { @start[tid] = nsecs; }
             kretprobe:vfs_read /@start[tid]/ { @ns = hist(nsecs - @start[tid]); delete(@start[tid]); }'
```

バージョン0.21（2025年）でヒストグラム改善、エラー診断強化、libbpf統合の近代化。

**継続プロファイリング**: Parcaなどのツールがシステム全体のスタックサンプリングを本番環境で最小オーバーヘッドで実現。

**OpenTelemetry eBPF Instrumentation（OBI）**: Grafana Beylaから寄贈されたプロジェクト。2025年11月に最初のアルファリリース。**プロトコルレベル**（ライブラリレベルではなく）で動作し、全プログラミング言語・全ライブラリを単一コマンドでゼロ計装。HTTP/HTTPS、HTTP/2、gRPC、SQL、Redis、MongoDB、Kafka、GraphQL、Elasticsearchに対応。2026年の旗艦目標は安定版1.0リリース。

### 5. ユースケース② — ネットワーキング

**XDP（eXpress Data Path）**: NICドライバの最も早い段階でパケットを処理。カーネルネットワークスタック全体をバイパスし、**驚異的なパケット処理速度**を実現。Cloudflareは世界記録の3.8Tbps DDoS攻撃をXDPベースで自動緩和。秒速1,000万パケットのドロップレートを達成。

**Cilium — eBPFベースのKubernetesネットワーキング**: CNCFの卒業プロジェクトとして、2026年現在**本番Kubernetes環境の60%以上**がCiliumを使用（Azure CNI powered by Cilium、GKE Datapath V2含めると75%以上）。

Ciliumがiptablesを置き換えた理由:
- iptablesのO(n)ルール評価 → BPFマップによるO(1)ルックアップ
- Pod数増加時のパフォーマンス劣化が解消
- L3/L4に加えL7（HTTP、gRPC）レベルのネットワークポリシー

**Cilium 1.19**（2026年2月）の主要機能:
- **暗号化の厳格化**: IPsec/WireGuardのストリクトモード。暗号化されていないノード間トラフィックをドロップ。ベストエフォートから**ゼロトラスト要件**へ
- **Ztunnel統合**（ベータ）: サイドカープロキシなしで透過的なTCP暗号化・相互認証
- **Hubble強化**: IPオプションによるパケットトレーシング、暗号化ステータスでのフィルタリング、ドロップイベントに原因ネットワークポリシーのタグ付け
- **Multi Pool IPAM安定化**: 大規模・セグメント化されたアドレス空間のサポート

**クラウドプロバイダー採用**: AWS EKSがCiliumをデフォルトCNIとして採用。Azure AKSもAzure CNI with Cilium推進。Google GKE Datapath V2もCiliumベース。

### 6. ユースケース③ — セキュリティ

**Falco**: CNCFの卒業プロジェクト。eBPFでsyscallを監視し、異常な振る舞いをリアルタイム検知。最新ハードウェアで10,000+イベント/秒をCPU使用率5%未満で処理。コンテナエスケープ、不正なファイルアクセス、特権昇格などを検出。

**Tetragon**: Ciliumのセキュリティ拡張。高レベルのセキュリティポリシーをeBPFエンフォースメントコードに変換。カーネル内でリアルタイムにポリシーを適用し、悪意ある操作をsyscallレベルでブロック。

**BPF LSM（Linux Security Module）**: 動的なセキュリティポリシー適用。従来のLSM（SELinux、AppArmor）がカーネルビルド時に固定されるのに対し、BPF LSMはランタイムでプログラマブルにフックを追加できる。

**TraceeShark**: Wiresharkスタイルの分析UIをeBPFのsyscallキャプチャに適用。セキュリティインシデントの事後分析を効率化。

**SafeBPF**: ハードウェアメモリ保護キー（MPK）を活用した多層防御研究。eBPF検証器に加え、ランタイムでの追加安全性を提供。

### 7. 主要フレームワーク・ツール

| ツール | 種別 | 特徴 |
|--------|------|------|
| **libbpf** | ライブラリ | C言語の標準eBPFライブラリ。CO-RE対応。自動アタッチ機能 |
| **CO-RE** | フレームワーク | Compile Once – Run Everywhere。BTF情報でカーネルバージョン差異を吸収 |
| **bpftrace** | トレーシング | 高水準トレーシング言語。ワンライナーで即時調査 |
| **Cilium** | ネットワーク | K8s CNI。ネットワーキング+セキュリティ+オブザーバビリティ統合 |
| **Hubble** | 可観測性 | Ciliumのオブザーバビリティレイヤー。フローレベル可視化 |
| **Falco** | セキュリティ | ランタイム脅威検知 |
| **Tetragon** | セキュリティ | カーネルレベルセキュリティポリシー適用 |
| **Aya** | ライブラリ | RustネイティブのeBPFフレームワーク。メモリ安全性保証 |
| **Parca** | プロファイリング | 継続プロファイリング。本番環境対応 |

**CO-RE（Compile Once – Run Everywhere）**の成熟は特に重要。従来eBPFプログラムは対象カーネルのヘッダーに依存してコンパイルする必要があったが、BTF（BPF Type Format）情報を利用することで、一度コンパイルしたバイナリを異なるカーネルバージョンで実行可能に。ディストリビューションがカーネルBTF情報を標準出荷するようになり、CO-REの実用性が大幅に向上した。

**Aya（Rust）**の成熟も注目に値する。Rustの所有権システムとeBPFの安全性モデルの親和性が高く、メモリ安全なeBPFプログラムをイディオマティックなRustで記述できる。

### 8. eBPFのクラウドネイティブインフラへのインパクト

eBPFはKubernetesネットワーキングの**根本的なアーキテクチャ変革**をもたらした:

**Before（iptables時代）**:
```
Pod → veth → bridge → iptables（O(n)ルール評価）→ routing → 出力
```

**After（eBPF時代）**:
```
Pod → veth → eBPF（BPFマップO(1)ルックアップ）→ 出力
```

- iptablesのルール数がPod数に比例して増大する問題が根本解決
- NetworkPolicyの評価がマップベースの定数時間に
- kube-proxyの置換（Ciliumのkube-proxy replacement）

**LLMインフラでの新用途**: 2025-2026年、eBPFベースのテレメトリがLLMオブザーバビリティに拡張。トークン生成、モデルサービング、LLM APIを支えるネットワークパスの計測にeBPFが活用され始めている。Microsoft、Google、TikTokがCiliumで最大級のAIトレーニングクラスターのネットワーキングを運用。

**Alibaba Cloud**はeBPFによるアダプティブL7ロードバランシングでインフラコストを19%削減。

### 9. eBPF vs WebAssembly — サンドボックス化ランタイムの比較

| 観点 | eBPF | WebAssembly |
|------|------|-------------|
| **実行空間** | カーネル空間 | ユーザ空間 |
| **チューリング完全性** | 不完全（意図的） | 完全 |
| **安全性モデル** | 検証器（静的解析）| サンドボックス（ケイパビリティベース） |
| **スタック制限** | 512バイト | 制限なし（設定可能） |
| **ヒープ** | 禁止（BPFマップで代替） | 線形メモリ（成長可能） |
| **主な用途** | ネットワーク・セキュリティ・可観測性 | アプリケーション・エッジ・プラグイン |
| **ポータビリティ** | Linux中心（Windows実験中） | 全プラットフォーム |
| **性能特性** | カーネル直接実行（ゼロコピー） | ネイティブの80-95% |

**本質的な違い**: eBPFは**カーネルの拡張性**に特化し、WasmはPortable **アプリケーションランタイム**に特化。競合ではなく、動作する層が根本的に異なる。

**ハイブリッド研究**: eBPFでWebAssemblyランタイムのサンドボックスを強化する研究が進行中。eBPFプログラムがhostcallラッパーのセキュリティチェックを代替し、Wasmモジュールごとの細粒度ポリシーを実現するアプローチ。

**NIST（2025年）**: クラウドネイティブサービスメッシュにおけるセキュリティ強化としてWebAssemblyの統合を推奨する文書を公開。eBPFのカーネルレベルセキュリティとWasmのアプリケーションレベルサンドボックスの**多層防御**が方向性として示されている。

### 10. eBPFとOpenTelemetryの統合

OpenTelemetry eBPF Instrumentation（OBI）は、従来のOTel計装の課題を根本的に解決する:

| 計装方式 | 実装コスト | 言語制約 | アプリ変更 | 深さ |
|----------|-----------|---------|-----------|------|
| **手動（OTel SDK）** | 高 | 各言語SDK必要 | 必要 | 最深（ビジネスコンテキスト） |
| **自動（OTel Auto）** | 中 | 対応言語のみ | 軽微 | 中（ライブラリレベル） |
| **eBPF（OBI）** | 極低 | 言語非依存 | 不要 | 浅（プロトコルレベル） |

実践的なアプローチは**ハイブリッド**: OBIでベースラインの可視性を即座に確保し、重要サービスにはOTel SDKで手動計装を重ねてビジネスコンテキスト（ユーザID、注文番号等）を付加する。

OBIの制約として、分散トレーシングのサポートは言語により差がある。Go、Node.js、Python、NGINX、PHPでは良好だが、リアクティブフレームワーク、Java仮想スレッド、複雑なスレッドプールでは苦労する場面がある。

## 気づき・洞察

**「カーネルをプログラマブルにする」という発想の破壊力**。eBPFが革命的なのは、個別の機能（パケットフィルタ、トレーシング、セキュリティ）ではなく、**カーネルの拡張メカニズム自体を汎用化**したこと。これにより、ネットワーキング・オブザーバビリティ・セキュリティという本来別々のドメインが**単一の技術基盤**に統合された。Ciliumがネットワーキング+セキュリティ+オブザーバビリティを一つのプラットフォームで提供できるのは、eBPFがこの3つを同じ抽象化レイヤーで扱えるからだ。

**意図的なチューリング不完全性**が安全性の源泉。eBPFの検証器は「何でもできる」をあえて捨て、「安全に実行できることだけ許可する」設計。これは3/17に学んだWebAssemblyのケイパビリティベースセキュリティ（「明示的に許可されたことだけできる」）と思想的に通じるが、実装アプローチが対照的——eBPFは静的解析（事前証明）、Wasmはランタイムサンドボックス（実行時隔離）。

**iptables → eBPFの移行は「O(n)→O(1)」の典型**。Pod数1万の環境でiptablesルールが数万行に膨れ上がりパフォーマンスが崩壊する問題は、BPFマップという正しいデータ構造の選択で根本解決された。アルゴリズムとデータ構造の選択がアーキテクチャレベルの問題を解決する好例。

## 他分野との接続

- **OpenTelemetry（3/16学習）**: OBIはeBPFとOTelの直接的な統合。3/16で学んだ「自動計装でベースライン確保→手動計装でビジネスコンテキスト追加」のアプローチに、eBPFが**第3のゼロ計装レイヤー**を追加する。OTelの4番目のシグナルであるプロファイリングもeBPF（Parca等）が基盤技術
- **WebAssembly（3/17学習）**: サンドボックス化ランタイムとして比較対象。カーネル空間（eBPF）vs ユーザ空間（Wasm）、チューリング不完全 vs 完全、静的検証 vs ケイパビリティベースという対照的な設計哲学。両者のハイブリッド（eBPFでWasmサンドボックスを強化）が研究段階
- **ゴルフ**: 「コースマネジメント」の発想と通じる。eBPFはカーネル内の最適なフックポイント（XDP=ティーショット、tc=アプローチ、kprobe=パッティング）を選択し、最小限の処理で最大の効果を得る。過剰な処理（=力み）は検証器が拒否する
- **解剖学**: eBPFのフックポイントは人体の神経系に類似。XDP（皮膚の反射弓=即座の反応）、kprobe（深部受容器=詳細な情報収集）、BPF LSM（免疫系=異物検知と排除）
- **哲学**: 検証器の「安全性を事前に証明できるものだけ許可する」アプローチは、認識論的な保守主義——「知り得ることの範囲内でのみ行動する」という立場
