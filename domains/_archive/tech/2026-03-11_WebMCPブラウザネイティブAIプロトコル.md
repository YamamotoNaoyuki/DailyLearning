# WebMCP——ブラウザネイティブAIプロトコルの衝撃
**日付**: 2026-03-11
**分野**: テクノロジー
**タグ**: #WebMCP #MCP #ブラウザ #W3C #AIエージェント

## 学んだこと

### WebMCPとは何か

WebMCPは、AnthropicのMCP（Model Context Protocol）をブラウザに持ち込むW3C標準提案である。2026年2月10日にW3C Draft Community Group Reportとして公開され、GoogleとMicrosoftのエンジニアがW3C Web Machine Learning Community Groupで共同開発している。

従来のMCPがバックエンド（JSON-RPC）でAIエージェントとサーバーサイドツールを接続するのに対し、WebMCPは**ブラウザネイティブのJavaScript API** `navigator.modelContext` を提供し、ウェブサイトが構造化されたツールをAIエージェントに直接公開できるようにする。

### 解決する問題

従来のAIエージェントによるWeb操作は「スクリーンショット撮影→Vision Modelで解析→クリック位置推測」という脆弱で遅い方法に依存していた。WebMCPにより、ウェブサイトが「私にできること」を機械可読で宣言し、エージェントが直接APIを呼び出せる。**トークン効率が89%改善**する。

### 2つのAPI面

**宣言的API（Declarative）**: HTMLフォーム属性による定義。

```html
<form toolname="search" tooldescription="Search products" toolautosubmit>
  <input name="query" type="text" />
  <button type="submit">Search</button>
</form>
```

ブラウザがフォーム定義から自動的にツールスキーマを生成する。既存のHTMLフォームへの最小限の変更で対応可能。

**命令的API（Imperative）**: JavaScriptによる動的登録。

```javascript
navigator.modelContext.registerTool({
  name: "addToCart",
  description: "Add a product to the shopping cart",
  inputSchema: {
    type: "object",
    properties: {
      product_id: { type: "string" },
      quantity: { type: "number" }
    },
    required: ["product_id"]
  },
  execute: ({ product_id, quantity }) => {
    return { content: [{ type: "text", text: `Added ${quantity}x ${product_id}` }] };
  }
});
```

ツール管理メソッド: `registerTool()`, `unregisterTool()`, `provideContext()`, `clearContext()`。

### 3層プロトコルスタックの完成

WebMCPの登場により、AIプロトコルは3層構造に収斂しつつある：

| レイヤー | プロトコル | 軸 | トランスポート |
|---------|-----------|-----|-------------|
| バックエンド（ツール接続） | **MCP** (Anthropic) | エージェント↔ツール | JSON-RPC（サーバーサイド） |
| エージェント間 | **A2A** (Google) | エージェント↔エージェント | HTTP + JSON-RPC + SSE |
| フロントエンド（Web接続） | **WebMCP** (W3C) | エージェント↔ウェブサイト | postMessage（ブラウザネイティブ） |

これまでMCP（垂直）とA2A（水平）の「2軸体制」と理解してきたが、WebMCPの登場で**第3の軸**が加わった。MCPがサーバーサイドの「裏口」なら、WebMCPはブラウザの「正面玄関」である。

### 実装状況

- **Chrome 146 Canary**（2026年2月）でフラグ付き早期プレビューとして搭載。本番対応は2026年中盤〜後半の見通し
- 2025年12月にAnthropicがMCPをLinux Foundation AAIF（Agentic AI Foundation）に寄贈。WebMCPもAAIFの傘下で開発
- Microsoft Edgeも追随予定

### エコシステム

| プロジェクト | 特徴 |
|------------|------|
| Browser MCP (browsermcp.io) | MCP Server + Chrome拡張。VS Code/Claude/Cursorからブラウザ操作 |
| Chrome DevTools MCP | コーディングエージェントにDevTools全機能を提供 |
| MCP-B Extension | Chrome Web Store拡張。ユーザーのアクティブセッションでサイトAPIにアクセス |
| Chrome MCP Server | 20+ツール（スクリーンショット、ネットワーク監視、ブックマーク管理等） |
| webmcp-core (keak-ai) | 任意のWebサイトからWebMCPツール定義を自動生成 |

### セキュリティ

**同意アーキテクチャ**: 高リスク操作には**同意境界（Consent Boundary）**を設ける。ブラウザのC++層で実装されたネイティブ同意プロンプトで、JavaScriptからプログラム的に承認を回避できない。

**「致命的トライアド」問題**: W3Cレビューアーが指摘した、AIエージェントが複数の機密タブ（銀行+メール+ショッピング）に同時アクセスするシナリオ。対策としてドメインレベル隔離とハッシュ検証が提案されている。

**Agent Invocation Tracking**: `SubmitEvent.agentInvoked`フラグにより、サーバーサイドで人間の操作とエージェントの操作を区別できる。

**残る課題**:
- ユーザーがエージェントに付与するツール/権限の制御が不十分（現状はWebサイト定義のみ）
- プロンプトインジェクション（ツール説明文経由）、ツールシャドウイング、クロスタブデータリークの新攻撃面

## 気づき・洞察

WebMCPの登場は、AIプロトコル戦争の新章を開いた。これまでの「MCP vs A2A」の2軸整理が不十分だったことが明らかになった。

真に興味深いのは、WebMCPが**既存のWeb標準（CORS、CSP、同一オリジンポリシー）の上に構築される**点だ。これはMCP（新しいトランスポート層が必要）やA2A（新しいプロトコルが必要）とは根本的に異なるアプローチで、「ブラウザが既に持っているセキュリティモデルを再利用する」という実用主義がある。

宣言的API（HTMLフォーム属性）の存在は特に示唆的だ。既存のWebフォームに数個の属性を追加するだけでAIエージェントから操作可能になるという段階的移行パスは、MCP普及における「N×M→N+M」の成功と同じ戦略——**既存のインフラへの最小限の変更で最大の効果**を狙っている。

## 他分野との接続

- **哲学（和辻風土論）**: WebMCPの宣言的APIは「環境（ブラウザ）が存在（エージェント）の行動を規定する」構造。エージェントは訪れるWebサイトの「風土」に適応する
- **パン（オートリーズ）**: 既存のHTMLフォームにWebMCP属性を追加する段階的アプローチは、オートリーズの「既存の粉と水の結合を待つ」発想に似ている。急いで全てを一度に変えるのではなく、既存構造の自然な延長として機能を加える
- **ゴルフ（コースマネジメント）**: スクリーンショット方式→WebMCP方式への移行は、力任せのショット→戦略的なクラブ選択への移行に類似。効率的な手段を選ぶことで結果が劇的に改善する
