# SQLiteの設計哲学
**日付**: 2026-03-22
**分野**: テクノロジー
**タグ**: #SQLite #組み込みDB #設計哲学 #エッジコンピューティング #テスト戦略

## 学んだこと

### 「なぜサーバーが必要なのか？」——起源の問い

SQLiteは2000年、D. Richard Hippが米海軍のミサイル駆逐艦向けソフトウェアを開発していた際に生まれた。当時のシステムはInformixをバックエンドに使っていたが、データベースサーバーが停止するとアプリケーション全体が動作不能になるという問題を抱えていた。Hippの問いは根本的だった——「そもそもなぜサーバーが必要なのか？ディスクから直接読み書きすればいい。コンピュータがアプリケーションを実行できる程度に健全であれば、それ以上の依存関係は失敗の原因にしかならない」。

この問いがSQLiteの設計哲学の全てを決定づけた。サーバーレス・単一ファイル・ゼロコンフィギュレーションという三つの設計原則は、「依存関係の最小化」という一つの思想から導出される。

### サーバーレスアーキテクチャ——「ライブラリ」としてのデータベース

SQLiteは「データベースエンジン」であって「データベースサーバー」ではない。アプリケーションプロセス内にリンクされるCライブラリとして動作し、別プロセス・スレッド・マシンへの通信は一切発生しない。OSのファイルシステムAPIを通じて通常のディスクファイルを直接読み書きする。

この設計は、PostgreSQLやMySQLのようなクライアント/サーバー型RDBMSとは根本的に異なる。クライアント/サーバー型では、プロセス間通信（IPC）のオーバーヘッド、サーバー管理の運用コスト、ネットワーク障害という障害点が不可避的に存在する。SQLiteはこれらを全て排除した。代わりに失ったものは、同時書き込みの並行性と、複数マシンからのネットワーク越しアクセスである。

重要なのは、これが「制約」ではなく「設計判断」であるということだ。Hippは「SQLiteはPostgreSQLの代替ではない。fopen()の代替だ」と明言している。つまり、SQLiteが競合するのはRDBMSではなく、カスタムファイルフォーマットやfopen/fwrite/fcloseによるアドホックなファイル入出力なのだ。

### 単一ファイル——データベースの「原子性」

SQLiteデータベースは単一のクロスプラットフォームなディスクファイルである。テーブル、インデックス、トリガー、ビュー——全てが一つのファイルに格納される。この設計には深い意味がある。

ファイルの内部はページ単位（デフォルト4096バイト）で管理され、各ページがB-treeノードとして機能する。テーブルごとにTable B-tree（RowIDをキー、行データをペイロード）、インデックスごとにIndex B-tree（インデックスキーをキー、ペイロードなし）が独立に存在し、全てが同一ファイル内に共存する。ファイルの先頭100バイトにはヘッダがあり、ページサイズ、ファイルフォーマットバージョン、スキーマバージョン等が記録される。

単一ファイルであることの実用上の利点は絶大だ。データベースのバックアップはファイルコピーで完了する。メール添付で送れる。USBメモリで持ち運べる。`rsync`で同期できる。git管理すら（バイナリだが）可能だ。32ビット/64ビット、ビッグエンディアン/リトルエンディアンを跨いでポータブルである。そしてSQLite 3.0.0（2004年）で作成されたデータベースファイルは、2026年現在の最新版でもそのまま読み書きできる——20年以上の後方互換性が維持されている。

### ゼロコンフィギュレーション——「インストール」という概念の排除

SQLiteの使用にインストール手順は存在しない。サーバープロセスの起動・停止・設定は不要。管理者によるデータベースインスタンスの作成やアクセス権限の設定も不要。設定ファイルもない。`sqlite3.c`と`sqlite3.h`の2ファイルをプロジェクトに追加してコンパイルするだけで、完全なRDBMSがアプリケーションに組み込まれる（Amalgamation方式）。

### 世界で最も多く配備されたデータベース——1兆個のインスタンス

SQLiteは世界で最も広く配備されたデータベースエンジンである。40億台以上のスマートフォンそれぞれに数百のSQLiteデータベースが存在し、全世界で1兆個以上のSQLiteデータベースがアクティブに使われていると推定される。Android、iOS、macOS、Windows 10、主要ウェブブラウザ（Chrome、Firefox、Safari）、Python、PHP、Skypeなど、SQLiteを使わないコンピューティングデバイスを見つけるほうが困難だ。

これが可能になったのは、パブリックドメインライセンス（著作権放棄）という決断による。BSD/MITですらなく、完全なパブリックドメインとすることで、あらゆる組み込みシナリオでの法的障壁を完全に排除した。

### テスト哲学——コードの711倍のテストコード

SQLiteのテスト戦略は、ソフトウェアエンジニアリングにおける一つの極北である。約16万行の本体コード（C言語）に対し、テストコードは1億5千万行以上（ドキュメントによってはコード比711:1とされる）。テストは複数のハーネスで構成される。

**TH3（Test Harness #3）**: プロプライエタリなテストスイート。約76.9MBのCコードで50,362の独立テストケースを実装。100%のブランチカバレッジと100%のMC/DC（Modified Condition/Decision Coverage）を達成。MC/DCはNASAのフライトソフトウェア規格（DO-178B Level A）で要求されるカバレッジ基準であり、民間のデータベースエンジンがこの水準を達成しているのはSQLiteだけと言っていい。

**OOM（Out-of-Memory）テスト**: mallocの失敗をシミュレートするカスタムラッパーで、N回目のメモリ確保を意図的に失敗させる。これをNを1から順次増加させながら繰り返し、全てのメモリ確保失敗パスが正しくエラーハンドリングされることを検証する。

**I/Oエラーテスト**: ディスクI/Oの失敗を同様にシミュレートし、全てのI/Oエラーパスが安全に処理されることを保証する。

**クラッシュテスト**: トランザクション中の任意のタイミングでプロセスクラッシュをシミュレートし、データベースの整合性が維持されることを検証。

**ファジング**: AFL、OSS-Fuzz、dbsqlfuzzによる継続的ファジング。数十億のテストケースから「興味深い」ケースを抽出し、リグレッションテストに組み込む。

リリース前のソークテストでは約2億4850万のテストインスタンスが実行され、数十億のSQL文が評価される。この異常なまでのテスト投資が、航空宇宙・自動車・医療機器など、障害が人命に関わる領域でのSQLite採用を支えている。

### WAL（Write-Ahead Logging）モード——並行性の改善

SQLite 3.7.0（2010年）で導入されたWALモードは、デフォルトのロールバックジャーナルモードに対する重要な改善だった。ロールバックジャーナルモードでは、書き込み時にデータベース全体がロックされ、読み取りも書き込みもブロックされる。WALモードでは、変更をまずWALファイル（-walファイル）に書き込み、定期的にデータベース本体にチェックポイントする。

WALモードの利点は、読み取りと書き込みが互いをブロックしないことだ。複数の読み取りと単一の書き込みが並行して実行できる。ただし、書き込みは依然として単一プロセスに限定される——これは単一ファイルアーキテクチャの根本的な制約である。

### SQLiteをアプリケーションファイルフォーマットとして使う

sqlite.orgは、SQLiteをアプリケーションのネイティブファイルフォーマットとして使うことを積極的に推奨している。これは「pile-of-files」（XML/JSON/CSVなどの独自ファイル群）アプローチに対する代替案だ。

利点は明確だ。（1）差分更新——1バイトの変更に対してファイル全体の再書き込みが不要。変更されたページのみディスクに書かれる。（2）トランザクション——複数の変更をアトミックに実行でき、途中クラッシュしても整合性が保証される。（3）スキーマによる構造化——データの検証と制約を宣言的に記述できる。（4）クエリ言語——SQLで複雑な検索や集計が即座に実行可能。（5）長期保存——米国議会図書館がデジタルコンテンツの長期保存フォーマットとしてSQLiteを推奨している。

### JSONB——バイナリJSONの内部表現（2024年〜）

SQLite 3.45.0（2024年1月）で導入されたJSONBは、JSONのバイナリ内部表現である。従来のSQLiteのJSON関数はテキストJSON（RFC 8259）をパース→処理→テキストに戻すという3段階を踏んでいたが、JSONBでは中間表現（パースツリー）をそのままBLOBとしてディスクに保存できる。結果として、処理に必要なCPUサイクルが半分以下に削減され、ストレージサイズも5〜10%縮小される。

注意すべきは、PostgreSQLのJSONBとSQLiteのJSONBは名前こそ同じだが内部表現は全く異なるという点だ。SQLiteのJSONBは各要素がヘッダ（型＋サイズ）＋ペイロードで構成されるコンパクトなバイト配列であり、PostgreSQL互換ではない。

### WASM対応——ブラウザ内SQLite

SQLiteはWebAssemblyへのコンパイルを公式にサポートしており、ブラウザ内でSQLiteデータベースを操作できる。OPFS（Origin Private File System）をストレージバックエンドとして使用し、SQLite 3.47以降ではOPFS上でのWALモードも利用可能になった（ただし排他ロックモードが必須で、並行性サポートは失われる）。

64ビットWASMのサポートも追加され、大規模データベースのブラウザ内処理が現実的になりつつある。これはLocal-First Softwareのムーブメントと直接接続する。

### Litestream——WALベースの連続レプリケーション

Ben Johnson（元Bolt DB開発者）が開発したLitestreamは、SQLiteの弱点である「レプリケーション不在」を解決するツールだ。SQLiteのWALモードを利用し、WALページをS3やローカルファイルに連続的にストリーミングすることで、非同期レプリケーションを実現する。

技術的に巧妙なのは、LitestreamがSQLiteのチェックポイントプロセスを「乗っ取る」点だ。長時間の読み取りトランザクションを開始してWALファイルのチェックポイントを防止し、代わりに「シャドウWAL」というステージングエリアに新しいWALページをコピーしてからレプリカに転送する。

さらに、読み取り専用レプリカへのライブレプリケーション機能により、エッジでの地理的レプリケーションが可能になった。プライマリDBの変更がストリームとして複数のリードレプリカに配信され、ネットワークレイテンシの改善と読み取りスケーリングを同時に実現する。

### エッジデータベースとしてのSQLite

Fly.ioが提唱した「All-in on Server-Side SQLite」の概念は、SQLiteをエッジコンピューティングの中核に据えるものだ。従来、SQLiteはクライアントサイドの組み込みDBとして認識されてきたが、Litestream＋Fly.ioのエッジノード配置により、サーバーサイドでもSQLiteが有力な選択肢となった。

各エッジノードにSQLiteを配置し、ユーザーに最も近いノードで読み書きを完結させる。書き込みはLitestreamでS3にレプリケートされ、他のノードに伝播する。この「データベースをアプリケーションの隣に置く」アプローチは、クライアント/サーバー型RDBMSのネットワークレイテンシを根本的に排除する。

## 気づき・洞察

**「fopen()の代替」という位置付けの重要性**。SQLiteの設計判断は全て「RDBMSの代替」ではなく「ファイルI/Oの代替」として一貫している。この自己認識の明確さが、PostgreSQLとの不毛な比較を避け、実際の使用場面（スマートフォン、IoTデバイス、デスクトップアプリ）で圧倒的なシェアを獲得した理由だろう。

**テストコード比711:1の意味**。これはコスト度外視の品質追求ではなく、合理的な投資判断だ。SQLiteは1兆個以上のインスタンスで動作しており、バグ1件の影響範囲は想像を絶する。テストへの異常な投資は、このスケールでは正当化どころか必然である。MC/DCカバレッジの達成は、航空宇宙・医療グレードの信頼性を「民間ライブラリ」として実現した稀有な例だ。

**パブリックドメインという戦略的決断**。BSD/MITでもなくApache 2.0でもなく、著作権そのものを放棄するという判断は、最大限の普及を優先した設計哲学の帰結だ。ライセンスコンプライアンスの議論を完全に消滅させることで、あらゆる組み込みシナリオへの障壁を最小化した。

**「制約を制約として受け入れる」設計**。同時書き込みが単一プロセスに限定されることを「解決すべき問題」ではなく「トレードオフとして受け入れた設計判断」と位置付けることで、アーキテクチャ全体のシンプルさが保たれている。制約の受容がシンプルさを生み、シンプルさが信頼性を生むという連鎖だ。

## 他分野との接続

**cs（B木とデータベースインデックス）**: 先日学んだB木の知識がSQLiteの内部構造を理解する直接的な鍵になる。SQLiteのファイルフォーマットは本質的に「B-treeのページがフラットなディスクファイルに敷き詰められたもの」だ。Table B-tree（B+木変種、葉にデータ）とIndex B-tree（B木、葉にRowIDポインタ）の使い分けは、B木の理論が実装に落ちる典型例。ページサイズ4096バイトがOSのページサイズと一致するのも、CPUキャッシュ・メモリ階層の議論と直結する。

**bread（シンプルさという美徳）**: サワードウの基本は小麦粉・水・塩・酵母の4要素であり、これ以上の単純化は不可能だが、これ以上の複雑化も不要だ。SQLiteの「sqlite3.c + sqlite3.h = 完全なRDBMS」というAmalgamation方式は同じ哲学を体現している。必要十分な構成要素だけを残し、それ以外を徹底的に排除する。

**music（ソロ vs アンサンブル）**: クライアント/サーバー型RDBMSがオーケストラ（指揮者＝サーバー、楽器群＝クライアント、協調が本質）なら、SQLiteはソロ演奏だ。協調のオーバーヘッドがゼロになる代わりに、表現の多声性には限界がある。しかしバッハの無伴奏チェロ組曲が証明するように、ソロであっても深い表現は可能だ。

**golf（コースマネジメント）**: 「飛距離を最大化する」のではなく「ミスの影響を最小化する」というコースマネジメントの思想は、SQLiteの「機能を最大化する」のではなく「障害点を最小化する」という設計哲学と通底する。

**anatomy（身体の冗長性と単純性）**: 膝関節のACL損傷メカニズムで学んだように、複雑なシステムは障害モードも複雑になる。SQLiteのサーバーレス設計は、システムの構成要素を最小化することで障害モードを単純化している。

## 次に深掘りしたいこと

- **SQLiteのVFS（Virtual File System）層**——OS抽象化レイヤーの設計。カスタムVFSでの暗号化（SQLCipher）や分散ストレージ対応
- **SQLite内部のクエリプランナー**——NGQP（Next Generation Query Planner）のコストベース最適化アルゴリズム
- **libSQL（Turso）**——SQLiteのフォーク。SQLiteの設計制約（単一ライター）を崩さずにどこまで分散化できるか
- **cr-sqlite**——CRDTベースのSQLiteレプリケーション。先日学んだCRDTの知識との接続
- **SQLiteのロック機構**——5段階ロック（UNLOCKED→SHARED→RESERVED→PENDING→EXCLUSIVE）の設計意図
- **rqlite / dqlite**——Raft合意プロトコル上のSQLiteレプリケーション。分散システムとしてのSQLite
