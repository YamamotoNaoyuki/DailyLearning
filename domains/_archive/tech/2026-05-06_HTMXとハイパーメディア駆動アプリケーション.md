# HTMX とハイパーメディア駆動アプリケーション (HDA)
**日付**: 2026-05-06
**分野**: tech
**タグ**: #htmx #hypermedia #HATEOAS #web-architecture #REST #LoB #server-driven-ui

## 学んだこと

### 出発点：Roy Fielding の REST はもう死んでいる、という前提
Carson Gross が 2020 年に intercooler.js を書き直す形で公開した htmx は、たかだか 13 KB min.gz の JavaScript ライブラリだ。表面的には `hx-get`, `hx-post`, `hx-swap`, `hx-target`, `hx-trigger` といった HTML 属性を追加して AJAX を宣言的に書けるようにしただけに見える。しかし背後にあるのは、**業界が「REST API」と呼んでいるものの大半は Fielding の REST ではなく単なる JSON-RPC である**、という挑発的な主張である。

Fielding 自身が 2008 年に "I am getting frustrated by the number of people calling any HTTP-based interface a REST API" と書いた、あの有名な怒りの根本原因は HATEOAS（Hypermedia As The Engine of Application State）の喪失だ。htmx の哲学的核は、HTML はすでにハイパーメディアであり、HATEOAS はブラウザという完全な REST クライアントが何十年も実践してきた現役の制約である、という再発見にある。SPA はこの遺産を「JSON + クライアント状態」のために捨ててしまった、というのが Carson Gross の診断だ。

### Hypermedia-Driven Application (HDA) の定義
htmx 公式エッセイによると HDA は二つの制約で定義される。

1. **宣言的 HTML 埋め込みシンタックス**でフロントエンドのインタラクションを表現する（命令的スクリプトではなく）
2. **サーバとの通信はハイパーメディア（= HTML）**で行う（JSON ではなく）

つまり、サーバはアプリケーション状態の唯一のソースであり、状態遷移はサーバが返す HTML の中に「次に取りうるアクション」として埋め込まれる。クライアントはドメイン知識を持たない汎用ハイパーメディア処理エンジン（ブラウザ + htmx）として振る舞う。

```html
<!-- アクティブサーチの実装。これだけ -->
<input type="search" name="q"
       hx-get="/search"
       hx-trigger="input changed delay:500ms"
       hx-target="#results"
       hx-swap="innerHTML">
<div id="results"></div>
```

サーバは `/search?q=...` に対して JSON ではなく `<ul><li>...</li></ul>` のような HTML フラグメントを返す。クライアントには検索結果のスキーマも、ソート順のロジックも、ページネーションの有無も知る必要がない。すべてが返ってきた HTML の中で自己記述されている。

### Locality of Behaviour (LoB)：1990 年代の Separation of Concerns への反逆
Carson Gross が「LoB（行動の局所性）」として明示化した原則は、単なるショートカットではなく Separation of Concerns (SoC) との明示的な対立軸として提示されている。LoB は次のように述べる。

> **コードの単位の挙動は、その単位だけを見れば可能な限り明らかであるべき。**

jQuery 時代の「HTML / CSS / JS を分離せよ」という SoC 教義は、結果として `$('#submit-btn').on('click', ...)` のような **spooky action at a distance**（離れた所からの不気味な作用）を生んだ。HTML を見ても何が起こるかわからず、別ファイルの JS を grep して回ることになる。React/JSX が結局コンポーネント単位で HTML/JS/CSS を再統合したのは、SoC 教義から LoB への振り子の戻りと解釈できる。htmx はこの振り子をさらに HTML 属性まで押し戻した。

LoB と SoC は完全には両立しない。htmx 側もこれを認め、「両者はトレードオフであり、文脈依存の主観的判断が必要」と明言している。HDA は LoB に重みを置く立場を選んだ、というだけのことだ。

### SPA との根本対比
| 観点 | SPA (React/Vue) | HDA (htmx) |
|------|-----------------|------------|
| 状態の真実源 | クライアントの JS メモリ | サーバ |
| ワイヤフォーマット | JSON | HTML フラグメント |
| クライアント能力 | ドメイン知識を持つ厚い層 | 汎用ハイパーメディアエンジン |
| ルーティング | クライアント側ルータ | URL = サーバリソース |
| ビルドパイプライン | bundler / TS / webpack | 不要（CDN 1 ファイル） |
| バックエンド独立性 | API 契約で疎結合 | テンプレート言語と密結合 |
| HATEOAS | 放棄 | 保存 |

JSON API では「`POST /accounts/123/withdraw` は残高 < amount のときどう振る舞うか」をクライアントが知っている必要がある。HTML を返す HDA では、口座が当座貸越なら **サーバが `<form action="/withdraw">` を出力しない**だけでよい。クライアントは「画面にあるリンクをクリックする」「画面にあるフォームを送信する」以上のことを理解しない。これが Fielding が要求した「out-of-band な情報なしに状態遷移可能」の正しい姿である。

### 2026 年現在の生態系：HTMX 2.x と意図的なフィーチャーフリーズ
htmx の現行安定版は 2.0.x（2024 年中盤リリース）で、内部的には依然 `XMLHttpRequest` ベースだ。一方 htmx 4.0 が 2025 年末からアルファで進行中で、`fetch()` への全面移行と内部の整理を行うが、**新機能は基本的に追加されない**。これは事故ではなく方針である。

Carson Gross は "100 Year Web Services" というキーワードで、jQuery 的な長期安定を htmx の最大の価値命題と位置付けた。リリースは四半期定期、機能追加は extension API 経由、コアは凍結。これは React の「メジャーバージョンごとに mental model が変わる」エコシステムへの強烈なアンチテーゼだ。

類似技術との比較は重要だ。

- **Hotwire / Turbo (Rails)**: HTML over the wire の同類だが、Turbo Frames / Streams という独自概念に乗る、より opinionated な選択。22 KB と htmx より重い。Rails への寄り添いが強い。
- **Phoenix LiveView (Elixir)**: WebSocket 常時接続で、サーバ側に **ユーザごとの BEAM プロセス**が状態を保持する。htmx が「リクエスト毎に独立」のステートレス HTTP を貫くのに対し、LiveView は明確にステートフル。リアルタイム協調・push 型 UI に強いが BEAM/Elixir に縛られる。
- **Livewire (Laravel)**: 思想は LiveView 寄りだが PHP リクエストモデル上で疑似的に再現したもの。
- **Alpine AJAX**: 3 KB のミニマル版 htmx クローン。

htmx の差別化は **バックエンド非依存性**（Django/Rails/Laravel/Go/Phoenix どれでも動く）と **ステートレスへの徹底**にある。LiveView のような persistent process は強力だが運用コストが上がる。

### 限界と批判
HDA は万能ではない。htmx 側も明示的に向き不向きを論じている。

- **オフライン / 低レイテンシ要件**: ネットワーク往復前提なので、Figma のようなリッチエディタや、オフラインファースト PWA には不利。
- **複雑な双方向状態**: 複数コンポーネントが同じクライアント状態を読む（カート/フィルタ/メイン領域が連動）パターンは、サーバ往復で書くと N+1 ラウンドトリップになりやすい。`hx-swap-oob`（out-of-band swap）で多重ターゲット更新は可能だが、SPA のリアクティブグラフほど自然ではない。
- **モバイルネイティブ**: HDA の発想を React Native 的領域に持ち込む試みが Hyperview（Carson Gross が共著）だが、エコシステム規模は小さい。
- **テンプレート層への密結合**: フロントエンドとバックエンドのチーム分割を前提とする組織には向かない。むしろフルスタック小規模チームの武器。

## 気づき・洞察

htmx を「軽量 React 代替」と読むのはほぼ確実に誤読である。これは技術選定の問題というより**ソフトウェア工学の歴史認識の問題**だ。Fielding の REST は学術的には記述されていたが、商業的には 2005 年頃に「JSON over HTTP の総称」として骨抜きになった。htmx は「失われた REST を実装で取り戻す」というアジェンダを持った政治的プロジェクトでもある。

もう一つの洞察は、**HTML の表現力を 30 年ぶりに再評価している**点だ。`<form>` は GET/POST しか発行できない、`<a>` は GET しかできない、これらのブラウザネイティブハイパーメディアコントロールの貧弱さに業界は失望して JS に逃げた。htmx は単に「任意の要素が任意の HTTP 動詞を任意のイベントで発行できる」という HTML の自然な拡張を行っただけで、それで多くの SPA が要らなくなることを示した。Carson Gross が後援する Triptych プロジェクトはこの拡張を HTML 仕様自体に逆提案している。これが採用されれば htmx すら不要になる、というのが究極の自己消滅戦略だ。

LoB の重要性は、私が普段感じている「コードを読むときに mental jump が多いほど認知負荷が高い」という直感を概念化してくれる。React/JSX が JS に HTML を埋め込むことで局所性を回復したように、htmx は HTML に振る舞いを埋め込むことで同じ問題を別方向から解いた。両者は「分離より局所」という同じ命題の双子だ。

## 他分野との接続

- **piano（スロー・プラクティスの神経科学）**: スロー練習は「動作の単位を狭め、その単位だけで挙動を完結させて学習する」運動学習。これは LoB の身体版だ。複雑な passage を分解せず一気通貫で弾くと、神経表象が「離れた要素間の相互作用」に依存して脆くなる。htmx の `<button hx-get>` で挙動を要素に閉じ込める発想と同型。
- **cs（Hybrid Logical Clock）**: HLC が物理時計と論理時計を **単一の構造に折り畳む**ことで分散システムの reasoning を局所化したのと同じく、htmx は構造（HTML）と振る舞い（属性）を単一要素に折り畳んで reasoning を局所化する。両者とも「複数の懸念を一つの局所的単位で表現する」設計判断。
- **anatomy（迷走神経と自律神経バランス）**: 迷走神経は中枢に大量の制御を集約せず、末梢で多様な反応を起こす分散制御の代表格。HDA の「サーバが状態の真実源、クライアントは末梢の汎用エフェクタ」とは逆構造に見えるが、本質は「制御の局在」。HDA はサーバ集中型、迷走神経は末梢分散型だが、共通するのは **責務の所在を明示的に固定する**という設計原理。曖昧な責務分散こそが SPA の状態管理地獄と神経系の制御不全の共通病理。
- **bread（パネトーネとリエヴィト・マードレ）**: リエヴィト・マードレは「**サーバが状態の真実源**」の生物版だ。母種という単一の真実源から各バッチの生地が派生し、状態（pH・菌叢比）は母種に集約・継承される。商業イーストの SPA 的アプローチ（毎回新しい培養を JS でブートする）に対し、HDA は「長期に渡って手入れされる単一の状態源」を選ぶ思想。100 年続く母種と "100 Year Web Services" は字面以上に同じ哲学。

## 次に深掘りしたいこと

- Triptych 提案の具体的な仕様（`<form method="put">`、任意要素からの HTTP 発行、部分更新など）と HTML Living Standard のレビュー状況
- Hyperview（モバイル HDA）のレンダリング層と、React Native / SwiftUI 比較における再描画コスト
- HDA における Server-Sent Events / WebSocket 統合パターン（`hx-ext="ws"`）とアクター型 LiveView との性能比較
- Out-of-band swap (`hx-swap-oob`) と Turbo Streams の差分、複数領域同期更新の表現力限界
- htmx 採用組織のフルスタックチーム構造への影響（Conway の法則の逆方向への作用）
