# Sigstore とソフトウェアサプライチェーンセキュリティ
**日付**: 2026-05-10
**分野**: tech
**タグ**: #Sigstore #SupplyChain #SLSA #Cosign #Fulcio #Rekor #SBOM

## 学んだこと

### サプライチェーン攻撃という構造的脅威
SolarWinds (2020), Codecov (2021), 3CX (2023), XZ Utils backdoor (2024) ── 「コードを書いた人」と「実行されるバイナリ」の間に存在する**ビルド・配布パイプライン全体**が攻撃面になった。従来の署名は「PGP 鍵を作ってサーバに置き、たまに更新する」運用で、(1) 鍵管理が破綻 (Fedora 開発者の鍵の 9 割が紛失または弱い)、(2) 失効が困難 (revoke しても古い署名は依然有効)、(3) **誰が**署名したかは追跡できるが**いつ・どこで**署名したかは検証不能、という三重の弱点を抱えていた。Sigstore は Linux Foundation 傘下の OpenSSF プロジェクトとして 2021 年に Red Hat / Google / Purdue 大学が立ち上げ、2024 年に GA、2026 年現在 npm / PyPI / Maven Central / RubyGems / Homebrew が標準採用している。

### Keyless Signing — 鍵を持たない署名
Sigstore のコア発明は **「永続的な署名鍵を作らず、OIDC ID と短命証明書で署名する」** という発想転換。署名フローは次の 5 段階：(1) 開発者または CI が **OIDC プロバイダ** (GitHub Actions, Google, GitLab) から ID トークンを取得、(2) **Fulcio** (Free CA) が ID を検証して **10 分間有効な x.509 証明書**を発行 (subject に GitHub Actions のワークフロー URL が埋め込まれる)、(3) Cosign がローカルで一時鍵ペアを生成し、その公開鍵を CSR で Fulcio に送付、(4) 受け取った証明書で artifact (コンテナイメージ・SBOM・ファイル) に署名、(5) 署名・証明書・タイムスタンプを **Rekor** という公開トランスペアレンシーログ (Trillian Merkle Tree ベース) に追記。**署名後、秘密鍵は破棄される**。検証時には Rekor から署名エントリを取得し、Fulcio 証明書チェーンと OIDC subject (例: `repo:foo/bar:ref:refs/heads/main`) を照合する。

### Fulcio / Rekor / Cosign のアーキテクチャ
**Fulcio** は CT log (Certificate Transparency) と同じ「全証明書を公開ログに刻む」原則で動く Free Code Signing CA。証明書は 10 分しか有効でないが、その間に Rekor に記録された署名は**永久に検証可能** ── これは「失効ではなく不変ログで信頼を証明する」発想。**Rekor** は append-only Merkle tree で、各エントリはハッシュチェーンで連結され、後から改竄しようとすると tree の root hash がズレる。Sigstore Public Good Instance は Rekor の現在の tree size を毎時間 Twitter ボットで公開しており、誰でも一貫性証明を検証できる。**Cosign** は CLI とライブラリで、`cosign sign --keyless image:tag` の一行で署名、`cosign verify --certificate-identity-regexp ...` で検証。

### SLSA フレームワークとの結合
**SLSA (Supply-chain Levels for Software Artifacts)** は Google が提唱した成熟度モデルで、Level 1 (provenance あり) → Level 2 (ホスト型ビルド・改竄不可ログ) → Level 3 (隔離ビルド) → Level 4 (二者署名・密閉) と進む。Sigstore は SLSA Level 3+ の provenance 署名の事実上の実装。GitHub Actions の `actions/attest-build-provenance` は SLSA provenance JSON を Sigstore で署名し Rekor に記録、これにより「このバイナリは確かに main ブランチの commit X からビルドされた」が暗号学的に検証できる。

### Gitsign と policy-controller
**Gitsign** は git commit を Sigstore で署名する仕組み。従来 GPG 鍵が必要だった `git commit -S` を keyless で代替し、commit に埋め込まれた x.509 証明書を Rekor で検証する。**policy-controller** は Kubernetes admission webhook で、ClusterImagePolicy CRD で「`registry.io/foo/*` のイメージは `repo:foo/bar` の GitHub Actions ワークフローからの署名のみ許可」のようなポリシーを書ける。

## 気づき・洞察

### 「鍵管理問題」を「ID 管理問題」に変換した
PGP の根本問題は「鍵 = アイデンティティ」と仮定したこと。だが現実には鍵は紛失・盗難・忘失する。Sigstore は **「鍵は使い捨て、アイデンティティは OIDC で検証」** とパラダイムを反転させた。これはクラウド時代の **「鍵を持たない方が安全」** ── サービスアカウント鍵をローテートし忘れて漏洩するなら最初から永続鍵を作らない、という思想と同じ。CI/CD 環境ですでに OIDC ID を持っているなら、署名鍵を別管理する意味はない。

### Transparency Log の哲学的意義
Rekor は **「秘密ではなく公開で守る」** という発想。従来の CRL/OCSP (失効リスト) は「中央が信頼できる」ことを前提にした pull モデルだったが、CT log と Rekor は「世界中の誰もが log の不変性を検証できる」push モデル。これは **Bitcoin のブロックチェーンと同じ思想 — 信頼を中央に置かず、改竄が暗号学的に不可能な公開構造に置く**。ただし Rekor は Proof of Work ではなく gossip protocol と signed tree heads で一貫性を保証する点が異なる。

### 「いつ署名したか」を一級市民にした
Rekor のタイムスタンプは TSA (Time Stamping Authority) によって署名されるため、**「Fulcio 証明書は失効していたが、Rekor のタイムスタンプ時点では有効だった」が証明できる**。これは長期保存される署名 (= 配布されたバイナリ) で根本的に重要 ── 「証明書の有効期限と署名の有効期限を分離する」設計判断は、長期署名の永続性問題への正解。

## 他分野との接続

- **cs (分散合意)**: Rekor の Merkle tree は Lamport 論理時計と同じく「時刻ではなく順序」で整合性を保証する。Trillian の signed tree head は分散合意を通じて gossip され、Byzantine fault tolerance の faint version と見ることもできる。
- **cs (Bloom filter)**: Rekor は entry の高速検索のため Bloom filter を内部で使用 ── 確率的データ構造が大規模 transparency log の現実解になる。
- **音楽 (DSCH 動機)**: Shostakovich が自分の名前を音符に埋め込んだのと同じく、Sigstore は GitHub Actions の workflow URL を証明書 subject に埋め込む。**「私が私であること」をどう不変に証明するか**という問いは、芸術と暗号で同型。
- **anatomy (固有受容感覚)**: 筋紡錘が「自分の筋肉の状態」を中枢に伝えるのと、Rekor が「自分のビルド履歴」を世界に伝えるのは構造的に類似。**自己認識のフィードバックループ**。

## 次に深掘りしたいこと

- **TUF (The Update Framework)** との関係 ── TUF も Linux Foundation 傘下で、配布側の役割分離を扱う。Sigstore は署名、TUF は配布信頼の root of trust ── 両者の組み合わせ (例: Pythonの PyPI 採用) を理解する。
- **SLSA Build L3 の隔離ビルド要件**を満たす具体的な GitHub Actions 設定。reusable workflow と OIDC subject の関係。
- **EU Cyber Resilience Act (CRA, 2027 適用)** で SBOM 配布が義務化される中、Sigstore + SPDX/CycloneDX SBOM 署名の標準化動向。
- **Sigsum** (Sigstore に類似する KTH 研究プロジェクト) との設計思想の比較。
