# マルチエージェント協調の設計パターン
**日付**: 2026-03-05
**分野**: テクノロジー
**タグ**: #マルチエージェント #設計パターン #オーケストレーター #AI #エージェント

## 学んだこと

### マルチエージェントシステムの爆発的成長
- Gartnerによると、マルチエージェントシステムへの問い合わせが2024年Q1→2025年Q2で**1,445%**増加
- 市場規模は78億ドル（2025年）→520億ドル（2030年）へ成長予測
- 2026年末までにエンタープライズアプリの40%がAIエージェントを搭載（Gartner予測、2025年は5%未満）

### 6つのコア設計パターン

#### 1. Prompt Chaining（逐次連鎖）
- 1つのタスクを複数のステップに分解し、前のステップの出力を次の入力にする
- 最もシンプル。直線的なワークフロー向き
- 例: テキスト生成→校正→フォーマット調整

#### 2. Routing（ルーティング）
- 入力に応じて適切な専門エージェントに振り分ける
- カスタマーサポートの問い合わせ分類など
- 例: 技術的質問→テックサポートAg、請求質問→経理Ag

#### 3. Parallelization（並列化）
- 独立したタスクを複数エージェントに同時実行させる
- レイテンシの削減に有効
- 例: 同時に複数のデータソースを検索

#### 4. Orchestrator-Worker（オーケストレーター-ワーカー）
- 中央のオーケストレーターがタスクを動的に分解・割り当て
- ワーカーは独立して専門タスクを実行
- **動的な分解**が特徴——実行中に新しいサブタスクが発見されることを想定
- Claude Codeのチーム機能がまさにこのパターン

#### 5. Evaluator-Optimizer（評価-最適化）
- あるエージェントが出力を生成し、別のエージェントがそれを評価・改善
- 品質の反復的な向上に有効
- 例: コード生成Ag→コードレビューAg→修正ループ

#### 6. Reflection（リフレクション）
- エージェントが自身の出力を振り返り、自己修正する
- 単体でも使える最も基本的なパターン
- 例: 回答生成→「この回答は正確か？」と自問→修正

### 実運用での組み合わせ
- 本番システムは**単一パターンでは動かない**
- 典型的な組み合わせ: Orchestrator-Worker（タスク分解）＋ Reflection（各ワーカー内の自己修正）＋ Tool Use（外部データ接続）
- パターンの選択でトークン使用量が**200%以上**変動することも——コスト効率は重要な設計要因

### DailyLearningとの関係
- このシステム自体がOrchestrator-Workerパターン
- 「本体」＝オーケストレーター、各分野＝ワーカーエージェント
- 統合レポート＝Evaluator-Optimizer的な品質向上ステップ

## 気づき・洞察
- 6つのパターンは「一人で全部やる」→「チームで分担する」というスケーリングの段階を反映している
- Orchestrator-Workerの「動的分解」がポイント。事前にすべてのサブタスクを予測できないからこそ、実行中に適応する柔軟性が必要
- 「どのパターンを使うか」より「いつパターンを切り替えるか」が実運用の鍵

## 他分野との接続
- **音楽**: オーケストラの指揮者はまさにオーケストレーター。各パート（ワーカー）が専門を持ち、指揮者が全体を統合する
- **パン**: パン作りの工程（計量→ミキシング→発酵→成形→焼成）はPrompt Chainingそのもの
- **ゴルフ**: 運動連鎖もシーケンシャルなパイプライン。各関節が順番に「処理」を受け渡す
- **哲学**: チームワークの本質——個の専門性と全体の協調のバランス
