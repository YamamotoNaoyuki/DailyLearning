# OpenTelemetryとオブザーバビリティ
**日付**: 2026-03-16
**分野**: テクノロジー
**タグ**: #OpenTelemetry #オブザーバビリティ #分散トレーシング #CNCF #監視 #SRE

## 学んだこと

### 1. オブザーバビリティの3本柱（Three Pillars）

オブザーバビリティとは、システムの外部出力から内部状態を推測する能力。従来の「モニタリング」が既知の障害を検知するのに対し、オブザーバビリティは**未知の未知（unknown unknowns）**を探索する能力を提供する。

| シグナル | 役割 | 問いへの答え |
|---------|------|-------------|
| **トレース（Traces）** | リクエストの旅路を追跡 | 「どこで遅くなった？」 |
| **メトリクス（Metrics）** | 集約された数値の時系列 | 「今どのくらい健康？」 |
| **ログ（Logs）** | 個別イベントの詳細記録 | 「何が起きた？」 |

3つは独立ではなく**相互補完的**。メトリクスで異常を検知→トレースで原因箇所を特定→ログで詳細を確認、という流れが典型的なトリアージフロー。

### 2. OpenTelemetry（OTel）の位置づけ

- **CNCF（Cloud Native Computing Foundation）プロジェクト**。Kubernetesに次ぐ2番目にアクティブなプロジェクト
- **OpenTracing + OpenCensus の統合** として2019年に誕生。2つの競合標準が「統一」を選んだ稀有な成功例
- **ベンダー中立のオブザーバビリティ標準**: 計装（instrumentation）をバックエンドから切り離す
- **GA状況**（2026年3月時点）:
  - トレース: GA（2023年〜）
  - メトリクス: GA（2023年〜）
  - ログ: GA（2024年〜）
  - **プロファイリング**: Development段階（2025年〜本格開発）

### 3. OTelアーキテクチャ

```
アプリケーション
  ├── OTel API（計装インターフェース）
  │     └── ベンダー非依存の計装ポイント定義
  ├── OTel SDK（データ処理）
  │     ├── Sampler（サンプリング戦略）
  │     ├── SpanProcessor（バッチ処理）
  │     └── Exporter（送信先）
  └── OTLP（OpenTelemetry Protocol）
        └── gRPC / HTTP でデータ転送
              ↓
        OTel Collector
        ├── Receivers（受信）
        ├── Processors（加工）
        └── Exporters（送信）
              ↓
        バックエンド（Jaeger, Grafana Tempo, Datadog等）
```

**API と SDK の分離**が設計上の核心。APIだけに依存して計装すれば、SDKを差し替えるだけでバックエンドを自由に変更できる。これはMCPの「N×M → N+M問題の解消」と同じ発想。

### 4. 分散トレーシングの中核概念

- **スパン（Span）**: 処理の単位。開始時刻・終了時刻・属性・ステータスを持つ
- **トレース（Trace）**: スパンの有向非巡回グラフ（DAG）。1つのリクエストの全経路
- **トレースコンテキスト伝搬**: HTTPヘッダーを通じてtrace_id/span_idを伝搬
  - **W3C Trace Context**（2020年W3C Recommendation）: `traceparent`ヘッダーが標準
  - 形式: `00-{trace-id}-{parent-span-id}-{trace-flags}`
- **サンプリング戦略**:
  - **Head-based**: リクエスト開始時に確率的に決定。シンプルだが重要なトレースを逃す可能性
  - **Tail-based**: リクエスト完了後にトレース全体を見て決定。エラーや高レイテンシのトレースを確実に捕捉。メモリコスト大
  - **Adaptive sampling**: レート・エラー率に応じて動的に調整

### 5. バックエンドの多様性とOTelによるデカップリング

| バックエンド | 特徴 |
|-------------|------|
| **Jaeger** | Uber開発、CNCF卒業。分散トレーシング特化。OSS |
| **Zipkin** | Twitter開発。軽量。Google Dapperの影響 |
| **Grafana Tempo** | ログとの相関に強い。Lokiとの統合。オブジェクトストレージ利用でコスト効率良 |
| **Datadog** | フルスタックSaaS。APM+インフラ+ログ統合。高機能だがベンダーロックイン |
| **New Relic** | OTLP直接受信対応。段階的移行に好適 |

OTelの価値は**計装の投資を保護する**こと。バックエンドを切り替えても、アプリケーションコードの計装は一切変更不要。これは「インターフェースと実装の分離」というソフトウェア設計の根本原則の体現。

### 6. 自動計装 vs 手動計装

- **自動計装（Auto-instrumentation）**: ライブラリやフレームワーク（Express, Django, Spring等）のHTTP/DB呼び出しを自動でトレース。Java Agent、Python sitecustomize、Node.js require hookなどの仕組み
  - メリット: コード変更ゼロ。即座に可視性を獲得
  - デメリット: ビジネスロジック固有のスパンは作れない
- **手動計装（Manual instrumentation）**: 明示的にスパン作成・属性追加
  - メリット: ビジネスコンテキスト（注文ID、ユーザー種別等）を付与可能
  - デメリット: コード侵入性がある
- **実践的アプローチ**: 自動計装でベースラインを確保→手動計装でビジネスに重要な箇所を強化

### 7. OTel Collector パイプラインアーキテクチャ

```
Receivers → Processors → Exporters
(受信)       (加工)        (送信)
```

- **Receivers**: OTLP、Jaeger、Zipkin、Prometheus等の形式でデータ受信
- **Processors**:
  - `batch`: バッファリングしてバッチ送信（ネットワーク効率化）
  - `memory_limiter`: メモリ使用量を制限（OOM防止）
  - `attributes`: 属性の追加・削除・変換
  - `tail_sampling`: テールベースサンプリング
  - `filter`: 不要なデータの除外
- **Exporters**: OTLP、Jaeger、Prometheus、各種SaaS

Collectorは**Agent（サイドカー）モード**と**Gateway（中央集約）モード**の2つのデプロイパターンを持つ。実運用では両方を組み合わせた2層構成が一般的。

### 8. 最新動向（2025-2026年）

- **OTel Profiling Signal**: CPU/メモリプロファイリングを第4のシグナルとして追加。トレースのスパンとプロファイルデータを直接紐付け、「このスパンが遅い理由はこの関数のCPU消費」まで一気通貫で辿れる
- **eBPFベース計装**: カーネルレベルで計装。アプリケーション無変更でシステムコール・ネットワークパケットを捕捉。Grafana Beyla、Odigos、Cilium Hubbleが代表例。言語非依存で計装できる革命的アプローチ
- **AI/LLMオブザーバビリティ**: LLM呼び出しのトレーシングが急成長分野
  - OpenLLMetry（Traceloop）: LLM呼び出しをOTelスパンとして自動計装
  - `gen_ai.*` セマンティック規約: プロンプト、トークン数、モデル名等の標準属性定義（2025年Experimental）
  - トークン使用量・レイテンシ・ハルシネーション率のメトリクス化
- **OTel OpAMP（Open Agent Management Protocol）**: Collectorの遠隔管理プロトコル。設定変更・ヘルスチェック・アップデートをリモートで
- **Query-based observability**: PromQLやTraceQLによるアドホッククエリ。Grafana Tempoがフルテキスト検索+構造化クエリを統合

### 9. 実践的メトリクス手法

**REDメソッド**（マイクロサービス向け、Tom Wilkie提唱）:
- **R**ate: 1秒あたりのリクエスト数
- **E**rror: 失敗したリクエストの割合
- **D**uration: リクエストの所要時間分布

**USEメソッド**（インフラ向け、Brendan Gregg提唱）:
- **U**tilization: リソースがどれだけ使われているか（CPU使用率等）
- **S**aturation: キュー待ちの深さ（過負荷の兆候）
- **E**rrors: エラーイベント数

**4つのゴールデンシグナル**（Google SRE Book）:
- Latency, Traffic, Errors, Saturation

REDはサービス視点、USEはリソース視点。両方を組み合わせることで「サービスが遅い（RED）→CPUが飽和している（USE）」と因果関係を辿れる。

### 10. トレース・メトリクス・ログの相関（Correlation）

「Single Pane of Glass（一枚板のガラス）」＝全シグナルを統合したビュー。

- **Exemplar**: メトリクスの特定データポイントからトレースへのリンク。「p99レイテンシが跳ねた瞬間の具体的なトレースはこれ」
- **Resource attributes**: `service.name`, `service.version`, `host.id`等の共通属性で3シグナルを横断検索
- **Trace ID injection**: ログにtrace_idを埋め込み、ログ→トレースの直接遷移を可能に
- **Baggage**: トレースコンテキストに任意のキーバリューを乗せて伝搬。ユーザーIDや実験フラグ等

相関の実現により「メトリクスダッシュボードで異常検知→該当トレースにドリルダウン→関連ログで詳細確認」というシームレスなトリアージが可能になる。

## 気づき・洞察

### 「標準化による解放」の再帰的パターン
OTelの本質は、計装（データ収集）とバックエンド（データ分析）を分離する標準化。これはMCPがツール接続を標準化し、A2Aがエージェント通信を標準化するのと完全に同じ構造。テクノロジーの進化は「N×M問題を標準レイヤーでN+M問題に変換する」ことの繰り返し。USB、HTTP、SQL、OTel、MCP——すべて同じパターン。

### OTel Collectorは「AgentGateway」と同じアーキテクチャ
Receivers→Processors→Exportersのパイプラインは、AgentGatewayの受信→ポリシー適用→転送と同型。「中間に変換・制御レイヤーを挟む」というプロキシパターンは、異種システム統合の普遍的解法。

### eBPFは「透明な計装」の究極形
アプリケーションコードに一切触れずにカーネルレベルで観測する。これは自動計装のさらに先——「ゼロ計装」とも呼べる。侵入性ゼロで可視性を得る思想は、テスト駆動開発における「テスタビリティのための設計変更は不要であるべき」という理想に通じる。

### 第4のシグナル「プロファイリング」の統合意義
トレースが「どのサービスが遅い」を示し、プロファイリングが「なぜ遅い（どの関数が重い）」を示す。マクロからミクロへの一気通貫。これはオブザーバビリティの解像度が「サービス単位」から「関数単位」に引き上げられることを意味する。

## 他分野との接続

### ゴルフ × コースアーキテクチャの設計哲学
ゴルフコースの設計はプレイヤーの「戦略的選択」を可視化するアーキテクチャ。グリーンの傾斜、バンカーの配置、フェアウェイの幅——これらはプレイヤーの判断と結果を「観測可能」にする装置。OTelがマイクロサービスの振る舞いを可視化するように、優れたコース設計はプレイヤーの意思決定プロセスを可視化する。REDメソッド（Rate/Error/Duration）は、ゴルフでいえば「ショット頻度/ミスショット率/ホール所要時間」のスコアカード分析に相当する。

### 解剖学 × 横隔膜は「隠れたインフラストラクチャ」
横隔膜は呼吸・姿勢・内臓安定の基盤だが、意識されることは少ない。オブザーバビリティなき横隔膜は「モニタリングなきインフラ」と同じ。身体の不調（腰痛、肩こり）がメトリクスの異常であり、そこから横隔膜の機能不全というroot causeをトレースバックする——まさに分散トレーシング的診断。USEメソッド（Utilization/Saturation/Errors）で横隔膜の状態を評価すると、浅い呼吸=高Utilization低効率、息切れ=Saturation、横隔膜攣縮=Errorと読める。

### その他 × 地図投影法のトレードオフ
地図投影法は3次元の地球を2次元に写す際に必ず何かを犠牲にする（面積・角度・距離）。オブザーバビリティの3シグナルも同様で、トレース（経路の正確さ）、メトリクス（全体の面積感）、ログ（局所の詳細）はそれぞれ異なる投影法のように現実の異なる側面を捉える。「Single Pane of Glass」の理想は、複数の投影法を重ね合わせて地球の全体像を復元しようとする試みに似ている。
