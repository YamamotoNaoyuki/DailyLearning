**日付**: 2026-04-18
**分野**: cs
**タグ**: #GarbageCollection #TriColorMarking #GenerationalGC #G1GC #ZGC #Go-GC #MemoryManagement

## 学んだこと

### GC の基本問題
プログラムが動的に確保したメモリのうち、**「今後どこからもアクセスされない」ものを自動的に解放する**問題。手動メモリ管理（malloc/free）のバグ（use-after-free, double-free, memory leak）を構造的に排除する。本質的な難しさは **「到達可能性」を実行中のプログラム状態から判定する** こと。

### 到達可能性解析の基礎
- **ルート（Root Set）**: スタック、グローバル変数、レジスタ、CPU の既知のオブジェクト参照
- **オブジェクトグラフ**: オブジェクトとその参照関係をノード/エッジとするグラフ
- **到達可能**: ルートから有向パスで到達できるオブジェクト
- **ガベージ**: 到達不可能なオブジェクト → 回収対象

### 古典アルゴリズム群
**参照カウント（Reference Counting）**:
- 各オブジェクトに被参照数を保持。0になったら即座に解放
- **利点**: 即時性、予測可能な停止
- **欠点**: 循環参照が解放できない、カウント更新が atomic で遅い、スレッド安全性コスト
- 使用例: Python（補助的に循環検出）、Swift ARC、C++ shared_ptr

**Mark-Sweep**:
- **Mark フェーズ**: ルートから到達可能なオブジェクトを全てマーク
- **Sweep フェーズ**: ヒープを線形走査し、マークされていないオブジェクトを解放
- **利点**: 循環参照を扱える
- **欠点**: 断片化（fragmentation）、STW（Stop-The-World）の長さ、ヒープ全体走査

**Mark-Compact**:
- Mark 後、生存オブジェクトをヒープの片側に寄せる
- 断片化解消、アロケーションが bump-pointer で O(1) 化
- 欠点: オブジェクト移動のコスト、参照更新が必要

**Copying（Cheney's Algorithm, 1970）**:
- ヒープを From/To の2領域に分割
- GC 時に生存オブジェクトを To 領域にコピー
- 利点: 断片化なし、アロケーション高速
- 欠点: ヒープの半分しか使えない、移動コスト

### 世代別GC（Generational）
**世代別仮説（Weak Generational Hypothesis）**: 「多くのオブジェクトは若くして死ぬ（short-lived）」

- **Young Generation（Eden + Survivor）**: 新規オブジェクト。頻繁に GC（Minor GC）
- **Old Generation**: 複数回の Minor GC を生き残ったオブジェクト。稀に GC（Major GC / Full GC）
- 若い世代に GC 作業を集中させることで、全体の GC コストを大幅削減
- Java の Parallel GC、.NET CLR、Go（間接的）、V8 など広く採用

**Card Table / Remembered Set**:
世代間参照（Old → Young）の追跡機構。GC 時に全 Old を走査せずに済む。Java は 512バイト単位で「このカードに世代間参照があるか」を記録する。

### 三色マーキング（Tri-Color Marking, Dijkstra 1978）
並行 GC の理論的基盤。オブジェクトを3色で分類：

- **白（White）**: 未訪問（ゴミの候補）
- **灰（Gray）**: 訪問済みだが、子ノードの一部は未訪問
- **黒（Black）**: 訪問済み、全ての子も訪問済み

**不変条件（Strong Tri-Color Invariant）**: 黒→白への直接参照は存在しない

**アルゴリズム**:
1. 全オブジェクトを白で初期化、ルートから直接参照されるものを灰に
2. 灰オブジェクトを1つ選び、その参照先を灰（既に黒でなければ）にし、自身を黒に
3. 灰集合が空になるまで繰り返す
4. 残りの白オブジェクトを解放

### ライトバリア（Write Barrier）— 並行GCの肝
並行GC ではアプリケーションスレッド（Mutator）が動作中に参照が変化する。これを捕捉するのが**ライトバリア**：

- **Dijkstra スタイル（Insert Barrier）**: 「黒オブジェクトが新規に白を参照したら、白を灰にする」
- **Yuasa スタイル（Delete Barrier）**: 「黒オブジェクトから参照が削除される前に、削除される対象を灰にする」
- **Hybrid（Go）**: 両方の要素を組み合わせた独自バリア

バリアは全ての参照書き込みに挿入される小コードで、オーバーヘッドはあるが STW を劇的に短縮できる。

### Go の GC — Pacer と並行三色
Go は Pacer（ペーサー）という機構で GC のタイミングを制御する：

- **GOGC 環境変数**: 例 100 なら「前回のライブヒープの2倍まで膨らんだら次のGC」
- 並行マーク・並行スイープ。STW は初期化と完了チェックのみ（数十マイクロ秒）
- Write Barrier は hybrid 型（Yuasa delete + Dijkstra insert の一部）
- 目標: STW 500μs 以下、Go 1.12 以降ほぼ達成

### Java の G1GC（Garbage First, Java 9 default）
ヒープを**等サイズのリージョン（1-32MB）**に分割、ガベージの多いリージョンから優先的に回収：

- **Young/Old の区別は論理的** — 同じリージョンが時によって Young として使われる
- Mixed GC: Old リージョンも含めた部分的 GC
- 一時停止目標（pause target）を設定可能（-XX:MaxGCPauseMillis）
- 数GB〜数十GBヒープに最適

### Java の ZGC（低レイテンシGC）
**Sub-millisecond pauses regardless of heap size** を目指した設計：

- **Colored Pointers**: 64bit ポインタに状態メタデータ（mark, remap, finalizable）をビットで埋め込む
- **Load Barrier**: オブジェクト参照の**読み取り時**にバリアを挿入し、必要なら self-healing（参照更新）
- 並行コピー（Concurrent Relocation）: GC がオブジェクトを移動する間もアプリは動作
- STW は roots scan の短時間のみ
- Java 15 で production-ready、Java 21 で generational ZGC が標準化
- TB 級ヒープでも pause 1ms 未満を維持

### Shenandoah（Red Hat, Java 12+）
ZGC と類似の低レイテンシ設計。**Brooks Pointer** 方式で forwarding を実現。ZGC が Hotspot にタイトに結合するのに対し、Shenandoah はより移植性重視。

### 最新の研究方向（2025-2026）
- **Epsilon GC（Java）**: GC を行わない（ベンチマーク用）
- **Lilliput Project**: オブジェクトヘッダを 96bit → 64bit に削減しメモリ効率化
- **Project Loom（仮想スレッド）との統合**: 大量の仮想スレッドスタックのGC最適化
- **Far Memory GC**: CXL / RDMA でリモートメモリを含むヒープ管理

## 気づき・洞察

- **「停止時間とスループットのトレードオフ」は本質的**: 並行GC は全体スループットを犠牲にしてレイテンシを稼ぐ。リアルタイムシステム（低レイテンシ優先）vs バッチ処理（高スループット優先）で選択が正反対になる。**最適なGCはワークロード依存**。
- **世代別仮説の経験的真実**: オブジェクト寿命分布のロングテール性（多くは短命、一部は超長命）は、**あらゆるメモリアクセスパターン** で観測される普遍的統計的事実。CPUキャッシュの時間的局所性も同じ原理。
- **三色マーキングの抽象美**: 3色という最小限の状態で、並行環境での到達可能性解析を厳密に定義できる。これは CRDT の合流半束（tech 2026-03-21）や Raft の状態機械（tech 2026-03-22）と並ぶ**分散アルゴリズム設計の傑作**。
- **ZGC の Colored Pointer という発想**: 64bit アドレス空間のうち実際に使うのは 48bit 程度、残りを**メタデータに転用**する発想。「未使用リソースの発見」という最適化の極致。アドレス空間を型と見なす視点。
- **Rust の「GC不要」の意味**: 所有権・借用・ライフタイムの静的検査で、**コンパイル時に解放タイミングを決定**する。GC は「動的な解決策」、Rust は「静的な解決策」。両者は根本的に異なるアプローチで同じ問題を解く。
- **GC と言語設計の密接性**: 言語の表現力（クロージャ、高階関数、persistent data structures）は GC の効率と深く結合する。Haskell の純粋関数型＋遅延評価のランタイムは大量の短命オブジェクトを生むため、generational GC との相性が特に良い。

## 他分野との接続

- **イベントソーシング（tech 2026-03-23）の「墓標問題」との類似**: CRDT の tombstone は物理削除不可能、GC も「到達不能だが物理的にはまだある」状態を扱う。**「不要を検出してから実際に消すまで」のギャップ**。
- **仮想メモリとページング（cs 2026-03-23）との深い接続**: OS の仮想メモリと GC は「メモリの階層管理」という共通課題を扱う。ページフォールトとGC サイクルはどちらも「実行中の透明な背景処理」。
- **CPU キャッシュ（cs 2026-03-20）との同型性**: キャッシュも GC も**サイズ制限のあるリソースをどのように管理するか**の問題。LRU vs Generational 仮説の類似性。
- **骨盤底と横隔膜のピストン（anatomy 2026-04-18）との遠い共鳴**: GC は「メモリ領域の圧力管理」、ピストンは「腹腔内圧の管理」。**圧の動的バランス**というメタファが通底。
- **Nix の「古い generation は残る」（tech 2026-04-18）**: Nix も generation 概念を持つが、自動 GC でなく**ユーザー明示の `nix-collect-garbage`**。対照的なアプローチ——動的検出 vs 明示的管理。
- **TCP 輻輳制御（cs 2026-03-21）の動的調整**: GC Pacer は「ヒープサイズの動的調整」を行い、TCP は「送信速度の動的調整」を行う。**制御理論の異なる応用**。
- **ゴルフの Strokes Gained（golf 2026-04-18）との共通構造**: どちらも「ある操作（1打/1オブジェクトアロケーション）のコストを、ベンチマークからの差分で評価」するメトリクス。
- **暗号と GC の忘却**: Passkeys（tech 2026-04-17）は「秘密を長期保持しない」設計。GC も「不要データを速やかに忘れる」。**忘却の設計思想**。
