# TLA⁺ と形式手法 — 並行・分散システムの仕様検証
**日付**: 2026-05-16
**分野**: cs
**タグ**: #TLA+ #形式手法 #Lamport #モデル検査 #分散システム

## 学んだこと

**TLA⁺**（Temporal Logic of Actions Plus）は Leslie Lamport が 1990 年代に作った仕様記述言語。「実装ではなく **設計** を検証する」という思想で、ソースコードを書く前に **アルゴリズムや並行プロトコルの正しさ** を機械検査する。AWS（DynamoDB、S3、EBS、Cosmos DB）、Azure、Microsoft の Xbox One メモリマネージャ、Docker Datacenter、CockroachDB、MongoDB の Raft 実装などが本番投入前に TLA⁺ で設計を検証した実績がある。

### TLA⁺ の設計哲学

Lamport は分散システムを「**状態と状態遷移の集合**」として捉える。プログラミング言語の抽象化ではなく、**数学（特に集合論と一階述語論理 + 時相演算子）** で書く：

- **状態（state）**：変数の値の組
- **アクション（action）**：状態を遷移させる述語、`x' = x + 1` のように **プライム記号** で次状態を表す
- **挙動（behavior）**：状態の無限列
- **仕様（specification）**：初期条件 + 「常に何らかのアクションが起こる」+ 公平性条件

時相演算子：
- `□P`（Box, always）：常に P が成立
- `◇P`（Diamond, eventually）：いつかは P が成立
- `□◇P`：常にいつかは（無限回）成立 — 公平性
- `◇□P`：いつか以降ずっと成立 — 安定性

### PlusCal — 擬似コードの皮

TLA⁺ の数学的記法はいきなり読むと辛いので、Lamport は **PlusCal** という擬似コード風の記法を作った。`while`、`if`、`await` といった構文で書け、自動的に TLA⁺ に変換される。

例：2 つのプロセスが共有変数 `x` をインクリメントする：
```
--algorithm Incrementer {
  variable x = 0;
  process (P \in {1, 2}) {
    a: x := x + 1
  }
}
```
これを TLA Toolbox に通すと、TLA⁺ 仕様に翻訳される。

### モデル検査（TLC）

TLA⁺ Toolbox の核は **TLC モデル検査器**。仕様から **全到達状態を列挙** して、安全性属性（不変条件）と活性性属性（liveness）を検査する。

検査されるバグの種類：
- **デッドロック** — 全プロセスが await 状態
- **不変条件違反** — 例「アカウント残高 ≧ 0 が破られる」
- **時相属性違反** — 例「リクエストはいつか応答される（◇）が破られる」

状態空間爆発のため、**有限のモデルパラメータ**（プロセス数 3、メモリサイズ 4 など）で網羅検査する。これは小さいサイズで反例を見つける戦略で、「大半のバグは小さい例で再現される」という経験則（Small Scope Hypothesis）に依拠する。

### TLAPS — 定理証明

無限状態空間や任意パラメータの証明には、モデル検査では足りない。**TLAPS（TLA⁺ Proof System）** は機械支援の定理証明系で、手動でステップを書き、ZFC 集合論ベースの公理から証明する。Lamport の Paxos の正しさ証明、Disk Paxos、Byzantine Paxos の完全証明、Memoir セキュリティアーキテクチャの証明などが TLAPS で機械検証されている。

### 実例：DynamoDB の不変条件発見

AWS のチームが DynamoDB の **マルチペーパー（multi-paper）レプリケーション** プロトコルを TLA⁺ で書いたとき、35 ステップ深さで反例が出た。これはコードレビューで何百時間かけても見つかっていなかった。AWS の Chris Newcombe らの論文「How Amazon Web Services Uses Formal Methods」（CACM 2015）は、TLA⁺ が **大手クラウドプロバイダの本番品質保証手段になっている** ことを公にした転換点。

### TLA⁺ の限界

- **学習曲線が急** — 数学記法に慣れた人でも数週間〜数ヶ月。Lamport 自身が「俺の本 [Specifying Systems] が無料 PDF なのに誰も読まない」と嘆く
- **実装との乖離** — TLA⁺ で正しいと示せても、C++/Java 実装にバグがあれば無意味。**仕様と実装のギャップ問題**
- **モデルサイズの制約** — 状態空間爆発で、大規模パラメータでは検査できない

これらの限界を補うのが [[tech/FoundationDBと決定論的シミュレーションテスト]] のような実装レベルのテスト。**TLA⁺ で設計を検証し、DST で実装を検証する** のが現代の最先端の組み合わせ。

### TLA⁺ vs その他の形式手法

| 手法 | 強み | 弱み |
|------|------|------|
| **TLA⁺** | 並行・分散系に強い、時相論理 | 学習コスト高、実装と乖離 |
| **Alloy** | 集合論ベース、可視化良好 | 時相論理が弱い |
| **Coq/Lean** | 定理証明完備、純粋数学にも使える | プログラミング言語に近い実装 |
| **Z notation** | 古典的、英国系で人気 | 並行性記述が弱い |
| **Spin/Promela** | プロトコル検証に特化 | プログラム的、抽象度低い |
| **CBMC** | C コードの bounded model checking | 高レベル設計には向かない |

### 最近の動向

- **PGo** — PlusCal を **Go コード** に自動変換する MIT のプロジェクト。仕様と実装のギャップを縮める試み
- **Apalache** — TLC の代替で、**SMT ベース**のシンボリックモデル検査器。状態空間爆発を緩和
- **Quint** — 2024 年に Informal Systems が出した「TLA⁺ の現代的シンタックス」。TypeScript 風で読みやすい
- **Stateright** — Rust ライブラリで、ホスト言語（Rust）で書ける TLA⁺ 風モデル検査

## 気づき・洞察

形式手法の根本的価値は **「コード書く前に設計の穴を見つける」**。バグはコードでなく **設計判断** に潜むことが多く、コードレベルのテストでは捕まらない。Newcombe（AWS）の発言が印象的：「我々が TLA⁺ で見つけた **35 ステップ深さの反例** は、レビュー数百時間でも、人間の頭では辿りえない複雑性だった」。**人間の認知限界が露出する深さ** こそ形式手法の価値領域。

なぜ業界で広く採用されないか — 「**書ける人がいない**」が最大の要因。プログラミングと数学の両方ができる人材は希少で、AWS のような巨大組織でも形式手法エンジニアは少数。学習投資が「3 ヶ月で書けるようになる」程度では、ROI が見えない経営判断者には売りにくい。**形式手法の人材市場が痩せている**ことが普及を妨げる。

最も重要な認識 — TLA⁺ は **「証明する」のではなく「設計を強制的に厳密化する」** ツール。仕様を書く過程で、曖昧だった部分が **必ず** 露わになる。「何が共有変数か」「いつ書き込めるか」「失敗時の状態は何か」を **数学的に書くことを強要される** ので、ホワイトボード議論で済ませた曖昧さが消える。**検査結果より、書く過程に価値がある**。

[[tech/FoundationDBと決定論的シミュレーションテスト]] との対比：
- TLA⁺ は **演繹的（top-down）** — 仕様の全状態空間を網羅
- DST は **実験的（bottom-up）** — 実装の状態空間を高速サンプリング
- 両者は **補完関係** にあり、AWS では両方を併用
- 業界で「TLA⁺ + DST」の組み合わせが分散システム品質保証の **新しい標準** になりつつある

時相論理の **□（always）** と **◇（eventually）** は、自然言語の「常に」「いつか」と一致する。**並行性を語る人間の直観を形式化した** ことが Lamport の天才。「リクエストは常に応答される」は `□(req ⇒ ◇resp)` と素直に書ける。

### 形式手法と人類の認知限界

人間の作業記憶は **7±2 個** のチャンクが限界（Miller 1956）。分散プロトコルの状態空間は数千〜数億状態。**人間がレビューで全状態を頭に持つことは原理的に不可能**。形式手法は人間の脳の限界を補う計算的補綴具で、これは [[anatomy/小脳と運動学習の内部モデル]] が運動制御における脳の予測補助をしているのと同型。**機械が人間の認知を拡張する** 時代の典型例。

## 他分野との接続

- **[[tech/FoundationDBと決定論的シミュレーションテスト]]**: 仕様検証（TLA⁺）と実装検証（DST）の二刀流。両者の組み合わせが現代分散システム品質保証の最先端
- **[[cs/PaxosとRaft_分散合意アルゴリズム]]**: Lamport 自身が Paxos を発明し、その正しさを TLA⁺ で証明した。形式手法は分散合意理論と双子のように発展
- **[[anatomy/小脳と運動学習の内部モデル]]**: 形式手法は人間の認知拡張ツール、内部モデルは身体運動の認知拡張機構。**「予測の前倒し」が知能の本質**
- **[[bread/発酵管理の科学_pH_TTA]]**: 発酵プロセスを状態機械として TLA⁺ で書ける（「pH が 4 を下回ったら停止」「TTA が 8 に達したら投入」）。**プロセス制御と分散プロトコルは同じ言語で書ける**
- **[[music/シューマンのクライスレリアーナ]]**: ロマン派音楽の「断片性」は、線形時間でなく**並行的・分岐的時間**を扱う。時相論理の □/◇ は、こうした非線形時間の構造化にも使える
