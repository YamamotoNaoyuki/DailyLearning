## 学んだこと

### 1. メモリウォール問題 — なぜキャッシュが必要になったか

1980年代から2000年代にかけて、CPUの演算速度とメモリのアクセス速度の乖離は指数関数的に拡大した。Hennessy & Patterson (Computer Architecture: A Quantitative Approach) が示すデータ:

- **CPU性能向上率**: 年間約60%（1986–2003年）
- **DRAM性能向上率**: 年間約7%（同期間）
- **累積ギャップ**: 1980年代初頭に比べ、2000年代には**数百倍**の乖離

Wulf & McKee (1995) の論文 "Hitting the Memory Wall" がこの問題を定式化した。CPUがいくら速くなっても、メモリアクセス待ちでストールすれば実効性能は上がらない。ここから**メモリ階層（memory hierarchy）**の設計が計算機アーキテクチャの中心課題となった。

設計原理は2つの局所性（locality）に依拠する:
- **時間的局所性（temporal locality）**: 最近アクセスしたデータは再度アクセスされやすい
- **空間的局所性（spatial locality）**: あるアドレスにアクセスしたら、近傍のアドレスもアクセスされやすい

これは経験的法則ではなく、プログラムの構造的必然（ループ、配列の逐次走査、スタックフレーム）から導かれる。

### 2. キャッシュ階層の設計トレードオフ — L1/L2/L3

現代のプロセッサ（例: Intel Alder Lake, AMD Zen 4）の典型的な構成:

| レベル | サイズ | レイテンシ | 共有 | 結合度 |
|--------|--------|-----------|------|--------|
| L1d (データ) | 32–48 KB/コア | ~4–5サイクル | コア専有 | 8-12ウェイ |
| L1i (命令) | 32 KB/コア | ~4サイクル | コア専有 | 8ウェイ |
| L2 | 256KB–1.25MB/コア | ~12–14サイクル | コア専有 | 8-16ウェイ |
| L3 (LLC) | 16–96 MB | ~30–50サイクル | 全コア共有 | 12-16ウェイ |
| DRAM | 数十GB | ~100–200サイクル | 全コア共有 | — |

**設計上の核心的トレードオフ**:

**サイズ vs レイテンシ**: SRAMは高速だが面積・消費電力が大きい。L1を32KBに留めるのは、配線長を短く保ちクロックサイクル内でアクセスを完了させるため。物理的に「光速の壁」がある — 5GHzのプロセッサで1サイクルは0.2ns、光（電気信号）は6cm程度しか進めない。L1はCPUのパイプラインに密結合する必要があるので、物理的距離の制約が容量を規定する。

**L1をデータ/命令に分離する理由**: Harvard architectureの思想。データアクセスと命令フェッチは同時に起きるため、ポートを共有すると構造的ハザードになる。L2以降はunifiedにすることで、命令キャッシュとデータキャッシュの配分を動的に調整できる。

**L3の共有設計**: コア間のデータ共有を高速化し、キャッシュコヒーレンスのトラフィックをメモリに出す前にLLC内で吸収する。Intel のinclusive L3は「コヒーレンスフィルタ」として機能するが、有効容量が減るトレードオフがある。AMDのnon-inclusive/victimキャッシュ方式はこれを回避するが、ディレクトリ管理が複雑になる。

### 3. キャッシュラインの設計 — なぜ64バイトなのか

キャッシュの転送単位は「キャッシュライン（cache line）」であり、現代のx86プロセッサではほぼ例外なく**64バイト**。この値は偶然ではなく、複数のトレードオフの均衡点:

**空間的局所性の活用**: ラインが小さすぎる（例: 16バイト）と、近傍データを取りこぼしプリフェッチ効果が減る。配列走査ではライン境界を頻繁に跨ぎ、ミスが増える。

**帯域幅の効率**: DRAMはバースト転送（burst mode）で連続アドレスを高速に読む設計。DDR4/DDR5のバーストサイズは8ビート × 8バイト = 64バイトで、キャッシュラインサイズと一致する。これは偶然の一致ではなく、共最適化（co-optimization）の結果。

**大きすぎるラインの問題**: 128バイトや256バイトにすると:
- **帯域幅の浪費**: 実際に使うデータが数バイトでも、常に大きな塊を転送する
- **false sharingの悪化**: マルチコアでの競合単位が大きくなる
- **ミスペナルティの増大**: 1ラインの充填に時間がかかる
- **キャッシュ汚染**: 不要なデータがキャッシュを占有する

**タグオーバーヘッド**: ラインが小さいとタグ記憶の比率が増え、面積効率が下がる。64バイトはタグ/データ比が妥当な水準。

結局、64バイトは「DRAMバースト転送サイズ」「空間局所性の典型的スパン」「タグオーバーヘッド」「false sharing影響」の4変数の最適化点として収束している。

### 4. 結合度（Associativity）の設計空間

- **ダイレクトマップ（1-way）**: インデックスビットでセットが一意に決まる。高速だがconflict missが多い。同じセットにマップされるアドレスが2つあるだけでスラッシングが起きる。
- **フルアソシアティブ（N-way）**: 任意のスロットに格納可能。conflict missゼロだが、全エントリの比較が必要で面積・消費電力が爆発。TLBのような小規模構造でのみ実用的。
- **セットアソシアティブ（k-way）**: 折衷案。8-way以上でフルアソシアティブに近いミス率を達成（Hill & Smith, 1989）。

L1が8〜12ウェイに設定される理由: ミス率の改善がk=8前後で飽和する（収穫逓減）のに対し、比較器の面積・レイテンシは線形に増加する。

### 5. 置換ポリシー — なぜ単純が勝つか

**理想**: Béládyの最適アルゴリズム（OPT）— 将来最も遠くで使われるラインを追い出す。実装不可能だが理論的下界を与える。

**LRU（Least Recently Used）**: 時間的局所性の直接的な実装。問題は**実装コスト**。N-way LRUには各セットでN!の順序状態を追跡する必要がある。8-wayで40320状態。これは実用的に困難。

**Pseudo-LRU（Tree-PLRU）**: 二分木で近似LRU順序を追跡。8-wayで7ビットの状態のみ。LRUとのミス率差は通常1–2%以内。コスト対効果が圧倒的に優れる。

**Random置換**: 状態の記録が一切不要。直感に反するが、8-way以上でPLRUとの差はわずか。Patterson & Hennessy曰く「associativityが十分高ければ、置換ポリシーの選択はほとんど影響しない」。

**なぜ単純が勝つか**:
- 置換ポリシーが重要になるのはconflict missが多いとき。結合度が十分高ければconflict miss自体が少なく、ポリシーの差が出ない
- ハードウェアの消費電力・面積・タイミングクリティカルパスへの影響の方が性能間接効果より大きい
- LLCではSRRIP（Static Re-Reference Interval Prediction）等の適応型ポリシーが使われるが、これもLRUの欠陥（スキャン耐性のなさ）を補うための設計

### 6. 書き込みポリシー — Write-Through vs Write-Back

**Write-Through**: 書き込み時にキャッシュとメモリの両方を更新。
- 利点: メモリとの一貫性が常に保たれ、設計がシンプル
- 欠点: 書き込みのたびにメモリバスを消費。帯域幅がボトルネック
- ストアバッファで緩和可能: CPUはバッファに書いて先に進み、バッファがバックグラウンドでメモリに反映。バッファが溢れればストールするが、書き込みのバースト性を吸収

**Write-Back**: 書き込みはキャッシュのみ更新し、ダーティビットを立てる。追い出し時にメモリに書き戻す。
- 利点: メモリバストラフィックを大幅削減。同一ラインへの複数書き込みが1回の書き戻しに統合される（write coalescing）
- 欠点: キャッシュとメモリの不一致が常態化。コヒーレンスプロトコルが複雑化

**Write-Allocate vs No-Write-Allocate**:
- Write-Allocate（fetch-on-write）: ミス時にラインを読み込んでから書く。Write-Backと組み合わせるのが標準
- No-Write-Allocate: ミス時はメモリに直接書き、キャッシュには入れない。Write-Throughと組み合わせることが多い

現代のプロセッサは**L1/L2はWrite-Back + Write-Allocate**が主流。メモリバス帯域幅が最も制約の厳しいリソースだからである。

### 7. キャッシュコヒーレンス — MESIプロトコルとスケーラビリティ

マルチコアでは各コアが独立キャッシュを持つため、同一アドレスのコピーが複数存在しうる。これが**コヒーレンス問題**。

#### MESIプロトコル

各キャッシュラインは4状態のいずれか:
- **Modified (M)**: このコアだけが持ち、メモリと異なる。他コアへの提供責任がある
- **Exclusive (E)**: このコアだけが持ち、メモリと同じ。書き込み時にMに遷移（バストラフィック不要）
- **Shared (S)**: 複数コアが持ちうる。書き込み時はInvalidateを送信してから M に遷移
- **Invalid (I)**: 有効なデータなし

E状態の設計意図: 「自分だけが持っている」ことを知っていれば、write時にバスに通知せずMに遷移できる。これがなければ全writeにバストランザクションが必要。

**MOESI拡張**（AMD）: O（Owned）状態を追加。ダーティデータを他コアにSharedとして渡しつつ、メモリへの書き戻しを遅延。メモリバストラフィック削減。

**MESIF拡張**（Intel）: F（Forward）状態。Shared間のデータ要求に対し、1つのコアだけが応答する役割を明示。バスの応答衝突を回避。

#### スヌーピング vs ディレクトリベース

**スヌーピング（Bus Snooping）**:
- 全コアがバスを監視し、自分のキャッシュに関係するトランザクションに反応
- 実装がシンプルで低レイテンシ
- **スケーラビリティの壁**: コア数が増えるとバス帯域幅が飽和。O(N)のブロードキャストが必要
- 実用上限: 8–16コア程度

**ディレクトリベース**:
- 各キャッシュラインの共有状態を中央（または分散）ディレクトリに記録
- Invalidateは実際に共有しているコアにだけポイントツーポイントで送信
- **スケーラビリティ**: O(1)のユニキャスト。64コア以上でも動作
- **オーバーヘッド**: ディレクトリ自体のストレージ（各ラインにNビットのシェアラービットベクトル）、間接参照のレイテンシ追加
- Intelのメッシュインターコネクト、AMDのInfinity Fabricはディレクトリベースの分散型

現代のサーバCPUは**ハイブリッド**: クラスタ内（CCX/タイルなど）はスヌーピング、クラスタ間はディレクトリ。局所性の高い通信は低レイテンシ、遠距離通信はスケーラブルに。

### 8. False Sharing — マルチスレッドの静かな性能殺し

```
// 典型的なfalse sharing
struct Counters {
    int64_t counter_a;  // Thread A が更新
    int64_t counter_b;  // Thread B が更新
};
```

`counter_a` と `counter_b` は論理的に独立だが、同一キャッシュライン（64バイト）に載る。Thread Aが `counter_a` を書くと、Thread Bの持つそのラインがInvalidateされ、Thread Bは `counter_b` を読むだけでもキャッシュミスが発生する。

**性能影響**: false sharingは真の共有（true sharing）より検出が難しく、影響は深刻。Intel VTuneや `perf c2c` で検出可能。

**対策**:
```c
// パディングによる分離
struct Counters {
    alignas(64) int64_t counter_a;  // 独立したキャッシュライン
    alignas(64) int64_t counter_b;  // 独立したキャッシュライン
};
```

C++17の `std::hardware_destructive_interference_size`（= 通常64）がこの用途に用意されている。Linuxカーネルの `____cacheline_aligned_in_smp` マクロも同様の目的。

Java の `@Contended` アノテーション（JEP 142）も同一の問題を言語レベルで解決する設計。

### 9. プリフェッチ — レイテンシ隠蔽の技法

**ハードウェアプリフェッチャ**: CPUがアクセスパターンを観測し、次にアクセスされるであろうラインを投機的にフェッチ。

- **ストライド検出**: 配列のシーケンシャルアクセス（stride = 要素サイズ × N）を検出。これは非常に効果が高く、行列乗算やベクトル演算のパイプラインを満たす
- **ストリームバッファ**: 連続するキャッシュラインミスのパターンからストリームを検出し、先読み
- **限界**: ポインタチェイス（リンクリスト、ツリー走査）はアドレスが予測不能でプリフェッチが効かない。これがB木のリーフノード内をソート配列にする動機の一つ

**ソフトウェアプリフェッチ**: `__builtin_prefetch()` (GCC/Clang), `_mm_prefetch()` (Intel intrinsics)。

- プログラマがアクセスパターンの知識を注入
- ハッシュテーブルのルックアップ前にバケットをプリフェッチ、等
- **リスク**: プリフェッチの距離（先読みラインの先行度）の調整が難しく、キャッシュ汚染を起こしうる。命令スロットを消費する。ハードウェアプリフェッチャと干渉しうる
- 現代のCPU（特にApple M系やAMD Zen 4+）ではハードウェアプリフェッチャが非常に優秀になり、**ソフトウェアプリフェッチが不要になるケースが増えている**

### 10. Cache-Oblivious Algorithms — ハードウェアを知らずに最適化する

Frigo, Leiserson, Prokop, Ramachandran (1999) の画期的なアイデア: **キャッシュのサイズやラインサイズを知らなくても、任意のキャッシュ階層で漸近的に最適なキャッシュ性能を達成するアルゴリズム**。

代表例: **cache-oblivious 行列転置**
- 通常の行列転置は行優先と列優先のアクセスが衝突し、キャッシュミスが O(N²)
- 再帰的に行列を4分割し続け、サブ問題がキャッシュに収まるサイズになったところで自然に局所的アクセスになる
- キャッシュミス数: O(N² / B)（B = キャッシュラインサイズ）で最適

**なぜ重要か**:
- Cache-awareアルゴリズム（タイルサイズをキャッシュサイズに合わせる）は特定のハードウェアに最適化されるが、移植性がない
- Cache-obliviousアルゴリズムは**全レベルのキャッシュ階層で同時に最適**。L1, L2, L3, TLBすべてに適応
- 再帰的分割統治がこのクラスのアルゴリズムの主要パターン

**van Emde Boas レイアウト**: B木の各ノードをメモリ上に再帰的に配置し、任意のサブツリーアクセスがキャッシュフレンドリーになるレイアウト。Prokop (1999) の修士論文で提案。

### 11. 昨日のB木エントリとの接続

昨日学んだB木の「ノードサイズをディスクページに合わせる」設計は、今日のキャッシュライン設計と**同一の設計原理**のスケール違い:

| メモリ階層 | 転送単位 | データ構造の適応 |
|-----------|---------|----------------|
| L1–L3 キャッシュ | 64バイト（キャッシュライン） | T木、CSB+木（キャッシュ意識B+木）|
| DRAM → ディスクキャッシュ | 4KB–16KB（ディスクページ） | B+木のノード |
| ディスク → NVMe | 512B–4KB（セクタ/ページ） | B+木のリーフ |

**CSS-Tree / CSB+ Tree**: B+木のノードサイズをキャッシュラインに最適化した変種。Rao & Ross (1999, 2000) が提案。内部ノードからポインタを排除し（子は暗黙的に連続配置）、1キャッシュラインにより多くのキーを詰め込む。メインメモリデータベースで有効。

**昨日の「ファンアウト最大化」原則の一般化**:
- ディスクベース: ページサイズに合わせてファンアウト最大化 → B木
- キャッシュベース: キャッシュラインに合わせてファンアウト最大化 → CSB+木
- 共通原則: **転送単位あたりの有用データ密度を最大化せよ**

### 12. techドメインとの接続

**eBPF とキャッシュ意識**:
- eBPFマップ（BPF_MAP_TYPE_HASH, BPF_MAP_TYPE_LRU_HASH）はカーネル内のキャッシュフレンドリーなデータ構造。per-CPUマップ（BPF_MAP_TYPE_PERCPU_HASH）はfalse sharingを構造的に回避する設計
- XDP（eXpress Data Path）がカーネルバイパスで高速な理由の一つは、パケットバッファがキャッシュに載った状態で処理が完結するから
- eBPFプログラムのサイズ制限（100万命令）は命令キャッシュに収まることも意図

**gRPC / Protocol Buffers のバッファ管理**:
- Protocol Buffersのワイヤフォーマットはフィールドが連続してパックされ、デシリアライズ時のキャッシュライン利用効率が高い
- gRPCのゼロコピー最適化: データをコピーせずにカーネル→ユーザ空間で共有し、キャッシュの無駄なフラッシュを避ける
- Arenaアロケーション: Protocol Buffersのメッセージを連続メモリに割り当て、空間局所性を向上。フラグメンテーション防止とキャッシュ効率を同時に達成

## 気づき・洞察

- **「転送単位への最適化」は計算機科学の通底パターン**。ディスクページ（4KB）→ キャッシュライン（64B）→ SIMDレジスタ幅（32/64B）→ ネットワークMTU（1500B）。あらゆるレイヤーで「ハードウェアの転送粒度に合わせてデータを詰め込む」同じ設計判断が繰り返される
- **単純さが勝つ法則**: 置換ポリシーでRandomがPLRUに迫る、E状態1つの追加でMESIが劇的に効率化する、等。ハードウェア設計ではコンプレキシティのコスト（面積、消費電力、タイミング）が性能効果を上回ることが多い。ソフトウェア設計にも通じる教訓
- **B木の設計は「階層を跨ぐ最適化」の好例**。昨日のエントリではディスクI/O観点で理解したが、キャッシュ階層まで含めると、B木のノードサイズ選択は「メモリ階層全体の転送単位を意識した多段最適化」だと再解釈できる
- **False sharingの教訓**: 抽象化（高級言語の変数）と物理実体（キャッシュライン上の配置）のミスマッチが性能バグになる。これは「抽象化の漏れ（leaky abstraction）」の典型例

## 他分野との接続

- **ピアノ演奏**: 練習の「チャンキング」（小節単位で区切って練習）は、空間局所性の活用に似ている。脳の短期記憶のワーキングメモリ容量（7±2チャンク）が「キャッシュサイズ」、チャンク内の音符の連結が「キャッシュライン内の空間局所性」に対応する
- **パン発酵**: サワードウの発酵プロセスでは、酵母と乳酸菌が基質（糖分）の「局所的」な濃度に応じて活動する。キャッシュ階層のように、基質は均一に分布せず、パン生地内に濃度勾配がある。ミクロな局所性が全体の発酵パターンを決定する
- **ゴルフ**: コースマネジメントにおける「次の一打の選択肢を先読みする」はプリフェッチに似ている。良いプレーヤーは現在のショットを打つ前に、次のショットのために最適なポジションを投機的に計算している

- **NUMAアーキテクチャ**: メモリ階層にソケット間の距離が加わる。リモートメモリアクセスのレイテンシとキャッシュコヒーレンスの関係
- **GPUのメモリ階層**: 共有メモリ（scratchpad）、テクスチャキャッシュ、コアレスドアクセスパターン。CPU とは根本的に異なる設計思想
- **Persistent Memory（Intel Optane後の世界）**: メモリ階層に不揮発性層が加わった場合のキャッシュ設計への影響
- **cache-oblivious B木の実装**: van Emde Boasレイアウトを用いた実際のデータベースインデックスへの適用事例
