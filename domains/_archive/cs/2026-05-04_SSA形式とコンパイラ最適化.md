# SSA形式とそれに基づくコンパイラ最適化
**日付**: 2026-05-04
**分野**: cs
**タグ**: #compiler #SSA #optimization #register-allocation #IR #dominance-frontier

## 学んだこと

### 1. SSA形式の定義 — 「一度しか代入されない」が変える設計空間
**Static Single Assignment (SSA)** は、各変数が **静的に正確に一度だけ定義** される中間表現。動的な実行回数ではなく、ソース上の代入文の数を制約する。これにより各 use は唯一の def と対応し、**Use-Def chain が単一ポインタに退化** する（従来の dataflow では use→def は集合）。最適化パスは反復データフロー解析を必要とせず、def を直接辿ればよい — これが「sparse analysis」の基盤になる。

### 2. φ関数（phi function） — 合流点の value selection
分岐の合流点で複数の def が到達する場合、`x₃ = φ(x₁, x₂)` で **どちらの前任ブロックから来たかに依存して値を選ぶ** 仮想命令を挿入する。φ は**並列に**評価される（同一ブロックの全 φ は前任からの incoming 値を「同時」に取る）— これが後述の parallel copy 問題を生む。φ は実機命令ではなくIR上の仮想構造体で、out-of-SSA で copy命令に lowering される。

### 3. 支配 (Dominance) と 支配境界 (Dominance Frontier)
- **支配**: ノード d が n を支配 ⟺ entry から n への全パスが d を通る。`d dom n`。
- **直支配木 (Dominator Tree)**: 各 n の直支配ノード idom(n) を親とする木。
- **支配境界 DF(d)**: 「d が支配する領域の **すぐ外側** にある合流点」。形式的には `DF(d) = {n | d dom pred(n) ∧ ¬(d sdom n)}`（sdom は strict dominate）。φ は **DF に挿入** すべきという基本定理がある（Cytron 1991）。
- **Lengauer-Tarjan アルゴリズム (TOPLAS 1979)**: 支配木を **O(E·α(E,N))** で計算。α はAckermann 関数の逆関数で実用上ほぼ定数。simple 版は O(m log n)。Cooper et al. 2001 の iterative algorithm は実装が単純で、petgraph や LLVM の一部で採用される。

### 4. SSA構築アルゴリズム

**Cytron et al. (1991, TOPLAS)** の古典法（dominance frontier 法）:
1. 支配木と DF を計算（Lengauer-Tarjan）
2. 各変数 v の def を含むブロック集合の **iterated dominance frontier (DF⁺)** にφ挿入
3. 支配木を pre-order で巡り、stack ベースで rename（変数 v に対して新しいバージョン番号を採番）

ナイーブ版は最悪 O(N²) だが、worklist-based の linear-ish 実装で実用的に **almost linear**。LLVM の `mem2reg` が代表実装。

**Braun et al. (2013, CC)** のオンザフライ構築:
- DF を**計算しない**。AST/bytecode から直接 SSA を吐く。
- `writeVariable(var, block, value)` / `readVariable(var, block)` API を提供し、後者は前任を再帰探索。複数前任があれば φ を挿入。
- **incomplete CFG**（forward branch がまだ生成途中）を扱える点が革新 — JIT/IR builder に直接適合。**Trivial φ removal**（`φ(x, x) → x`）と **operand replacement** で minimal SSA を保証。
- LLVM の Cytron 実装より僅かに高速（SPEC CINT2000 計測）。Cranelift, libFirm, LuaJIT などが採用。

### 5. SSA に基づく最適化

| 最適化 | アルゴリズム | 効果 |
|---|---|---|
| 定数伝播 | **SCCP (Wegman-Zadeck 1991)**: lattice {⊤, const, ⊥} を SSA edge 上で worklist 伝播。**未到達ブロック**を `⊥` ではなく separately 扱うため、従来 CCP より多くの定数を発見 | ほぼ線形 (CFG + SSA edge 数) |
| 共通部分式除去 | **GVN (Alpern-Wegman-Zadeck 1988)**: 値番号で同値類分割。SSA だと def が一意のため hash-cons がそのまま機能 | RPVN, GVN-PRE, Click 1995 のhash-based |
| LICM | ループ不変式の同定が **def の支配関係**だけで決まる（オペランドが loop preheader を支配すれば不変） | 反復 dataflow 不要 |
| DCE | **Aggressive DCE**: 副作用ある命令と分岐を root に逆方向に必要命令を mark。SSA では def が一意のため到達不能 def は即削除可能 | O(命令数) |
| レジスタ割付 | 後述、SSA形式での干渉グラフが chordal | O(\|V\|·ω(G)) で最適彩色 |

### 6. SSAと依存表現の容易さ
- **データ依存**: use-def が単一ポインタ＝grafh 構造そのもの。
- **制御依存**: Cytron 1991 が **Control Dependence Graph (CDG)** を支配境界の dual として導出。**Program Dependence Graph (PDG)** は SSA + CDG で「データと制御の融合」表現。HotSpot C2 の **sea-of-nodes** はこの極致 — basic block を捨て、命令ノードがデータ/制御エッジで直結。

### 7. Out-of-SSA 変換（destruction）
φ を実機命令で実現するため、各前任ブロックの末尾に `copy` を挿入する素朴版（**Cytron-style**）には**lost-copy / swap problem** が起きる。

- **Briggs et al. (1998, SP&E)**: φ を per-predecessor の copy へ変換する際、coalescing の前に renaming を保ったまま安全に展開する手順を提示。
- **Sreedhar et al. (1999, SAS)**: **Method I/II/III** で φ-congruence class を計算し、congruence 内が干渉しない場合だけ copy を省略。**Parallel copy** （a←b ‖ b←a の swap）は scratch register もしくは XOR-swap で逐次化。
- **Boissinot et al. (CGO 2009) "Revisiting Out-of-SSA Translation"**: liveness を value ではなく **variable** で再計算し、最小コピーで正しい destruction を達成。LLVM の現行実装はこの系列。

### 8. SSA-based レジスタアロケーション
**Chaitin (1981)** は register allocation を graph coloring に帰着させ NP-完全と示した。が、**SSA 形式中の干渉グラフは chordal** という発見が革命を起こした。

- **Hack & Goos (IPL 2006)** および同時期の **Brisk, Bouchez** の独立証明: SSA プログラムの **支配関係が perfect elimination order を誘導** するため、干渉グラフは chordal。chordal graph は **多項式時間 O(\|V\|·ω(G)) で最適彩色** 可能（greedy on PEO）。
- これは **spilling・coloring・coalescing を decouple** できることを意味する（古典 Chaitin-Briggs では同時に解く必要があった）。
- **重要な caveat (Pereira-Palsberg 2006)**: classical SSA elimination *後* の register allocation は再び NP-完全。SSA-based RA は **destruction の前に色付けし、parallel copy を含む形で out-of-SSA する** 必要がある。
- **Linear scan (Poletto-Sarkar TOPLAS 1999)**: live interval を線形スキャンし O(V log R) で割付。chordal 性とは別系統だが JIT (HotSpot C1, V8 Crankshaft 旧) で採用。**Extended Linear Scan** (Wimmer 2010) が SSA に最適化され GraalVM, V8 で使用。

### 9. 実装事例
- **LLVM IR**: 純粋 SSA。memory は SSA 外（`alloca` + `mem2reg` で promote）。φ は `phi` 命令として明示。
- **GCC GIMPLE-SSA**: GCC 4.0 (2005) で導入。tree-SSA 表現で高位最適化、その後 RTL に lowering。
- **HotSpot C2**: **sea-of-nodes** (Click 1995, PhD)。basic block を「scheduling 後」まで持たず、データ依存と制御依存が一様グラフ。GVN が極めて自然に統合される。
- **MLIR (Lattner et al. CGO 2021)**: **multi-level + dialect** でSSAを核に置きつつ、affine/linalg/gpu など複数抽象レベルを単一 IR で表現。Mojo, IREE, TensorFlow XLA, CIRCT が採用。
- **Cranelift (Bytecode Alliance)**: Braun 2013 ベースのオンザフライ構築。Wasmtime のJITバックエンド。

### 10. SSA の限界と拡張
- **Gated SSA (GSA, Tu-Padua 1995, Ottenstein 1990)**: φ を γ (条件依存)、μ (loop entry)、η (loop exit) に分解。**条件情報を内包**し、PRE や thinned-GVN の精度向上。
- **Array SSA (Knobe-Sarkar 1998)**: 配列要素を element-level で SSA 化。要素の def に対し**実行時 φ-merge を表す δ 関数**を導入し parallelization 解析を可能に。
- **Concurrent SSA / CSSA (Lee-Padua-Midkiff 1999)**: マルチスレッドの shared memory を扱うため π/ψ 関数で同期点を表現。data race 検出と並列最適化に。
- **HSSA (Chow et al. 1996)**: alias を扱う μ/χ 関数。memory も SSA に乗せる試み（Open64 で採用）。

## 気づき・洞察

- **SSA の本質は「def-use の単一化による sparse analysis」**: 反復 dataflow（O(N²) になりがち）を、def を直接辿る O(use 数) に縮約する。これが SCCP / GVN / DCE が「ほぼ線形」になる構造的理由。
- **支配境界という幾何学**: φ を「どこに置くか」という問題が DF という graph-theoretic 不変量に reduce される美しさ。Cytron 1991 はこの幾何学的洞察そのものが革命だった。
- **Chordal graph の発見の衝撃**: Chaitin が25年間「NP完全」と封印した問題に、IR の選び方を変えるだけで多項式時間最適解が現れる。**問題の難しさは表現に依存する**という compiler 設計の根本教訓。
- **Sea-of-nodes と MLIR の対比**: C2 は basic block を捨てて極限まで自由度を上げた。MLIR は逆に**多層の構造化**で抽象を保つ。両極端がともに SSA を核に持つ点が示唆的。

## 他分野との接続

- **lock-free / MESI**: SSA の φ は「複数ソースから一つを選ぶ」 — ハードウェアの bus arbitration や cache coherence の選択論理と同型。HotSpot C2 のメモリ最適化は MESI semantics を前提とする。
- **Curry-Howard**: SSA は λ計算の **A-normal form (ANF)** とほぼ等価（Appel 1998 "SSA is Functional Programming"）。φ は continuation の引数渡しに対応し、CPS 変換と双対。
- **論理時計 / 一貫性モデル**: φ がどの前任から来たかは「分岐の因果関係」そのもの。Lamport happens-before が SSA dominance に対応。

## 次に深掘りしたいこと
