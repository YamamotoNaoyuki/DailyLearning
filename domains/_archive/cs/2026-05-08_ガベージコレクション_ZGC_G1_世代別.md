# ガベージコレクション — 世代別から ZGC まで
**日付**: 2026-05-08
**分野**: cs
**タグ**: #garbage-collection #ZGC #G1 #Shenandoah #JVM #colored-pointers

## 学んだこと

### GCの根本問題 — 「いつ・どのオブジェクトが死んでいるか」

GCの基本仕事は**「もはや到達不能なオブジェクトのメモリを回収すること」**だが、これを**安全かつ高速かつ低レイテンシ**で達成するのは難しい。トレードオフが多すぎる：

- **Throughput vs Latency**: 一気に全部回収すれば速いが、その間アプリは止まる（stop-the-world、STW）
- **Compaction vs Fragmentation**: 圧縮しないと断片化、圧縮はオブジェクト移動が必要
- **Generational vs Flat**: 世代仮説（弱い世代仮説：ほとんどのオブジェクトはすぐ死ぬ）は多くの場合正しいが、長期生存オブジェクトが多いシステムでは破綻
- **Concurrent vs Stop-the-world**: 並行GCはアプリと同時に走るが、ポインタ書き換えの整合性が難題

### Generational GC — 「弱い世代仮説」の活用

経験則：**ほとんどのオブジェクトは生成直後に死ぬ**（リクエスト処理中の一時オブジェクト、ループ内の局所変数 etc.）。一方、長期生存するオブジェクトは生き続ける傾向にある（キャッシュ、コネクション）。

世代別GCはこの統計的性質を利用：
- **Young Generation**: 新規オブジェクト。**Eden + Survivor space (S0/S1)**
- **Old Generation**: 一定回数のGCを生き残ったオブジェクト

Young GCは小さい領域を**頻繁に**スキャン（ms単位）、Old GCはまれに**全体を**スキャン（数百ms）。これでGC全体のコストが大幅に下がる。

**Card Table** という仕組みで、Old → Young への参照を効率的に追跡する：
- Old領域を **512バイト単位の "card"** に分割
- Old領域への書き込みがあった card に「dirty」マークをつける（**write barrier**）
- Young GCではdirty cardだけスキャンすれば良い

### G1 GC (Garbage-First) — リージョンベースの予測可能GC

JDK 9+のデフォルト。ヒープを**約2000個の同サイズリージョン**（1MB〜32MB）に分割：

- リージョンは動的に Young/Old/Humongous の役割を担う
- 各リージョンの**ガベージ密度**を統計で追跡
- 設定された pause time target（例: -XX:MaxGCPauseMillis=200）に収まるよう、**ガベージが多いリージョンから**回収（**Garbage-First**の名称由来）

G1の長所：
- 大ヒープ（>32GB）でも比較的予測可能なpause time
- Mixed GC（Young + 一部Old）で全体stop-the-worldを回避
- Concurrent marking phase（マーキングはアプリと並行）

G1の短所：
- それでも**evacuation phase（オブジェクト移動）はSTW**
- 100GB級ヒープでは pause が数百msに達することも
- 「**Concurrent**」を名乗るがリアルには「**Mostly Concurrent**」

### ZGC — Colored Pointers と Read Barrier

JDK 11+でexperimental、JDK 15で安定化、JDK 21でGenerational ZGC、**JDK 25 LTS（2025年9月）でデフォルト化**。

ZGCの設計目標：**ヒープサイズに依存しない sub-millisecond pause**。これを達成する2つの核心技術：

#### 1. Colored Pointers（着色ポインタ）
64-bit ポインタの**未使用上位ビット**にメタデータを埋め込む：
- 元々アドレス空間は48bit程度しか使われていない（残り16bit）
- ZGC は4bitを使い、ポインタの状態を表現:
  - **Marked0 / Marked1**: マークサイクル識別
  - **Remapped**: オブジェクトが移動済みか
  - **Finalizable**: ファイナライザ対象か

これにより**オブジェクトのメタデータをポインタ自体に持たせる**ことができ、別途ヘッダを操作する必要がなくなる（→ 並行性が大幅向上）。

#### 2. Load Barrier（読み込み障壁）
コンパイラが**ヒープからの参照ロードのたびに**barrier コードを挿入：

```
// 簡略化した擬似コード
ref = load_from_heap(addr);
if (ref.colored_bits != current_color) {
    ref = slow_path(ref);  // 移動済みなら新アドレスに更新、マーク等
}
return ref;
```

このバリアにより：
- Mutator（アプリ）が古いポインタを読んでも、即座に**healing**（修正）される
- 並行マーキング・並行コンパクションが可能
- ポインタ更新の同期が不要（バリアが lazy にやる）

JDK 21のGenerational ZGCで**世代別**にも対応。Young/Oldで異なるGCパラメータを使える。

### Shenandoah — Brooks Pointers の別解

Red Hat開発、ZGCとほぼ同時期。低レイテンシGCという目標は同じだが、別の実装：

- **Brooks Pointers**: 各オブジェクトの先頭に**自分自身への転送ポインタ**を持つ。GC中はこれを「移動先アドレス」に書き換える
- 全オブジェクトアクセスは**Brooks pointer経由の間接参照**
- ポインタを32bit/64bit丸ごと使えるので、ZGC のようなアドレス空間制約がない
- Red Hat版は**non-generational**（Amazon Corretto では generational 化）

ZGCとの比較：
| 特性 | ZGC | Shenandoah |
|------|-----|-----------|
| メタデータ | colored pointer (上位bit) | Brooks forwarding pointer |
| 世代 | Yes (JDK21+) | 実装次第 |
| 64bitアドレス | 必須 | 不要 |
| 移植性 | ARMv8+ x86-64 | 広い |
| pause目標 | <1ms | <10ms |

### 何を選ぶか

- **小〜中ヒープ（〜8GB）、throughput優先**: G1（デフォルト）か Parallel GC
- **大ヒープ（〜32GB）、予測可能なpause**: G1
- **超大ヒープ（>32GB）、低レイテンシ必須**: **ZGC**（JDK 21+）
- **メモリ制約環境（Windows on ARM等）**: Shenandoah
- **バッチ処理、レイテンシ無関係**: Parallel GC（throughput最大）

2026年現在、**JDK 25 LTS で Generational ZGC がデフォルト**で、これがマイクロサービス・大規模アプリの新標準。

## 気づき・洞察

GC設計の歴史は **「stop-the-world をいかに減らすか」** の連続だった。Mark-and-Sweep (1960) → 世代別 (1980s) → Concurrent Mark Sweep (CMS, 2000s) → G1 (2010s) → ZGC/Shenandoah (2020s)。各世代で**「並行性をどう拡大するか」**という同じ問題を別角度から解いている。

ZGCの **Colored Pointer** は **「データ構造を実装の制約として活用する」** 最高の例。ポインタを「ただのアドレス」と見るか「アドレス+メタデータ」と見るかで設計の自由度が変わる。これは **eBPF が BPF program type をポインタ的に持つ** のと同じ思考。**型情報・状態情報を data path に乗せる**という発想。

「**Read Barrier vs Write Barrier**」のトレードオフは興味深い。
- Write Barrier: 書き込み時に作業（読み込みは速い、書き込みコスト）。Card Table はこれ
- Read Barrier: 読み込み時に作業（書き込みは速い、読み込みコスト）。ZGC はこれ

ハードウェアの傾向として**読み込みのほうが頻度が高い**ので、Read Barrierは一見不利だが、CPUのbranch predictorで「ほぼ常にノーマルパス」になる場合は事実上ゼロコストにできる。**ハードウェアの統計的振る舞いを前提にした設計**。

「**世代別仮説**」は経験則であって**普遍真理ではない**。LRUキャッシュ、コネクションプール、メッセージキューなど**長期生存オブジェクトが大半**のシステムでは世代別GCが裏目に出る（promotion overhead が大きい）。Shenandoahが non-generational を選んだのはこの認識から。**「弱い世代仮説」が破れる workload では別のGCを選ぶ**判断が必要。

## 他分野との接続

- **bread (サワードウのマイクロバイオーム動的平衡)**: GCがメモリを「動的に回収・再配置」するのと同じく、サワードウのLAB:酵母比は動的平衡。**短命なオブジェクト（マルトース→グルコース）と長命なオブジェクト（菌体）**の世代別管理が自然に起きている
- **anatomy (横隔膜の予測収縮)**: ZGCの read barrier が ほぼ常にfast path になるよう設計されているのは、横隔膜が動作の30ms前に予測収縮するのと同じ「**統計的優位パスへの最適化**」
- **piano (ハーフペダル = 連続パラメータ)**: GC pause time target (-XX:MaxGCPauseMillis) はまさに**連続パラメータ化された制御**。Penal/Strategic を二値で考えず連続で混合できるのと同型
- **golf (DECADE = 統計的最適化)**: ZGC の generational 化は「**多くのオブジェクトはすぐ死ぬ**」という統計を活用。DECADE の「**統計的に良いショットを選ぶ**」と同じデータドリブン設計
