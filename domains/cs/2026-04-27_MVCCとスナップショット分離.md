# MVCCとスナップショット分離 — 「読み手は書き手をブロックしない」設計
**日付**: 2026-04-27
**分野**: cs
**タグ**: #MVCC #snapshot_isolation #PostgreSQL #Oracle #serializability #2PL #VACUUM #write_skew

## 学んだこと

### MVCC の根本問題 — Reader/Writer Lock の限界
データベースの古典的な並行制御は **2-Phase Locking (2PL)** に依存していた——共有ロック (shared, S) と排他ロック (exclusive, X)、phantoms 防止のための predicate locking。しかし 2PL には致命的な性能問題がある:

- **読み手が書き手をブロックする** (S ロック保持中は X 取れない)
- **書き手が読み手をブロックする** (X ロック保持中は S 取れない)
- 結果として**長時間 SELECT が UPDATE 全体を停止**——OLTP では致命的

**MVCC (Multi-Version Concurrency Control)** はこの問題を構造的に解消する: **書き込みは新バージョンを作成し、古い版は読み手のために保持する**。**読み手は書き手をブロックしない、書き手は読み手をブロックしない**——これが MVCC の核心スローガン。

### 起源と理論的基盤
**Phil Bernstein** と **Nathan Goodman** が 1981 年の論文 *"Concurrency Control in Distributed Database Systems"* (ACM Computing Surveys) で MVCC の理論基盤を整理し、David Reed の MIT博士論文 (1978) が時間ベース版の先行研究。最初の本格実装は **InterBase** (1984, Jim Starkey)、続いて **Oracle 7** (1992) が採用。**PostgreSQL** は 1994 年の Postgres95 から MVCC ベースとなった (Stonebraker の Postgres は当初から版管理を持つ設計だった)。

### バージョン管理の実装 — PostgreSQL の場合
PostgreSQL では各タプル (行) が以下のメタデータを持つ:

```
xmin: このタプルを INSERT/UPDATE したトランザクション ID
xmax: このタプルを DELETE/UPDATE したトランザクション ID (まだの場合 0)
cmin/cmax: 同一トランザクション内のサブシーケンス
ctid: 物理的位置 (block, offset)
```

UPDATE は**新タプルを INSERT し、古いタプルの xmax に現トランザクション ID を書く**——古いタプルは物理的に消去されない。これが MVCC の物質的基盤。

### Snapshot — 「いつの世界を見るか」
トランザクション開始時、PostgreSQL は以下を含む **snapshot** を生成:
- `xmin` (現在 active な最小 XID)
- `xmax` (これから割り当てられる XID)
- `xip[]` (現在 active な XID のリスト)

タプルが**自分のsnapshot に可視か**は以下の規則で判定:

1. `xmin < snap.xmin` AND `xmin ∉ snap.xip[]` → **コミット済みかつ snapshot 開始前** → 可視
2. `xmin == 自分の XID` → 自分が作った → 可視
3. `xmax == 0` OR `xmax > snap.xmax` → まだ削除されていない (snapshot 視点で) → 可視

このロジックがすべてのタプル参照に適用される。**結果として、各トランザクションは「snapshot 取得時点の整合的なデータベース全体」を見る**。

### Snapshot Isolation (SI)
**Snapshot Isolation** とは、各トランザクションが「**その開始時点の snapshot**」のみを参照する分離レベル。SI の保証:
- **Read 操作は snapshot を見る** (途中で他トランザクションがコミットしても変わらない)
- **Write は first-committer-wins**: 同じ行を 2 つのトランザクションが更新しようとすると、後にコミットしようとした方が**競合検出で abort**

これは ANSI SQL の **REPEATABLE READ** より強く、**SERIALIZABLE** より弱い分離レベル。Berenson et al. (*"A Critique of ANSI SQL Isolation Levels"*, SIGMOD 1995) が**ANSI 標準の不備**を指摘し、SI を独立した分離レベルとして体系化した。

### Write Skew — SI の有名な落とし穴
SI は**ほぼ Serializable に見える**が、**Write Skew (書き込み歪曲)** という異常を許す。古典例 ("doctors on call"):

```
制約: 常に 1 人以上の医師が on-call でなければならない
状態: Alice と Bob が on-call、二人とも off にしたい
```

T1: 「Bob が on-call」を確認 → Alice を off に
T2 (並行): 「Alice が on-call」を確認 → Bob を off に
両方 commit → **誰も on-call でなくなる** (制約違反)

T1 と T2 はそれぞれ自分の snapshot で「もう一人いる」と認識しているが、**互いの更新を見ていない**ため両方が「自分は off にしてよい」と判断する。**SI では同じ行への競合は検出するが、異なる行への論理的依存は検出しない**——これが SI と Serializable の本質的差異。

### Serializable Snapshot Isolation (SSI)
**Michael Cahill** らが PhD 論文 (2008) で提案した **SSI (Serializable Snapshot Isolation)** は、SI に**読み書き依存グラフ (rw-conflict graph)** の追跡を加え、**サイクルが検出されたらトランザクションを abort** する。これにより SI のパフォーマンスを保ちつつ Serializable を達成する。

PostgreSQL は **9.1 (2011)** で SSI を実装。Cahill のアルゴリズム (**Dangerous Structure** 検出) により、 saving の必要なケースのみを正確に abort——**過剰な保守性を避ける**。商用DBで本格的にSerializable がスケールする初の例となった。

### Tuple の死蔵問題と VACUUM
MVCC は古いバージョンを保持するため、**死んだタプル (dead tuple) が累積する**。何もしないと:
- ディスク容量肥大
- インデックスがdead tupleを指す → I/O 増
- XID は 32bit (約 21億) → wraparound でデータ消失リスク

PostgreSQL は **VACUUM** プロセスで死んだタプルを物理的に回収する。以下が定期実行される:
- **autovacuum**: バックグラウンドプロセス、頻繁に小規模回収
- **VACUUM FULL**: テーブル全体を再書き込み (排他ロック必要、運用稀)
- **VACUUM FREEZE**: 古い XID を「凍結」し wraparound を防ぐ

VACUUM の遅延は PostgreSQL 運用の代表的な問題——**テーブルが肥大化 (bloat) してパフォーマンスが急落**する。Uber が 2016 年に PostgreSQL から MySQL に移行した有名な記事 (*"Why Uber Engineering Switched from Postgres to MySQL"*) は、VACUUM 周辺の問題が大きな動機だった。

### Oracle vs PostgreSQL のアーキテクチャ差
| | Oracle | PostgreSQL |
|---|---|---|
| 古いバージョンの保管 | **UNDO segment** (専用領域) | **同じテーブル内** に残す |
| 回収 | UNDO の循環使用 (ORA-01555 "snapshot too old") | **VACUUM** プロセスが必要 |
| Snapshot 取得 | SCN (System Change Number) ベース | XID + Snapshot 構造体 |
| 長時間 SELECT のリスク | UNDO 枯渇で **読み取り失敗** | dead tuple 蓄積で **書き込み遅延** |

両者は同じ MVCC 思想だが、**実装トレードオフが対照的**。Oracle は読み手保護のために書き手にコストを払わせ、PostgreSQL は逆。

### 他DB の MVCC 実装
- **MySQL InnoDB**: undo log + MVCC、**REPEATABLE READ がデフォルト**で SI 相当 (但し**phantoms あり**)、SERIALIZABLE は Next-Key Locking で実装
- **MS SQL Server**: 2005 年から **READ COMMITTED SNAPSHOT** と **SNAPSHOT ISOLATION** モードを追加
- **CockroachDB / TiDB / Spanner**: 分散型、**HLC (Hybrid Logical Clock)** または **TrueTime** で snapshot を分散合意
- **YugabyteDB**: PostgreSQL ワイヤ互換だが内部は **Raft + MVCC**
- **DuckDB**: 単一ノード OLAP、**optimistic MVCC** で軽量実装

### 分散 MVCC — Spanner の TrueTime
Google Spanner (Corbett et al., OSDI 2012) は **TrueTime API** を使って**分散SI**を実現:
- すべてのトランザクションがコミット時刻 $t$ を持つ
- TrueTime が「**現実時刻が間違いなく $t$ より進んだ**」と保証してから commit を可視化
- これにより**外部一貫性 (external consistency)** = 「**ある transaction の commit を観測した後の transaction は、必ず後の commit time を持つ**」が成立

TrueTime は**GPS + 原子時計**でクロック誤差を 7ms 以下に抑える——**ハードウェアで時間の不確実性を有界化**することで、分散MVCC の理論的基盤を作った。

### Read Committed vs Snapshot Isolation — 微妙な差
PostgreSQL のデフォルトは **READ COMMITTED**——SI とは違い、**各 SQL 文ごとに新しい snapshot**を取る。同じトランザクション内の 2 回の SELECT で**異なる結果が見える可能性**がある (non-repeatable read)。

```sql
BEGIN;
SELECT balance FROM accounts WHERE id=1;  -- 100 が見える
-- (他トランザクションがコミット)
SELECT balance FROM accounts WHERE id=1;  -- 150 が見える
COMMIT;
```

SI (= REPEATABLE READ in PostgreSQL) では**両方とも 100** が見える。**多くのアプリは READ COMMITTED で十分**だが、analytics や金融処理では SI 以上が必須。

## 気づき・洞察

### 「ロックを増やす」ではなく「版を増やす」
MVCC の本質的な発想転換は「**競合を順序化する**」(2PL) から「**競合を発生させない**」(MVCC) へ。**競合の発生源を物理的に分離する**——古い版と新しい版を別物として保持することで、**読み手と書き手が論理的に交差しない**。これは並行性問題の構造的解決の典型例で、CRDT (合流半束で競合を不可能化) や Functional Programming (不変データで mutation を避ける) と**同じ哲学的家族**。

### 「snapshot」=「時間そのものを物質化する」
MVCC は「**ある瞬間のデータベース全体**」を物質的に保持する装置——時間を**離散的なバージョン**に切り出して並列保管する。これは Git が**コミットというsnapshotを保持する** のと完全同型。**現在状態は最新版の合計、歴史は版の連鎖**——イベントソーシングが「**履歴を本体、状態を導出**」とするのに対し、MVCC は「**最新状態を本体、履歴を版で保持**」する逆向き戦略だが、**時間を物質化する**点では同型。

### Write Skew — 「同期しない読み」の罠
SI が「ほぼ Serializable」に見えて Write Skew を許すのは、**同じ snapshot 内での読みは整合的だが、トランザクション間の読み書き依存は追跡されない**ため。これは**並行プログラミングの根本問題**——**個別ノードでは正しいが、ネットワーク全体では整合しない**。CAP 定理が示す「**整合性と可用性のトレードオフ**」のミクロ版が、データベース内部の MVCC レベルでも生じる。

### SSI = 「依存グラフを動的に構築する」
Cahill の SSI は**rw 依存グラフを実行時に構築し、サイクルを検出**する——これは**動的型検査が静的解析の代替**を担うのと同じ。**Serializable を保証するために事前に全順序を決める必要はなく、依存を観察して矛盾だけを断つ**。PostgreSQL 9.1 で実装されて以降、ほとんど性能オーバーヘッドなく Serializable が使えるようになった——**理論と実装の貴重な融合点**。

### VACUUM = 「ガベージコレクション as 運用問題」
MVCC は GC を必要とする (古い版を回収しないとディスクが破裂する)。これは**プログラミング言語の GC** と本質的に同じ問題——**メモリ・ディスクの解放を実行時に追跡**する。Uber の事例は「**運用上の GC 問題が言語選択を覆す**」典型例で、これは Java/Python アプリで GC pause が問題になるのと同じ構造。**理論的に美しい設計が運用で破綻する**ケースの教訓。

### 分散 MVCC は時間の物理学に到達する
Spanner の TrueTime は「**ソフトウェアで分散整合性を保証するには、ハードウェアで時間の精度を保証する必要がある**」ことを示した。これは Lamport clock や vector clock といった**論理時計**の限界——**実時間そのものを正確に共有しないと外部一貫性は得られない**。これが GPS/原子時計に到達する。**分散DB が物理学の領域に踏み込んだ瞬間**。

## 他分野との接続

- **cs (LSM ツリー)**: LSM の SSTable は不変版を順次追加する書き込みパターン——MVCC のタプル追記と同じ。**「上書きせず追記する」**設計哲学は同じ家族 (CRDT, イベントソーシング, Git も)
- **cs (Hindley-Milner型推論)**: SSI の動的依存グラフ構築は**型推論の制約解決**と同型——実行時に矛盾を検出して落とす。**動的解析が静的決定を肩代わりする**点が共通
- **cs (NP完全性)**: Serializable の判定は**サイクル検出**で多項式時間。MVCC が Serializable を達成できるのは**問題が NP 困難でないから**——**並行制御問題の計算複雑性**を逆手に取った設計
- **tech (eBPF verifier)**: verifier の「**全分岐の状態追跡**」は MVCC の「**全 active トランザクションの snapshot 保持**」と類比的——並列状態の一貫性管理
- **tech (io_uring SQ/CQ)**: 共有メモリリングを用いた SPSC キューは**ロックフリー**——MVCC が「**読み手と書き手の物理的分離**」を狙うのと同じ並行制御の解
- **piano (Chaffinの暗譜と再構築)**: 「**最新の演奏状態を保持しつつ、過去の準備状態を参照可能に**」——演奏は MVCC の snapshot と暗譜の retrieval structure の両立
- **bread (湯種法のデンプン糊化)**: 「**水を構造に閉じ込めて時間に対する変化を抑える**」のと、MVCC の「**snapshot を保持して時間に対する一貫性を確保**」は**時間-構造関係の同型性**
