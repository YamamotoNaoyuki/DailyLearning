# MESIプロトコルとメモリ整合性モデル — マルチコア時代の「正しさ」の意味
**日付**: 2026-04-30  
**分野**: コンピュータサイエンス  
**タグ**: #MESI #CacheCoherence #MemoryConsistency #TSO #SequentialConsistency #ReleaseAcquire

## 学んだこと

### 整合性 (coherence) と一貫性 (consistency) の区別

マルチコアアーキテクチャを語るとき、**cache coherence** と **memory consistency** はしばしば混同されるが、**異なるレイヤーの保証**である。

- **Coherence**: **同じメモリアドレス**に対する複数キャッシュ間の整合性。「アドレスXを読むと、最後の書き込みの値が読める」（単一アドレスの逐次的振る舞い）。
- **Consistency**: **異なるメモリアドレス**間の操作順序の保証。「スレッドAの書き込み順序を、スレッドBがどう観測するか」（プログラム全体の振る舞い）。

MESI が解決するのは coherence。Consistency は ISA レベルの memory model（TSO, ARM, RISC-V WMO 等）と、それを露出するプログラミング言語の memory model（C++11 / Java / LLVM IR）の階層で扱う。**両者は独立しつつ、相互に依存する**——consistency model が coherence の上に構築される。

### MESI: 4状態でcoherence を保証する

Illinois protocol (1984) として誕生し、x86 / ARM / RISC-V で広く採用される MESI は、各キャッシュラインに4状態を持たせる：

| 状態 | 意味 | 他キャッシュ | メモリと一致 |
|------|------|--------------|--------------|
| **M (Modified)** | 唯一の所有者、変更済み (dirty) | 全部 Invalid | NO（ライトバック必要） |
| **E (Exclusive)** | 唯一の所有者、未変更 (clean) | 全部 Invalid | YES |
| **S (Shared)** | 複数キャッシュが保有、未変更 | 同じく S | YES |
| **I (Invalid)** | 無効 | — | — |

**状態遷移の核**：

- 読み取りミス: 他キャッシュが M状態 → そのキャッシュがメモリへ書き戻し、両者 S へ。E or S → 自分も S。誰も持たず → メモリから読んで E へ
- 書き込みミス: **BusRdX (read for ownership)** をブロードキャスト。他キャッシュは I へ遷移、自分は M 取得
- 書き込みヒット (S→M): **BusUpgrade** で他者を I に追い出し、M へ昇格
- ライトバック: M ライン eviction 時にメモリへ書き戻し、I へ

**E 状態の存在意義**: M と S の間に「I'm alone, but clean」という状態を置くことで、**S→M の昇格時に bus トランザクションが必要**だが、**E→M は局所遷移のみ**で済む。シングルスレッドプログラムが大半を占める実世界で、これが性能を決定的に向上させる。Illinois protocol のオリジナルは MSI（3状態）だったが、Intel が E を加えた MESI が事実上の標準に。

### 拡張: MOESI / MESIF

- **MOESI** (AMD): O = Owner。M ライン共有時に「**メモリには書き戻さず、Owner が真の値を保持**」する。クロスコア共有データの書き戻しコストを削減。Opteron 以降の AMD x86, ARMv8 採用
- **MESIF** (Intel): F = Forward。S 状態の中の1つを「**この共有コピーが応答役**」と指定し、複数キャッシュからの応答衝突を排除。Intel Core i7 / Xeon

### Bus Snooping vs Directory-based

MESI の物理実装は2系統:

- **Snooping**: 全キャッシュがバスをリッスンし、関連トランザクションを検知して状態更新。**O(N²) スケール**——コア数 N が増えると bus traffic が二乗で増加。8-16 コア程度までが実用上限
- **Directory-based**: メモリ側 (or 専用ディレクトリ) が「どのキャッシュが持っているか」のビットベクトルを保持。Snoop なしで個別通知。**O(N)スケール**だが directory 自身が bottleneck になりうる
- **Hybrid**: **Distributed directory** (Intel Mesh Interconnect, AMD Infinity Fabric) は L3 をスライスし、各スライスがそのアドレス範囲の directory を保持。物理的にも論理的にも分散

2026年の主要 CPU (Intel Sapphire Rapids, AMD Zen 5, Apple M5) はすべて distributed directory ベース。MESI/MOESI の状態機械はそのまま生きているが、**bus snooping ではなく point-to-point メッセージ**で実装される。

### False Sharing: MESIの暗黒面

MESI の granularity は**キャッシュライン (64 バイト x86, 128 バイト ARM の一部)**。複数の論理的に独立した変数が同じキャッシュラインに乗ると、**片方を書き換えただけで他方も invalidated される**。

```c
struct {
    int counter_a;  // CPU 0 が書く
    int counter_b;  // CPU 1 が書く
} __attribute__((packed)) shared;
```

CPU 0 の `counter_a++` が CPU 1 のキャッシュを invalidate し、CPU 1 が `counter_b` を読むたびに miss が発生。**論理的には独立な変数が、物理レイアウトによって 100x 性能低下**を引き起こす。対策は `alignas(64)` で **cache-line padding**。Linux カーネルは `__cacheline_aligned` マクロを多用、Java は `@Contended` アノテーション、Rust は `crossbeam` の `CachePadded<T>`。

これは **「論理的抽象化が物理的現実を完全には隠蔽しない」** 古典的事例で、性能チューニングの永遠のテーマ。

### Memory Consistency Model: 「正しさ」の階層

異なる ISA は異なる memory consistency 保証を提供する：

#### Sequential Consistency (SC) — Lamport 1979

「全スレッドの操作を**何らかの全順序**に並べたとき、各スレッドのプログラム順は保たれる」。最も強い保証で、**直感的に推論しやすい**。が、現実のCPUは SC を**そのまま実装するとパイプラインが大幅に non-utilized**になるため、ほぼ採用されない（学術的標準モデル）。

#### Total Store Order (TSO) — x86 / SPARC

**store buffer**（書き込みキューを ROB から切り離し、バックグラウンドでキャッシュに書く）を許容するため、**store→load の並び替えのみ**を許す。古典的な store buffering 例:

```
Initial: x = 0, y = 0
Thread 1: x = 1; r1 = y;
Thread 2: y = 1; r2 = x;
```

SC では `r1=0, r2=0` は不可能。TSO では**両 store がそれぞれの store buffer に滞留したまま load が先行**するため、`r1=0, r2=0` が観測できる。x86 で実際に再現される驚き挙動。

**TSO が x86 で採用されている理由**: 大半のアプリケーションでは「同じスレッド内の store→load」の並び替えしか起きず、これは性能向上が大きく、安全性の影響は限定的（同期プリミティブを正しく書けば保証される）。

#### Weak Memory Order (WMO) — ARM, RISC-V, POWER

ほぼあらゆる並び替え (load/load, load/store, store/load, store/store) を許す。**最大の性能** だが、プログラマは fence (ARM の `dmb`、RISC-V の `fence rw,rw`) を**手で挿入**しなければならない。Apple が ARM 移行で「x86 互換 TSO モード」(M1のTSO mode for Rosetta) を提供したのも、x86 ソフトウェアが TSO を暗黙の前提としているから。

### 言語レベル: C++11 Memory Model

ハードウェア memory model は ISA 依存——だがアプリケーション開発者は**ポータブルなコード**を書きたい。C++11 (2011) は **memory_order_** ファミリーで言語標準を確立：

| memory_order | ハード命令 (x86 / ARM 想定) | 用途 |
|--------------|----------------------------|------|
| `seq_cst` | x86: 通常のmov + mfence、ARM: load-acq+ store-rel + dmb | デフォルト・最強 |
| `acquire` | x86: 通常のmov、ARM: ldar | 読みロック |
| `release` | x86: 通常のmov、ARM: stlr | 書きロック解放 |
| `acq_rel` | acquire + release | RMW (CAS等) |
| `consume` | データ依存の伝搬を保証 | (実用上 acquire と統一されつつある) |
| `relaxed` | 何の同期も保証しない | カウンタ等 |

**Release-Acquire ペアリング**:
スレッドA が `store(release)`、スレッドB が `load(acquire)` でその store を観測すると、**A の release より前のすべての書き込みが、B の acquire 以降から見える**。これが lock-free データ構造の基盤——Mutex は内部的に release-acquire で実装される。

### x86 / ARM 上の C++11 mapping

```cpp
std::atomic<int> x{0}, y{0};
// Thread 1
x.store(1, std::memory_order_release);

// Thread 2  
int v = y.load(std::memory_order_acquire);
```

x86 では `store(release)` は通常の `mov`、`load(acquire)` も通常の `mov`——TSO がほぼ acquire-release を提供するため fence 不要。

ARM では `store(release)` は `stlr`（store-release register）、`load(acquire)` は `ldar`（load-acquire register）の専用命令を使う。これにより**コンパイラが移植性とパフォーマンスを両立**する。

### data race と UB: 言語仕様の本質

C++11 / Java memory model の天才的ポイントは **「data race を起こすプログラムの動作は未定義 (UB)」** と決めたこと。これにより:

- **コンパイラは `seq_cst` でないアクセスを大幅に最適化できる**: data race がないと仮定してメモリ順序を入れ替え可
- **race を起こすコードは「合法的に」あらゆる値を返してよい**: hardware の弱い memory model が露出しても言語標準違反ではない
- **正しい同期を書けば、TSO 上でも ARM 上でも同じ振る舞いを保証**

Java は「**Java Memory Model**」（JSR-133, 2004）で**よりプログラマフレンドリー**なモデルを採った——data race があっても "out-of-thin-air values"（無から生じる値）は禁止し、最低限の安全性を保つ。これがJVM 実装の複雑さを増やすが、安全な言語というブランディングを支えた。

C++ は Java よりも**スピード優先**を選んだが、近年（C++17 以降）データレース UB の厳格性が問題視され、**C/C++11 memory model** の実用的修正案 (Lahav et al. 2017 "Repairing Sequential Consistency in C/C++11") が議論されている。

## 気づき・洞察

最も洞察的なのは、**「正しさ」が階層化されている**事実だ。MESI はキャッシュ間の単一アドレス coherence を保証する——これは**「同じ場所に書いたら同じ場所から読める」**という最低限の常識。memory consistency model はその上に **「異なる場所への書き込みの順序が観測できる」**保証を重ねる。プログラム言語 memory model はさらに上で**「同期プリミティブを正しく使えば、ハードウェアの差異から守られる」**を提供する。

この**抽象化階層は分散システムの consensus protocol と同型**だ。Raft の log replication は「単一エンティティの append-only 順序」を保証し、その上に**linearizability（外部一貫性）**が乗り、さらに**transaction isolation level（serializable, snapshot isolation）**がアプリ層に提供される。**メモリは小規模分散システム**であり、**分散システムは大規模メモリ**——両者の理論は深く結びついている。

もう一つの洞察は、**「TSO/WMO の選択は文化を作る」**点。x86 は強い TSO のおかげで「適当に共有変数を読み書きしても多くの場合動く」コードが20年蓄積された。ARM への移行は、この**暗黙の前提を破壊**する——Apple の M1 が x86 アプリ用に **TSO mode** を実装したのは、技術的判断ではなく**社会的判断**だった。「ハードウェアは社会的構造の物質化」という観点で見ると、TSO/WMO 論争は技術論争のふりをした **legacy software の保護論争**だ。

そして **C++11 memory model** が **40年間の野生 multicore C プログラム を救済する標準** であった事実は、**言語仕様の遅延更新**が技術的負債を増やす歴史的教訓。C++11 以前、MFC や Boost.Thread や POSIX Threads はそれぞれ別の memory model を提供し、ABIが互換でなかった。**統一仕様の存在自体が並行プログラミングの実用化を可能にした**。

## 他分野との接続

- **tech (CRDT, Iceberg)**: CRDT の **eventual consistency** は memory model の「弱い consistency」と同じ概念の分散版。MESI の M→S 遷移は CRDT の merge 操作と同型——両者とも**「全ノードが最終的に同じ値に収束」**する保証を異なる粒度で提供。
- **golf (D-Plane)**: D-Plane の「フェース・パス divergence が結果を決める」と、MESI の「複数キャッシュの divergence が coherence で解消される」は、**「独立した変数の差分管理」**という同じ抽象。
- **anatomy (神経筋接合部)**: 神経が筋に伝える signal も**release-acquire 的な同期**を持つ——シナプス前膜での神経伝達物質 release と、後膜での acquire (受容体結合)。生物の信号伝達と CPU の同期プリミティブの構造類似は深い。
- **piano (mental practice)**: 暗譜の四記憶 (kinesthetic, auditory, visual, analytical) は**同じ記憶を異なるキャッシュに格納**する分散構造で、MESI の冗長性と同型。1つが invalidate されても他から復元可能。
- **bread (lipid - water emulsion)**: 油脂と水の界面が安定するように、**強い consistency と弱い consistency** はソフトウェアアーキテクチャ層で**分離して共存**する。トレードオフを物理界面で解決する。
