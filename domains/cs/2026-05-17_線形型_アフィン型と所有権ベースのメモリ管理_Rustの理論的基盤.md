**分野**: cs
**タグ**: #線形型 #アフィン型 #Rust #型システム #リソース管理

## 学んだこと

### 線形論理 (Linear Logic) の発端

**線形論理 (Linear Logic)** は 1987 年に Jean-Yves Girard（パリ第七大学）が発表した、古典論理・直観主義論理の拡張。中心アイデアは **「資源としての命題」** — 古典論理では P を一度証明すれば P を何度でも使えるが、線形論理では **「P は使うと消費される」**。

例えば：
- 古典論理: 「100 円を持っている」→ コーヒーを買える、お茶を買える、コーラを買える（すべての結論を同時に導ける）
- 線形論理: 「100 円を持っている」→ コーヒーを買えば 100 円は消費される、お茶を買うことはできない

線形論理は **「使用回数」を厳密に管理する** という観点を論理に導入した。これがプログラミング言語の **リソース管理（メモリ、ファイル、ソケット）** に応用可能と認識され、1990 年代以降の型システム研究の中心テーマになった。

### Linear Types vs Affine Types — 「ちょうど 1 回」と「最大 1 回」

線形論理から派生した二つの型の概念：

- **線形型 (Linear Types)**: 値を**ちょうど 1 回**使用しなければならない（使用しない・複数使用も禁止）
- **アフィン型 (Affine Types)**: 値を**最大 1 回**使用できる（使用しない・1 回使用は OK、複数使用は禁止）

両者の差は **「消費しないことを許すか」**。アフィン型は値を「捨てる」ことを許容するため、実用言語ではアフィン型が主流。Rust の所有権システムは厳密にはアフィン型に基づく。

- 線形型を採用する言語: Linear Haskell（GHC 9.0+ の `LinearTypes` 拡張）、Idris2（部分的）、Clean
- アフィン型を採用する言語: **Rust**、ATS、Mercury、F* の一部の機能

### Rust の所有権モデル — アフィン型の実用化

Rust（Mozilla 由来、2010 年最初の公開、2015 年 1.0）は **アフィン型システムを実用言語に統合した最初の主流言語**：

```rust
let s1 = String::from("hello");
let s2 = s1;  // s1 から s2 への所有権の「move」
// println!("{}", s1);  // ❌ コンパイルエラー: s1 はもう使えない
println!("{}", s2);  // OK
```

このコードでは `s1` は `s2` に **move** され、**s1 は無効化** される。これは線形/アフィン型システムの直接の表現。コンパイラが「同じリソースを 2 度使う」ことを **静的に禁止** する。

### Copy vs Move — 二つの値の挙動

Rust のすべての型は次のいずれか：

- **Copy 型**: 値が「コピー」される。プリミティブ型（`i32`, `bool`, `f64`, タプルが Copy のみで構成）。**アフィン制約から免除**
- **Move 型** (デフォルト): 値が「移動」される。`String`, `Vec<T>`, `Box<T>`, ファイルハンドル、ネットワーク接続など、**所有権を持つ型**

`Copy` を実装する型は **「複数回使ってもよい」** という暗黙の許可。これは線形型システムの **「指数 (! 演算子)」** に対応する — 線形論理では `!P` で「P を何回でも使ってよい」を表現する。

### Borrowing と Lifetimes — 「貸し出す」セマンティクス

純粋なアフィン型システムは厳しすぎる — 関数に値を渡すと所有権が移動し、呼び出し元では使えなくなる。これを緩和するのが **借用 (borrowing)**：

```rust
fn calculate_length(s: &String) -> usize {  // & で「借用」
    s.len()
}

let s = String::from("hello");
let len = calculate_length(&s);  // s を貸し出す
println!("{}: {}", s, len);  // s はまだ使える
```

借用は **「所有権を移動せず、参照を一時的に貸し出す」** メカニズム。借用には二種類：

- **不変借用 (&T)**: 読み取り専用、複数同時に存在できる
- **可変借用 (&mut T)**: 書き込み可能、**同時に 1 つだけ**存在できる

これが Rust の有名な **「不変参照は複数 OR 可変参照は 1 つだけ」** ルール（aliasing-xor-mutability）。これによりデータ競合を **コンパイル時に防ぐ**。

**ライフタイム** は借用がどれだけ生きるかを追跡する。借用元より借用先が長く生きてはいけない（dangling reference の防止）：

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

`'a` はライフタイム引数。x と y の借用ライフタイムを揃え、返り値もそのライフタイム以内であることを保証する。

### Memory Safety without GC — 革命的な達成

Rust が達成したのは **「ガベージコレクション無しでメモリ安全性を保証する」** こと。これは長年「不可能」とされていた目標：

- C/C++: メモリ管理は完全に手動、Use-After-Free / Double-Free / バッファオーバーフロー の温床
- Java / Go / Python: GC で安全性を確保、しかし GC 停止・メモリオーバーヘッドのコスト
- **Rust**: 所有権システムで静的に安全性を保証、ランタイムコストなし

Microsoft Security Response Center の調査では、Windows のセキュリティ脆弱性の **約 70% がメモリ安全性の問題**。Rust への移行で、このカテゴリの脆弱性が **設計上不可能** になる。

Linux カーネルは 2022 年に Rust サポートを公式に取り入れ、Android は新規コードを Rust で書くことを推奨。**メモリ安全性が「個人の注意力」から「コンパイラの保証」へ移行する転換点**。

### RAII (Resource Acquisition Is Initialization)

Rust の `Drop` トレイトは **「値がスコープを抜けるとき自動的にデストラクタを実行する」** 機構。これは C++ の RAII を継承し、所有権モデルと組み合わせる：

```rust
{
    let f = File::open("data.txt").unwrap();  // ファイルを開く
    // f を使う
}  // ← スコープ終了、Drop が呼ばれ自動的にファイルが閉じる
```

リソース管理（メモリ、ファイル、ロック、ネットワーク接続）が **所有権スコープと結び付く** ことで、リソースリークが原理的に発生しない。

### Linear Haskell — 異なるアプローチ

Linear Haskell（2018 年論文、2021 年 GHC 統合）は線形型を関数の引数注釈として導入：

```haskell
f :: a %1 -> b      -- a を1回使う線形関数
g :: a -> b         -- 通常の関数（自由に使える）
```

`%1` は「1 回使う」を意味する multiplicity。これにより、**既存の Haskell コードを破壊せず、線形性を選択的に取り入れられる**。

```haskell
useFileOnce :: Handle %1 -> IO ()
useFileOnce h = do
    hPutStrLn h "hello"
    hClose h
    -- h はもう使えない
```

Linear Haskell の革新は **「既存の関数型言語に線形性を後付けで追加可能」** と示したこと。Rust が「最初から所有権ベース」で設計されたのに対し、Linear Haskell は「既存言語の漸進的拡張」アプローチ。

### Substructural Logic の他のメンバー

線形論理は **部分構造論理 (substructural logic)** という大きな枠組みの一部。古典論理が満たす 4 つの「構造規則」を選択的に除去する：

- **収縮 (Contraction)**: A, A ⊢ B から A ⊢ B（同じ前提を 2 つから 1 つに）→ 除去すると **線形/アフィン型**
- **弱化 (Weakening)**: A ⊢ B から A, C ⊢ B（不要な前提を追加できる）→ 除去すると **relevant type**
- **交換 (Exchange)**: 前提の順序を変えられる → 除去すると **non-commutative logic**
- **同一 (Identity)**: A ⊢ A → これは常に保持される

各構造規則の選択的除去から、さまざまな型システムが導かれる：

| 構造規則 | 採用 | 除去 | 言語例 |
|----------|------|------|--------|
| 収縮+弱化両方 | 古典 | - | C, Java, Python |
| 弱化のみ | アフィン | 収縮 | **Rust**, ATS |
| 収縮のみ | relevant | 弱化 | （理論的、実用例少） |
| 両方除去 | 線形 | 収縮+弱化 | Linear Haskell, Idris2 |

### Region-based Memory Management

Rust の所有権モデルに似た先行研究として **領域型 (region types)** がある（Tofte, Talpin 1994）。ML 系言語に対する代替の提案。**「メモリ領域 (region)」** を型で追跡し、領域がスコープを抜けると自動的に解放される。

```
let region r in
  let x = alloc r 42 in
  // x は r 領域に存在
end  // r が破棄、x も破棄
```

Cyclone（2002）はこれを C 拡張として実装。Rust のライフタイムシステムは **領域型の発展形**。Cyclone の研究者の一部は後に Rust 開発に関わった。

### Borrowing と Aliasing 制御の理論

Rust の **「不変借用は複数 OR 可変借用は 1 つ」** ルールは、**Permission System** とも呼ばれる。理論的には次のような表現：

```
Permission ::= Owned | Shared | Mutable
```

- **Owned**: 完全所有
- **Shared**: 不変な共有参照
- **Mutable**: 可変参照（排他的）

この 3 状態が状態機械として遷移し、コンパイラが追跡する。これは **線形論理の「論理的なリソース」が「実用言語の所有権状態」に翻訳された** 形。

### 実用上の困難 — Self-Referential Structures と Pin

純粋なアフィン型システムでは、**自己参照構造（リンクトリスト、グラフ）** の表現が難しい：

```rust
struct Node {
    value: i32,
    next: Option<Box<Node>>,
}
```

これは OK（単方向、所有が一直線）。しかし：

```rust
struct DoublyLinked {
    next: Option<Box<DoublyLinked>>,
    prev: Option<&DoublyLinked>,  // ❌ 所有モデルでは難しい
}
```

双方向リストや循環参照は **`Rc<T>` (参照カウント), `RefCell<T>` (内部可変性), unsafe** などの逃げ道を必要とする。これらは「アフィン制約からの脱出口」。

`Pin<T>` 型（2018 年導入）は **「メモリ上で動かない値」** を表現し、async/await の自己参照ジェネレータを安全に扱う仕組み。これは Rust の所有権モデルの応用拡張。

### 未来の方向 — Effect System との統合

近年、線形型と **エフェクトシステム (effect system)** の統合研究が進む。エフェクトシステムは「関数がどんな副作用を持つか」を型で追跡する：

```
f : Int -> Int @ {IO, State, Exception}
```

Koka, Effekt, OCaml 5 のエフェクトハンドラーがこの方向。**「リソースの線形性 + 効果の追跡」** で、コンパイル時に検証可能な事柄が劇的に増える。

## 気づき・洞察

### 「論理を計算機科学に翻訳する」 Curry-Howard の延長

線形型の Rust への応用は、[[cs/Curry_Howard同型と命題型対応]] の **「命題は型、証明はプログラム」** という Curry-Howard 同型の延長。線形論理の研究が **数十年遅れで実用言語に翻訳される** 過程。

これは **「理論研究は実用応用が出るまで何十年もかかるが、出たときの影響は莫大」** という基礎研究の典型例。Girard の 1987 年の論理が 2010 年代の Rust で形になった。

### 「制約が自由を生む」型システムの逆説

Rust の所有権モデルは初学者に **「コンパイルが通らない」** 苦痛を強いる。一見、自由を奪う制約。しかしこの制約があるからこそ、**並行プログラミングが安全に書ける**。データ競合が静的に排除されることで、開発者は「ロックを正しく取れているか」という認知負荷から解放される。

これは [[cs/TLA_plus_と形式手法_分散システム仕様検証]] の TLA+ の **「曖昧さの除去が表現力を生む」** と同じ哲学。**制約は奪うのでなく、別の自由を生む**。

### 「リソースとしての値」という認識転換

伝統的なプログラミング言語は **「値は永続的なエンティティ」** と見なす — 1 度作った変数は何度でも参照できる。アフィン型は **「値はリソースで、使えば消費される」** という認識転換を行う。

これは **計算機の概念モデルを物理世界に近づける** 動き。物理世界の「水を一度飲めば無くなる」を、プログラミングモデルが反映する。**抽象モデルが具体物理に接近することで、表現力が上がる**。

### Type System が「設計判断の codification」

Rust の所有権モデルは、優秀な C++ プログラマーが **暗黙のうちに守っていたルール** を、型システムで **明示化・強制化** したもの。「このポインタは関数の所有権を移すべきか、借用すべきか」という設計判断を、**コードに型として書く**。

これは **「経験的ベストプラクティスを言語仕様にコード化する」** トレンド。Go の goroutine、TypeScript の `strict` モード、Python の type hints もすべて同じ流れ。**「知っているべきこと」を「書かなければいけないこと」へ昇格させる**。

## 他分野との接続

### tech の Durable Execution との接続

今日の tech エントリ Temporal.io の Durable Execution も **「実行コンテキストをリソースとして扱う」** という線形型的視点。Workflow Execution は **一度開始したら明示的に完了するか、エラーで終了するまで「消費されない」** リソース。

Rust の所有権モデルが「メモリリソースを型で追跡」するように、Temporal は「実行状態をイベント履歴で追跡」する。**「リソースを明示的に管理する」** 哲学が両者の核心。

### music のショパン Op.28 との接続 — 「凝縮の数学」

今日の music エントリのショパン前奏曲は **「24 の調をすべて使い、一巡する」** 網羅性を持つ。線形型システムも **「すべての値が正しく扱われる（使われるか、明示的に破棄される）」** という網羅性を要求する。

両者とも **「可能性空間を完全に通過する」** 美学。完全性は美しい。

### anatomy の運動単位との接続 — 「動員の階層化」

今日の anatomy エントリの Henneman のサイズ原理は **「運動単位を順序付けて動員する」** 階層。Rust の所有権モデルも **「値の状態を Owned → Shared → Mutable」** と階層化して制御する。

両者とも **「複数の状態を階層的に区別することで、システム全体の動作を予測可能にする」** 戦略。

### 武道・哲学との接続 — 「執着を断つ」哲学

線形/アフィン型の **「使ったら無くなる」** という思想は、仏教の **「執着を断つ」** や禅の **「諸行無常」** に通じる。「永遠に保持される」と思っているリソースが、実は「使い終わったら手放すべきもの」という認識。

[[philosophy/西田幾多郎の絶対無]] の **「絶対無の場所」** にも繋がる — 何かを保持し続けるのではなく、生成と消滅を許容する流れこそが本質。
