# B木とデータベースインデックス — ディスクI/Oコストモデルから現代の変種まで
**日付**: 2026-03-19
**分野**: コンピュータサイエンス
**タグ**: #B木 #B+木 #LSM木 #データベース #インデックス #ディスクI/O #データ構造

## 学んだこと

### なぜB木なのか — ディスクI/Oコストモデル

コンピュータサイエンスにおけるデータ構造の教科書は、計算量をCPU命令数（比較回数）で評価する。しかしデータがメモリに収まらずディスクに置かれる瞬間、**支配的コストはCPU演算ではなくディスクI/O**に変わる。

ディスクアクセスの物理的特性:
- **シーク時間**: ヘッドの物理移動に数ミリ秒（SSDでも数十マイクロ秒）
- **転送単位**: ディスクはバイト単位ではなく**ブロック（ページ）単位**で読み書きする（典型的には4KB〜16KB）
- **ランダムアクセス vs シーケンシャルアクセス**: ランダム読み取りはシーケンシャルの100〜1000倍遅い

この物理的制約が**外部記憶モデル（external memory model）**の基盤となる。計算量は「何回ディスクページを読み書きするか」で測る。

#### 二分探索木がディスクで破綻する理由

赤黒木やAVL木は、メモリ上ではO(log₂ N)の美しい性能を持つ。しかしディスクベースの場合:

- **ファンアウト = 2**: 各ノードが2つの子しか持たない
- **木の高さ**: 100万要素でlog₂(10⁶) ≈ 20レベル
- **1ノード = 1ディスクアクセス**: ノードがディスクの異なるページに散らばるため、各レベルの遷移がランダムI/Oになる
- **結果**: 1回の検索に**20回のディスクシーク**が必要

これは致命的に遅い。ディスクシーク1回が5msなら、1回の検索に100ms。

#### B木の解決策 — ファンアウトの最大化

B木の核心的洞察は単純かつ強力: **ディスクの転送単位（ページ）に合わせてノードサイズを設計し、ファンアウトを最大化する**。

- ノードサイズ = ディスクページサイズ（例: 16KB）
- キーサイズ8バイト + ポインタ8バイトとすると、1ノードに約1000個のキーが入る
- **ファンアウト ≈ 1000**
- 木の高さ: log₁₀₀₀(10⁶) ≈ 2
- **1回の検索に2〜3回のディスクアクセス**

20回 → 3回。この劇的な改善がB木の存在意義。さらに、ルートノードと上位レベルのノードはメモリにキャッシュされるため、実際には**ほぼ1〜2回のディスクI/O**で済む。

### B木の構造と不変条件

B木（次数mのB木）の定義:

1. **すべての葉が同じ深さ**（完全にバランス）
2. ルート以外のノードは最低⌈m/2⌉個の子を持つ
3. ルートは最低2個の子を持つ（葉でない場合）
4. k個の子を持つノードはk-1個のキーを持つ
5. キーはノード内でソート済み

#### 高さの解析

N個のキーを持つ次数mのB木の高さh:

```
h ≤ log_{⌈m/2⌉}((N+1)/2)
```

つまりO(log_B N)。ここでBはファンアウト（分岐因子）。二分木のO(log₂ N)と比較すると、底がBであるため劇的に低い木になる。

#### フィルファクタ

B木のノードは最低半分（⌈m/2⌉）まで埋まっていなければならない。これが**フィルファクタの下限 = 50%**。実際のデータベースでは:

- ランダム挿入: 平均フィルファクタ ≈ 69%（ln 2の逆数に関連する理論的結果）
- 順序挿入: フィルファクタ ≈ 100%に近づく
- PostgreSQLの`fillfactor`パラメータ: デフォルト90%。更新が多いテーブルでは意図的に下げてページ分割を減らす（HOT更新のための空きを確保）

### B+木 — なぜデータベースはB木でなくB+木を使うのか

事実上すべての主要RDBMS（MySQL InnoDB、PostgreSQL、SQLite、Oracle、SQL Server）は**B+木**を採用している。B木とB+木の決定的な違い:

| 特性 | B木 | B+木 |
|------|-----|------|
| データの格納場所 | 全ノード | 葉ノードのみ |
| 内部ノードの役割 | キー + データ | キーのみ（ルーティング） |
| 内部ノードのファンアウト | 低い（データが場所を占有） | **高い**（キーだけなので多くのキーが入る） |
| 葉ノードのリンク | なし | **双方向リンクリスト** |
| 範囲クエリ | 木全体をトラバース | **葉をリンクで辿るだけ** |

#### なぜ内部ノードからデータを排除するのか

これは**ファンアウト最大化**の設計思想の延長。内部ノードにデータ（レコードへのポインタや実データ）を置くと、1ノードに入るキー数が減り、ファンアウトが下がり、木が高くなる。

仮にデータが100バイト、キーが8バイト、ポインタが8バイトで、ページサイズ16KBの場合:
- B木: 1ノードに約140個のキー+データ → ファンアウト ≈ 140
- B+木: 内部ノードに約1000個のキー → ファンアウト ≈ 1000

この差は木の高さに直結する。10億レコードで:
- B木: log₁₄₀(10⁹) ≈ 4.2 → 高さ5
- B+木: log₁₀₀₀(10⁹) ≈ 3 → 高さ3

たった2レベルの差だが、最下位レベルのI/Oはキャッシュされにくいため、実用上の差は大きい。

#### 葉ノードのリンクリスト — 範囲クエリの最適化

B+木の葉ノードは双方向リンクリストで接続される。これにより:

- `SELECT * FROM users WHERE age BETWEEN 20 AND 30`: 最初のキー（age=20）まで木を降り、あとは葉ノードをリンクで辿るだけ
- `ORDER BY`: インデックス順序でソート済みデータを返せる（追加ソート不要）
- フルスキャン: 内部ノードを経由せず葉を順にたどるだけ

**MongoDBのWiredTigerは意図的に葉間リンクを省略している**。MongoDBはドキュメント指向で範囲クエリよりポイントルックアップが主用途なため、リンク維持のオーバーヘッドを削減する判断。これはデータベースの使用パターンに合わせたトレードオフの好例。

### 挿入・削除 — バランスの維持

#### 挿入と分割（Split）

1. 適切な葉ノードを見つける
2. 葉に空きがあれば挿入して終了
3. 葉が満杯なら**分割**: 中央値キーを親に押し上げ、ノードを2つに分ける
4. 親も満杯なら再帰的に分割（最悪ケースでルートまで伝播→木の高さが1増加）

分割のコスト: 3回の書き込み（2つの新ノード + 親の更新）+ 場合によりさらに上位の更新。ただし分割はまれ。ランダム挿入では全挿入の約1/B回しか分割は起きない。

#### 削除とマージ/再配分

1. 葉からキーを削除
2. ノードのキー数が最小値（⌈m/2⌉-1）を下回ったら:
   - **再配分（Redistribution）**: 隣接兄弟ノードからキーを借りる（可能な場合）
   - **マージ（Merge）**: 兄弟とマージして1つのノードにする（親からキーを引き下ろす）
3. マージにより親のキー数が不足すれば再帰的にマージ

実際のデータベース実装では、**遅延マージ（lazy merge）**が一般的。即座にマージせず、ノードが空になったときだけ回収する。理由:
- マージ直後に挿入が来ると再分割が必要（スラッシング）
- アンダーフロー状態でもパフォーマンスへの影響は軽微
- PostgreSQLは右半分のみを削除する「半分削除」を実装し、ロック競合を減らしている

### B木 vs LSM木 — 書き込み増幅 vs 読み取り増幅

LSM木（Log-Structured Merge Tree）はB木と根本的に異なるアプローチを取る。

#### B木の書き込みパターン

- **In-place更新**: 既存のページを読み取り、変更し、書き戻す
- **ランダム書き込み**: 更新対象のページはディスク上のどこにでもある
- **書き込み増幅**: 1バイトの変更でも1ページ（4〜16KB）全体を書き込む → **Θ(B)の書き込み増幅**

#### LSM木の書き込みパターン

- **追記のみ（append-only）**: メモリ上のMemTableが満杯になるとソート済みファイル（SSTable）としてディスクにフラッシュ
- **シーケンシャル書き込み**: 常にシーケンシャルI/O
- **コンパクション**: 複数レベルのSSTファイルを定期的にマージ
- **書き込み増幅**: Θ(k · log_k(N/B))（kはレベル間のサイズ比）

#### 読み取りコスト

- **B+木**: O(log_B N)のページアクセス。上位ノードがキャッシュされれば実質1〜2回
- **LSM木**: 各レベルを探索する必要があり、最悪ケースではΘ((log²(N/B)) / log k)。Bloomフィルタで軽減可能だが、B+木には及ばない

#### RUM予想（Read-Update-Memory）

データ構造は**読み取り増幅・書き込み増幅・空間増幅の3つを同時には最適化できない**という予想。最大2つまでしか最適化できない:

- **B+木**: 読み取り最適化 + 空間効率 → 書き込み増幅が大きい
- **LSM木**: 書き込み最適化 + 空間効率 → 読み取り増幅が大きい

#### いつどちらを選ぶか

| ワークロード | 推奨 | 理由 |
|-------------|------|------|
| 読み取り重視のOLTP | B+木 | 予測可能な3〜4ページ読み取り、安定したp99レイテンシ |
| 書き込み重視のKVストア | LSM木 | シーケンシャル書き込みでスループット最大化 |
| 範囲クエリが多い | B+木 | 葉ノードのリンクリストで効率的 |
| 大規模・メモリ超過データセット + 小レコード | LSM木 | 書き込み増幅の差が顕著になる |

TiKVの設計判断が示唆的: **「読み取り性能はキャッシュで改善しやすいが、書き込み性能はハードウェアの制約を受ける」**ため、書き込み最適化のLSM木を選択し、読み取りはキャッシュで補う戦略。

### データベースインデックス — クラスタード、非クラスタード、カバリング

#### クラスタードインデックス（Clustered Index）

テーブルのデータそのものをインデックスのキー順に物理的にソートして格納する。InnoDB ではプライマリキーが自動的にクラスタードインデックスになる。

- **テーブルあたり1つだけ**（データの物理的並び順は1通りしかない）
- 範囲クエリに極めて有利: 連続するキー → 連続するディスクページ → シーケンシャルI/O
- 挿入がランダムだとページ分割が頻発 → auto-incrementの主キーが推奨される理由

#### 非クラスタードインデックス（Secondary Index）

データとは別に、インデックスキー → 主キー（またはRID）のマッピングを持つ別のB+木。

- テーブルに複数作成可能
- **ブックマークルックアップ**: インデックスでキーを見つけた後、実データを取得するためにクラスタードインデックスへの追加アクセスが必要（InnoDB）
- このセカンダリルックアップが高コストなため、**オプティマイザはフルスキャンの方が速いと判断する場合がある**

#### カバリングインデックス（Covering Index）

クエリが必要とする全カラムがインデックスに含まれていれば、実データへのアクセスが不要になる（**Index-Only Scan**）。

```sql
-- インデックス: (department_id, salary)
SELECT department_id, AVG(salary) FROM employees GROUP BY department_id;
-- → カバリングインデックスによりIndex-Only Scanが可能
```

PostgreSQLの`INCLUDE`句:
```sql
CREATE INDEX idx_dept ON employees (department_id) INCLUDE (salary, name);
```
`INCLUDE`カラムはB+木の葉ノードにのみ格納され、内部ノードのファンアウトを維持しつつ、カバリングの恩恵を得られる。

#### 複合インデックスのキー順序

複合インデックス`(A, B, C)`は辞書的順序でソートされる。この**プレフィックスルール**が決定的:

- `WHERE A = 1 AND B = 2 AND C = 3` → インデックスをフル活用
- `WHERE A = 1 AND B = 2` → Aの等値 + Bの等値 → 効率的
- `WHERE A = 1 AND C = 3` → Aだけ使える（Cでのフィルタはスキャン）
- `WHERE B = 2` → **インデックスが使えない**（プレフィックスが一致しない）

設計原則: **選択性（Selectivity）が高い順ではなく、等値条件→範囲条件の順**でキーを並べる。

```sql
-- 良い設計: statusは等値、created_atは範囲
CREATE INDEX idx ON orders (status, created_at);
-- SELECT * FROM orders WHERE status = 'active' AND created_at > '2026-01-01';
```

### 現代の発展 — B木を超える試み

#### Bw-tree（Lock-Free B-tree）

Microsoftが2013年に発表。マルチコア環境でのラッチ競合を排除するための設計:

- **ページへのポインタをマッピングテーブル経由で間接化**: ページIDから物理アドレスへの変換テーブルを持ち、CAS（Compare-And-Swap）操作で更新
- **差分更新（Delta Update）**: ページを直接変更せず、差分レコードをチェーン状に追加。一定数たまったら統合（consolidation）
- ラッチフリーだが実装の複雑性が高く、差分チェーンが長くなると読み取り性能が低下する問題がある
- SQL Server Hekaton、Azure Cosmos DBで採用

#### フラクタル木インデックス（Fractal Tree / Bε-tree）

Tokutek（現Percona）が商用化。B木とLSM木の中間的アプローチ:

- 各内部ノードにバッファを持つ: 挿入・削除・更新をバッファに蓄積
- バッファが満杯になると子ノードに一括フラッシュ
- **書き込み増幅がB木より大幅に低い**（最大2桁改善）一方で、読み取り性能はB木に近い
- εパラメータで読み取り/書き込みのトレードオフを調整可能

#### Adaptive Radix Tree（ART）

TUM（ミュンヘン工科大学）が2013年に発表。**メインメモリデータベース**向け:

- B+木はキャッシュラインに合わせた設計をしても、比較ベースの探索には限界がある
- ARTはキーのバイトごとに木を降りる（基数木）→ 比較不要、キー長に依存するO(k)のルックアップ
- 4種類のノードサイズ（4/16/48/256子）を適応的に切り替え、空間効率を維持
- **ルックアップ性能がB+木を上回り**、メモリ使用量も実用的
- HyPer（現Tableau Hyper）、DuckDBで採用

### 実世界の実装

#### InnoDB（MySQL）のB+木

- **クラスタードインデックス必須**: テーブル = クラスタードB+木。主キーがなければInnoDBが隠し6バイトROWIDを生成
- セカンダリインデックスの葉には主キー値を格納（物理RIDではない）→ ページ分割時にセカンダリインデックスの更新が不要
- **Change Buffer**: セカンダリインデックスへの書き込みをバッファし、後でマージ（書き込み最適化）
- ページサイズ: デフォルト16KB

#### PostgreSQLのB木

- **Heap + Index分離**: テーブルデータはヒープに、インデックスは別のB木。クラスタードインデックスの概念はあるが、InnoDBのように自動維持されない
- **MVCC対応**: インデックスはトランザクション可視性を直接判定しない。各タプルが独立したインデックスエントリを持ち、**重複排除（deduplication）**でバージョン増殖によるインデックス肥大を軽減
- **HOT（Heap-Only Tuple）更新**: インデックスカラムが変更されない更新では、インデックスを更新せずヒープ内でチェーンを作る
- 葉ノードは双方向リンクリスト。Lehman-Yaoアルゴリズムに基づく並行制御（右リンクを使った安全なトラバース）

#### SQLiteのB木

- **テーブル用B+木 + インデックス用B木**: テーブルはB+木（rowid順）、インデックスは通常のB木
- ページサイズ: デフォルト4KB（ファイルシステムのページサイズに合わせる）
- 1つの内部ページで約500の子ページを参照可能 → 476GBのデータベースでも4回のページ読み取りで任意のレコードに到達
- オーバーフローページ: レコードが1ページに収まらない場合、連続するオーバーフローページに分割格納
- セル形式: 可変長整数（varint）エンコーディングで空間効率を最大化

### ハッシュインデックスとの比較

| 特性 | B+木インデックス | ハッシュインデックス |
|------|-----------------|-------------------|
| ポイントルックアップ | O(log_B N) | **O(1)平均** |
| 範囲クエリ | **効率的** | 不可能 |
| ORDER BY | **インデックス順で返せる** | 不可能 |
| プレフィックス検索 | **可能** | 不可能 |
| ディスク適性 | **優秀** | ハッシュ衝突時の性能劣化 |
| 動的サイズ変更 | 段階的に成長 | リハッシュが必要 |

ハッシュインデックスがO(1)にもかかわらずB+木が支配的な理由: **データベースのクエリの大半は範囲条件、ソート、プレフィックスマッチを含む**。等値検索のみに使えるハッシュインデックスでは対応できない。

PostgreSQLのハッシュインデックスは長らくWAL非対応で使用非推奨だったが、v10以降でWAL対応し実用的に。それでも等値検索が圧倒的に多いケースに限定される。

#### スキップリスト

Redisの`ZSET`（ソート済みセット）はスキップリストを採用。B+木と同等のO(log N)検索・範囲クエリを提供しつつ:
- 実装が単純（B+木の分割・マージが不要）
- 並行制御が容易（ロックフリー実装が比較的簡単）
- ただしキャッシュ効率はB+木に劣る（ポインタチェイスが多い）
- メインメモリ環境に限定される

## 気づき・洞察

### ハードウェアが設計を決定する

B木の設計思想の核心は「**アルゴリズムの理論的最適性ではなく、ハードウェアの物理特性に適応する**」こと。ディスクのページサイズがノードサイズを決め、ランダムI/Oとシーケンシャル I/Oの速度差がB木 vs LSM木のトレードオフを生む。NVMeの登場でランダムI/Oのペナルティが縮小し、B木の書き込み増幅の不利が相対的に軽減されているのも、この「ハードウェアが設計を決定する」原則の表れ。

### 「何を最適化しないか」の選択

RUM予想が示すように、すべてを同時に最適化できない。**優れた設計とは「何を犠牲にするか」を明示的に選択すること**。TiKVが「読み取りはキャッシュで補えるが書き込みは補えない」と判断してLSM木を選んだのは、制約の中での合理的トレードオフの好例。

### B+木の「データを葉に限定する」判断の普遍性

「中間ノードを軽くしてファンアウトを最大化する」という設計は、ネットワークのルーティングテーブル（パケット転送に必要な最小限の情報だけ持つ）や、CPUのTLB（ページテーブルの要約だけキャッシュする）など、階層的なルックアップ構造に共通するパターン。

## 他分野との接続

- **音楽（music）**: B木のバランス維持は音楽の対位法に似ている——各声部（サブツリー）が独立して動きながら、全体のバランス（木の高さの均一性）を保つ。分割・マージは転調のように局所的な変化が全体の構造を維持しながら伝播する
- **解剖学（anatomy）**: 人体の循環系も階層的なファンアウト構造——大動脈 → 動脈 → 細動脈 → 毛細血管。各レベルでの「分岐数」が全体の効率を決め、1つのレベルだけ狭窄すると全体のパフォーマンスが劣化する（B木のボトルネックと同じ）
- **ゴルフ（golf）**: コースマネジメントにおける「何を最適化するか」の選択と同じ——飛距離最大化（書き込み速度）と正確性（読み取り速度）のトレードオフ。全ショットで両立はできない
- **哲学（philosophy）**: RUM予想は哲学における「トリレンマ」構造——全てを同時に得ることはできないという制約の中での選択

## 次に深掘りしたいこと

- **Write-Ahead Log（WAL）とクラッシュリカバリ**: B+木のin-place更新がクラッシュ時にどう整合性を保つか。WAL、ダブルライトバッファ、Copy-on-Write B木の比較
- **Buffer Pool Management**: B+木のパフォーマンスはバッファプールのヒット率に依存する。LRU、Clock、LRU-Kなどのページ置換アルゴリズム
- **クエリオプティマイザのインデックス選択**: 複数のインデックスがあるとき、オプティマイザはどのインデックスを使うか（またはフルスキャンを選ぶか）をどう決定するか。コストモデルの内部
- **Learned Index**: Kraska et al. (2018)の機械学習ベースのインデックス。CDFをモデル化してB木を置換する試み。実用化の現状と限界
- **LMDB/BoltDBのCopy-on-Write B+木**: MVCCをB+木レベルで実現するアプローチ。スナップショット分離の仕組み
