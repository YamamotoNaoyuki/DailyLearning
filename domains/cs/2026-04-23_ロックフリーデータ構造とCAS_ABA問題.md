# ロックフリーデータ構造とCAS・ABA問題
**日付**: 2026-04-23  
**分野**: cs  
**タグ**: #ロックフリー #CAS #ABA #線形化可能性 #メモリモデル  

## 学んだこと

### なぜロックフリーか — mutex の構造的限界
排他制御(mutex/spinlock)は「クリティカルセクション中のスレッドが任意時間停止すると、他の全スレッドも停止する」という性質を持つ。OSスケジューラによるプリエンプション、ページフォルト、ガベコレのSTW、ハードウェア割り込みなど、mutex保持中に発生するあらゆる「停止」が他スレッドの進捗を無限に阻害する。優先度逆転・デッドロック・コンボイ効果なども同じ根——**進捗が他スレッドに依存している**構造から生じる。

ロックフリー(non-blocking)同期は、この依存関係を切り離し、「一部のスレッドが任意時間止まっても**システム全体が進む**」性質を保証する設計パラダイム。単なる高速化技法ではなく、「停止しうる環境で停止耐性を得る」**正当性の道具**でもある。

### 進捗条件の階層 (Herlihy 1991, JPDC)
Maurice Herlihyの1991年論文 "Wait-free synchronization" は進捗条件を厳密に階層化した：

- **Wait-free**: 全てのスレッドが、他の挙動に関係なく**有界ステップ数**で必ず操作を完了。最強。リアルタイムシステムに必須だが実装が極めて困難。
- **Lock-free**: 少なくとも1つのスレッドが有限ステップで進む(system-wide progress)。個別スレッドは飢餓しうるが、システム全体は停止しない。
- **Obstruction-free**: 他スレッドを一時停止させれば単独スレッドは進む(Herlihy, Luchangco, Moir 2003)。最弱の非ブロッキング条件。Software Transactional Memory(STM)の多くがこのレベル。
- **Blocking(mutex)**: サスペンド中のスレッドが他スレッドを無限に阻害しうる。

重要なのは、これが**表現力の強弱**ではなく**進捗保証の強弱**の階層であること。wait-freeは常に望ましいが、コストが指数的に増える場合がある。実用上は lock-free が「コストと保証のスイートスポット」として最も使われる。

### CAS(Compare-And-Swap)プリミティブ
ロックフリー設計の中核は CAS という**アトミックread-modify-write**命令：

```
CAS(addr, expected, new):
    atomically {
        if *addr == expected:
            *addr = new
            return true
        else:
            return false
    }
```

CAS はハードウェアで実装される：
- **x86**: `LOCK CMPXCHG`(64bit) / `LOCK CMPXCHG16B`(128bit)。バスロック(古い実装)またはキャッシュライン所有権獲得(MESI Modified遷移)で実現。
- **ARM / RISC-V**: LL/SC(Load-Linked / Store-Conditional)ペア。`LDXR/STXR`(ARM), `LR/SC`(RISC-V)でキャッシュラインを「予約」し、他コアの干渉があればSCが失敗。
- **レイテンシ**: キャッシュHitで~10-40サイクル、同一ソケット内competedで~50-100サイクル、クロスソケットで~100ns(コア数百ns×アービトレーション)。mutex acquireが~25-50nsなのでCAS単体は非常に速いが、**失敗時のretryコスト**が本質的問題。

CAS の進捗性質: CASが失敗するのは他スレッドが成功した時のみ。よって少なくとも1つは進む → **CAS自体が lock-free 進捗の起源**。

### Treiber Stack (1986, IBM Research Report RJ5118)
R. Kent Treiber の古典的lock-freeスタック。LIFO構造をCASで：

```
struct Node { T val; Node* next; };
atomic<Node*> top;

push(v):
    n = new Node(v)
    loop:
        old = top.load()
        n.next = old
        if CAS(&top, old, n): return

pop():
    loop:
        old = top.load()
        if old == nullptr: return EMPTY
        next = old.next
        if CAS(&top, old, next): return old.val
```

push は安全。しかし pop には**ABA問題**が潜む。

### ABA 問題の詳細
次のインターリーブを考える：

1. T1: `old = top; // = A`
2. T1: `next = A.next; // = B` ここでプリエンプト
3. T2: pop() → A を取り出す、free(A)
4. T2: push(C), push(D), pop(D)
5. T2: 新しいノードをアロケータが**同じアドレス A** に置いて push → top = A'
6. T2: 状態は `top = A' → C`
7. T1: 再開、`CAS(&top, A, B)` が**成功してしまう**。ポインタ比較では A == A' だから。
8. 結果: top が B を指すが B は既に解放済み → Use-After-Free & silent corruption。

ABA の本質は**「ポインタの同一性 ≠ 状態の同一性」**。メモリアロケータがアドレスを再利用する限り避けられない。C/C++/低レベル言語固有の問題で、GC言語(Java/Go)は**GCが生存中のメモリを再利用しない**ため自動回避される。これが「GC-free lock-free は別次元の難しさ」と言われる理由。

### ABA 解決策

**1. タグ付きポインタ(Counter)**
(ptr, tag) のペアをアトミックに扱う。modifyごとにtagをインクリメント。x86 の `CMPXCHG16B` で128bit(48bit ptr + 80bit tag)をCAS。tag幅がオーバーフロー確率を決める — 64bit tagなら現実的にABA不発生。欠点: 2倍幅のCASが必要で、全プラットフォームで利用できない。ARM32に128bit CASは無い。

**2. Hazard Pointers (Michael 2004, IEEE TPDS)**
各スレッドが「今dereferenceしているポインタ」を専用スロットに**publish**する。メモリ解放は「全スレッドのhazardスロットを走査し、どこからも参照されていない」ことを確認してから。

```
read: hp = ptr; memory_fence(); if (*ptr_loc != ptr) retry; use(ptr);
free: retired_list.push(ptr); if (retired_list.size > threshold) scan_and_free();
```

長所: 境界のあるメモリ使用、portable。短所: 各readでfence + publishが必要でオーバーヘッド大(~20-50%)。Linux kernel や Folly(Facebook)が採用。

**3. Epoch-Based Reclamation (EBR, Fraser 2004)**
グローバルepoch counterを置き、スレッドはクリティカルセクション進入時にepochを記録。メモリ解放は「全スレッドが以前のepochを抜けた」時まで遅延。readは単にepoch読み書きのみで済み、極めて低オーバーヘッド(~5%以下)。欠点: スレッドがクリティカルセクション内で停止(プリエンプト・ページフォルト)するとepochが進まず、メモリが無限蓄積 = **unbounded memory**。crossbeam-epoch(Rust)の基盤。

**4. RCU(Read-Copy-Update)**
Linux kernel の主力技術(McKenney ら)。readerはロックなしでデータ構造を読む。writerは新バージョンを別途構築しatomic pointer swapで公開、旧バージョンは「全CPUがquiescent stateを通過」してから解放。read性能が桁違い(ほぼnative pointer load)、write/reclaimは重い → **read-heavyワークロード**で圧倒的。Linux内では`rcu_read_lock/unlock`, `synchronize_rcu`, `call_rcu`として広範に使われ、VFS・ネットワーキング・namespaceなどほぼ全サブシステムに浸透。

### Michael-Scott Queue (1996, PODC)
Maged Michael & Michael Scott の lock-free FIFO キュー。head/tailの2ポインタをCASで操作。`java.util.concurrent.ConcurrentLinkedQueue` の実装基盤。

重要テクニックが **cooperative helping**: enqueueは2段階(nextをCAS、次にtailをCAS)でtail更新が中断しうる。別スレッドが「tailが古い」ことを検出したら**自分が進めるべき処理の途中でも代わりにtailを進める**。

```
enqueue(v):
    n = new Node(v, nullptr)
    loop:
        t = tail.load()
        next = t.next.load()
        if t != tail.load(): continue  // 一貫性再確認
        if next == nullptr:
            if CAS(&t.next, nullptr, n):
                CAS(&tail, t, n)  // 進め、失敗は他スレッドが既にやった
                return
        else:
            CAS(&tail, t, next)  // 他人のために tail を進める

dequeue():
    loop:
        h = head; t = tail; next = h.next
        if h == head:  // 一貫性
            if h == t:
                if next == nullptr: return EMPTY
                CAS(&tail, t, next)  // 遅れたtail を進める(ヘルプ)
            else:
                v = next.val
                if CAS(&head, h, next): return v
```

**「他人の仕事を代行する」**のが lock-free 進捗保証の典型パターン。誰かが詰まっても他が先に進める = system-wide progress.

### メモリモデルとバリア
CAS だけでは lock-free 設計は完成しない。**メモリ順序**を適切に指定する必要がある：

- **x86-TSO**: store→load 以外の並び替えなし。CAS は暗黙の full fence。
- **ARM/POWER**: 弱順序。load-load, load-store, store-load, store-store 全て並び替え可能。明示的 DMB/ISB バリア必要。
- **C++11 memory_order**:
  - `relaxed`: atomicityのみ、順序保証なし(統計カウンタ等)
  - `acquire`(load): 以降のmemory accessは先行不可
  - `release`(store): 先行のmemory accessは後行不可
  - `acq_rel`(RMW): 両方
  - `seq_cst`: 全スレッドが同一順序を観測(最強・最重い)

典型ペア: publisherが`store(release)`で publish、consumerが`load(acquire)`で consume → happens-before確立。Treiber stack push も pop も実用上は acq_rel で十分(seq_cstは過剰)。

### 線形化可能性 (Herlihy & Wing 1990, TOPLAS)
並行データ構造の正しさの黄金基準。各operationが「invocationとresponseの間のある単一時点で瞬間的に効いた」と見なせ、その順序が**real-time orderと一致**すること。

- **Sequential consistency** より強い(real-time制約が追加)。
- **Strict serializability** と等価だが、こちらはトランザクション用語。
- **compositional**: 線形化可能な部品を組み合わせても線形化可能(重要な設計原則)。

ロックフリーアルゴリズムの論文は必ず「linearization point は CAS成功のこの瞬間」を明示する。MS-queue enqueueの線形化点は `CAS(&t.next, nullptr, n)` 成功時、dequeueは `CAS(&head, h, next)` 成功時。

### 不可能性結果
**Herlihy's Consensus Hierarchy (1991)**: オブジェクトが解ける consensus問題のスレッド数で分類。
- 通常レジスタ(atomic read/write): consensus number = **1**(2スレッドすら合意不能)
- FIFO queue, stack: consensus number = **2**
- CAS, LL/SC: consensus number = **∞**(任意スレッド数の合意を解ける)

**CAS が「ユニバーサル」**なのはこの結果による。どんな sequential オブジェクトも CAS と universal construction で wait-free 並行オブジェクトに変換可能(Herlihy 1991)。実用的ではないが理論的に完全性が示されている。

**FLP Impossibility (Fischer, Lynch, Paterson 1985)**: 非同期分散システムで1プロセス故障を許す決定的合意は不可能。共有メモリの問題(Herlihy)とは異なる設定だが、哲学的に「非同期性と耐障害性の緊張」という同じ主題を扱う。

### 性能特性と落とし穴
- **uncontended CAS**: 10-40ns(mutex ~25ns、大差ない)
- **contended CAS**: 指数的悪化。N個のCPUがホットスポットを叩くと O(N²)のretry爆発。**exponential backoff**が必須。
- **cache line ping-pong**: MESIプロトコルで同一ラインがコア間を行き来し、1 CAS あたり数百サイクル。隣接フィールドの意図しない共有 = **false sharing**。`alignas(64)` でパディング必須。
- **livelock**: 各スレッドが互いを打ち消しあい、「lock-free なのに進まない」状態。MS-queueの`if t != tail` 再確認も livelock対策の一部。
- **デバッグ困難性**: 競合バグは10^9-10^12 回に一度発現。Thread Sanitizer, Relacy Race Detector, 形式検証(TLA+, CBMC)が不可欠。

ロックフリーコードの**3日で書いて3ヶ月デバッグする**という経験則は誇張ではない。

### 実世界での使用
- **JVM**: `java.util.concurrent.atomic.*`, `ConcurrentLinkedQueue/Deque`, `AtomicReference`, `VarHandle`(Java 9+)。GCがABAを自動回避するためlock-free実装が比較的容易。
- **Linux kernel**: RCU が巨大(数万の call_rcu コールサイト)、ハザードポインタは networking fastpath で選択的、per-CPU カウンタがスケール戦略の主。
- **Rust**: `std::sync::atomic` で明示的orderings、`crossbeam` エコシステム(epoch reclamation)、`Arc<T>` は refcount CAS。
- **DB/KVS**: Memcached の lock-free hash(extstore), Redis の内部 LRU、**RocksDB の MemTable skip list**(LSM tree 前回記事参照 — skip list insertion が CAS ベース)。
- **Go**: `sync/atomic` + GC。`sync.Map` は lock-free read path + mutex write path のハイブリッド。

### ロックフリーを使わない判断
1. **低競合**: mutex と性能ほぼ同じ、シンプルさの価値が勝る。
2. **複雑な不変条件**: 複数オブジェクト間の整合性は lock-free で維持困難 → transactional memory か mutex。
3. **非クリティカルパス**: 開発/保守コストが性能利得を上回る。

「Lock-free ≠ 自動的に速い」。Tony Van Eerd(Boost)「ロックフリーは最後の手段であり、最初の衝動ではない」。

## 気づき・洞察

- **進捗条件は「表現力」ではなく「耐故障性」の階層**。wait-free は「故障=任意時間停止」に対する完全な免疫を持つ。mutex はゼロ免疫。この軸は従来の「効率性」軸と直交する設計次元。
- **メモリ回収こそが本質問題**。アルゴリズム自体の正しさよりも、「いつメモリを安全に解放するか」がロックフリー実装の9割を占める。C/C++ ではハザードポインタ/EBR/RCU のどれを選ぶかがアーキテクチャ決定。GC言語はこの問題を暗黙に解いている——GC は「分散ガベージ検出」であり、ロックフリー設計の隠れた土台。
- **CAS のユニバーサリティ**は、現代マルチコアCPUが必ずこのプリミティブを持つ理由の理論的正当化。「∞-consensus オブジェクトがあれば何でも並行化できる」というHerlihyの結果は、CPU命令セット設計への決定的ガイダンスとなった。
- **ヘルプパターン(cooperative helping)**は lock-free の本質的思想。「他スレッドが詰まっても自分が進む」ではなく「**他スレッドの詰まりを自分が解消しながら進む**」。individualismではなくcollectivism。
- **不可能性結果は制約ではなく設計指針**。Consensus hierarchy は「何を諦めるか/何に投資するか」を教えてくれる。通常レジスタでは合意不能 → だから CAS が必要、という論理。FLPも「非同期+故障+決定性」のどれかを捨てよと示唆。
- **メモリモデルは抽象化漏れの典型**。「逐次意味論」に見えるコードが実機では並び替えられる。Spectre/Meltdown(前回記事)とロックフリーの memory_order は、同じ「ハードウェア現実が抽象化を突き破る」現象の両面。

## 他分野との接続

- **Raft(前回cs分野)**: Raftは**分散ノード間**の合意、CAS ベース lock-free は**共有メモリ内**の合意。どちらも「過半数/単一成功者」という絞り込みで合意を構築。Herlihy の consensus number は両者を統一する理論枠組み。Raftのリーダー選挙 term 増加は CAS の tag カウンタと構造同型。
- **LSM tree(前回cs分野)**: RocksDB/LevelDB の MemTable は skip list で、**skip list insertion が CAS ベース**。レベルx を確率的に選ぶスキップリストの構造は、CASでの挿入を容易にする(head/tail の局所変更で済む)。lock-free skip list (Fraser 2004) はRocksDB MemTable の直系。
- **GC(前回cs分野)**: ハザードポインタは「分散並行GC」の一形態。mark フェーズ = 「どのポインタが生存か全スレッドから集める」、sweep = 「誰も hazard していないものを解放」。RCU の grace period も「世代交代時の並行mark」。**GC とメモリ回収アルゴリズムは同じ問題の別スケール**。
- **TCP輻輳制御(前回cs分野)**: CAS retry の exponential backoff は TCP の AIMD と同じ哲学 — 競合発生時の減速で系全体を安定化。cache line ping-pong は MESI プロトコル上の「輻輳崩壊」と捉えられる(共有リソースへの過剰アクセスが性能を破壊する正のフィードバック)。
- **CPU パイプライン(前回cs分野)**: Spectre/Meltdown が「ISAと μarch の抽象化漏れ」だったのと同様、C++ の memory_order は「ソースコード意味論と CPU実機実行の抽象化漏れ」の制御機構。ロックフリーは μarch レベルの現実に最も近づく高級言語コード。
- **公開鍵暗号(前回cs分野)**: 線形化可能性の「操作の全順序に real-time 制約」という要請は、暗号プロトコルの「メッセージ順序整合性」と構造的に近い。両者とも「部分的観測から全体順序を復元する」問題。
- **音楽(他分野)**: アンサンブル演奏は「中央指揮者のいない分散合意」。弦楽四重奏は拍を拘束的な共有状態(mutex)ではなく、呼吸・視線・音の重なりという**ヘルプ的・lock-free 的な合意**で進める。ミスしたメンバーを他が拾う = MS-queue の tail ヘルプ。
- **ゴルフ・身体制御(他分野)**: ショット実行中の身体は「いくつもの筋群の並列制御」で、ロック(固定)の多用は流動性を失う。lock-free 的に「進行中の動作に他の筋群が介入して補正する」方が結果が良い、というのは古典的な運動学習の知見。

## 次に深掘りしたいこと

1. **Relativistic Programming(McKenney)**: RCUを越えるread-side optimization理論。複数readerが「異なる相対的観測順序」を持つことを許容する設計。
2. **Transactional Memory の現状**: Intel TSX(RTM/HLE)はRowhammerと絡んで2019 disableされた経緯、ARM TME の展望、Software TM(GHC, Clojure refs)の実用成功例。
3. **wait-free universal construction の実装技法**: Herlihy 1991の理論的構造から、Kogan & Petrank 2012(fast-path slow-path)、Timnat & Petrank 2014 への実用化。なぜwait-freeが商用ライブラリで稀なのか。
4. **永続メモリ上の lock-free**: Intel Optane(2022終了)の経験から、clwb/sfence を含む persistence-aware lock-free の設計原理。Atlas, Mnemosyne など。
5. **RCU の形式検証**: Linux kernel RCU は Paul McKenney らによりTLA+/Coq で検証されている。Dennis-Ritchie 時代のコードがどこまで数学的に正しいか。
6. **ロックフリー vs 並列GC のハイブリッド**: ZGC の Colored Pointer や Shenandoah の Brooks Pointer は mutator と collector が lock-free に協調する。GC側から見た lock-free。
