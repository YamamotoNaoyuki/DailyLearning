# Hindley-Milner型推論とlet多相
**日付**: 2026-04-24
**分野**: コンピュータサイエンス
**タグ**: #型理論 #型推論 #ML #関数型 #多相性 #AlgorithmW

## 学んだこと

### Hindley-Milnerの位置づけ
Hindley-Milner（HM）型システムは、**ラムダ計算 + パラメトリック多相**に型を与える古典的型理論。1958年にJ. Roger Hindleyが組合せ論理で発見し、1978年にRobin Milnerが独立にML言語の型システムとして再発見した。後にLuis Damasが博士論文（1984、エジンバラ大）で形式的な健全性・完全性を証明し、**Damas-Hindley-Milner（DHM）**とも呼ばれる。

重要な性質：
- **推論可能（inferable）**: プログラマが型注釈を書かなくても、アルゴリズムが最も一般的な型（**principal type**）を自動的に見つけられる
- **健全（sound）**: 型がついたプログラムは実行時型エラーを起こさない
- **完全（complete）**: 型付け可能なプログラムはすべて型推論に成功する
- **最も一般性の保証**: 推論される型は、そのプログラムに付けられる全ての型の中で最も一般的

これらの性質を同時に満たす型システムとして初期に発見された驚くべき均衡点。ML、OCaml、Haskell、Standard ML、F#、Elmの型システムの理論的基盤。

### 型と型スキームの区別
HMの核となる区別：**型（type）** vs **型スキーム（type scheme）**。

- **型**: `Int`, `Int -> Bool`, `a -> a`（ここでaは自由型変数）
- **型スキーム**: `∀a. a -> a`——多相型（量化された型変数を持つ）

HMの制限は「**let束縛された変数のみ型スキームを持てる**」——これを**let多相**（let-polymorphism）という。ラムダ引数は常に単相型（monomorphic）。

```ml
let id = fun x -> x     (* id : ∀a. a -> a  (型スキーム) *)
in id 1, id "hello"     (* idを異なる型でインスタンス化できる *)

(* これはHMでは型付けできない: *)
fun f -> (f 1, f "hello")  (* fを異なる型で2回使うと失敗 *)
```

### Algorithm W
MilnerがアルゴリズムWとして定式化した推論手順。以下のように動作する：

1. **新しい型変数を導入**してAST上を走査
2. 各式に暫定的な型を与える
3. 各使用箇所で**型方程式**を集める（"この関数が呼ばれたから引数型 = 型変数X"等）
4. **単一化（unification）**で型方程式を解く
5. **let束縛**で、右辺の型を左辺のスコープで**一般化（generalize）**する
6. 変数の使用箇所で**インスタンス化（instantiate）**する（新しい型変数で量化変数を置き換え）

**核は単一化アルゴリズム**（Robinson 1965）: 2つの型を等しくする最も一般的な代入を求める。

```
unify(Int, Int) = {}
unify(a, Int) = {a ↦ Int}
unify(a -> b, Int -> Bool) = {a ↦ Int, b ↦ Bool}
unify(a, a -> b) = fail (occurs check)
```

**occurs check**——変数が自分自身を含む型と単一化されようとする場合（循環を作る）を検出して拒否する。

### 一般化（Generalization）とインスタンス化
**Generalization**: let束縛時、右辺の型に現れる**環境に束縛されていない**型変数を全て∀で量化する。

```
Γ ⊢ e : τ    Γ でfreeでないτの型変数a₁,...aₙ
---------------------------------------------
Γ ⊢ e : ∀a₁,...,aₙ. τ
```

なぜ「**環境に束縛されていない**」変数だけ量化するのか？環境中の変数に出現する型変数は、まだ他の式との制約関係を持ち続けるため自由に量化できない。これは微妙だが決定的なポイント。

**Instantiation**: 型スキームを使用するとき、量化変数を**新しい型変数**に置き換える。これにより同じ定義を異なる文脈で異なる型として使える。

### 多相性の制限——なぜLet多相か
ラムダ抽象の引数が多相にならない（つまり`fun f -> (f 1, f "hello")`が型付けできない）のは、**型推論を決定可能にするため**。より強力な**System F**（Girard-Reynolds、1972）は関数引数も多相にできるが、型推論は**決定不能**。

HMは「型推論可能性」と「多相性」の実用的な均衡点を見つけた。System FはCoq/Agda等の証明支援系で使われ、そこでは型注釈を書く前提なので決定不能でも問題ない。

### 複雑性の理論と実務のギャップ
理論上、Algorithm Wの最悪計算量は**DEXPTIME（二重指数時間）**。Mairson 1989の構成例：深くネストしたlet束縛で指数的な型が生成される。

しかし**実際のプログラムでは線形に近い**。これはコンパイラ理論の「最悪ケース分析と実務の乖離」の典型例。OCamlやHaskellコンパイラは百万行のコードでも合理的な時間で型推論する。

### Bottom-up (W) vs Top-down (M)
- **Algorithm W**: 文脈に依存せず、ボトムアップに型を構築。エラー発生時、どこでエラーが検出されるかが直感的でないことがある
- **Algorithm M**: トップダウン。期待型を文脈から受け取り、式を検査する。エラーメッセージが改善される

近年の関数型言語処理系は**両者のハイブリッド**（Hindley-Milner + Local Type Inference）を使う。Scala、Kotlin、TypeScriptなどは部分型付けがあるためHMをそのまま使えず、局所的な型推論と明示的注釈の混合となる。

### HMの限界——拡張の系譜
- **Let多相のみ**: Haskell/OCamlでは"value restriction"（MLで副作用と両立させる制限）が必要
- **サブタイピング非対応**: Scala、OCamlのオブジェクトは部分型推論を別途実装
- **高階多相（higher-rank）**: System Fレベルの多相が欲しい場面はあるが推論困難
- **依存型非対応**: 型が項に依存するCoq/Agdaは別系統
- **型クラス（Haskell）**: HMに拡張した制約付き多相、`∀a. Num a => a -> a`
- **GADT**: 型等式を導入する拡張、推論が難しくなる

現代の関数型言語は**HMを基盤に、必要に応じて拡張**する設計。

## 気づき・洞察

### "型はなぜ推論できるのか"という奇跡
高校数学で連立方程式を解くように、**プログラムの型は方程式系として解ける**——これは驚異的な事実。変数x,y,zの値が分かるように、型変数a,b,cの具体化が単一化で求まる。プログラムの意味論と方程式系の理論（Robinsonの統一）が深いところで結びついている。

### 一般化の"タイミング"が多相性を決める
HM型システムの核心は「**いつ一般化するか**」——let束縛時に一般化し、使用時にインスタンス化する。この一手で、同じ定義を異なる型で使いまわせる。ラムダ引数では一般化しないことで、推論を決定可能に保つ。**抽象化の時期の設計**が理論全体の性質を決める。

### 型推論と定理証明の等価性——Curry-Howard
Curry-Howard対応により、**型は命題、プログラムは証明**。HMの型推論は、プログラムから「このプログラムが証明する命題」を自動抽出することに相当。System FやCoqでは逆に、人間が命題（型）を書き、プログラム（証明項）を構築する。HMはこの対応の一方向（プログラム→命題）を自動化した。

### 経済的な型システム設計
HMは「できることを絞って、推論可能性を獲得」した経済的な設計。**最強の型システムではないが、最もバランスが取れている**。実用言語でHMを採用したMLとHaskellが計算機科学の研究と実務で長期的な成功を収めたのは偶然ではない。

### "書かずに済む"コストと"書けない"コスト
HMの利点は「型注釈を書かなくてよい」。欠点は「**書きたくても書きにくい**」（higher-rank多相ができない）。現代言語の設計は「**通常はHM風の推論＋必要時に明示的注釈**」の折衷が主流（Rust、Scala、TypeScript）。**何を自動化し、何を手動にするか**の線引きが言語デザイナの仕事。

### 型エラーの難しさという副作用
HMは「最も一般的な型」を推論するが、これは時として**人間の意図から遠い**型になる。例：

```haskell
f x y = if x then y else 42
-- 推論: f :: Bool -> Int -> Int
-- （yがIntに束縛される）
```

ユーザーは「yは何でもいい」と思っているかもしれない。型エラーが表面化するのは**使用箇所**で、**定義から遠く離れた場所**。Haskellが難しいと言われる一因は、この「離れた型エラー」の診断性の悪さ。

## 他分野との接続

### anatomy分野（今日の小脳・forward model）との構造類似
小脳のforward modelは「この運動指令に対し身体はどう動くか」を予測する。HMの型推論は「このプログラムに対し型はどうなるか」を予測する。**実行前に結果を予測する**という計算パターンが、神経系と言語処理系に独立に現れている。

### tech分野（今日のDurable Objects）との接続
DOのInput/Output Gateは「実行開始時刻に全体の整合性を保証」する仕組み。HMの単一化は「コンパイル時に全体の型整合性を保証」する仕組み。**静的な整合性検証**という同じ哲学の動的版と静的版。

### music分野（今日のスクリャービン）との類比
神秘和音は6音を「積み重ね方のルール」で生成する構文。HMは型を「型方程式を解くルール」で生成する構文。**明示的なリストではなく、生成規則で対象を定める**構造主義的発想が共通する。

### piano分野（今日の跳躍）との類比
跳躍の成功は「離陸前に着地位置を予測すること」——事前に全体を解いておく。HMの型推論も「実行前に型を解いておく」。**実行時に解かず、実行前に解く**ことで速度と安全性を稼ぐ発想。

### ゲーム理論（2026-04-21）との接続
ナッシュ均衡は「各プレイヤーの最適反応の不動点」。単一化は「型方程式の不動点（最一般単一子、most general unifier）」。**制約系の解を不動点で特徴づける**という数学的構造が共通。
