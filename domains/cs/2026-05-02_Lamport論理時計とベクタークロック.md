# Lamport 論理時計とベクタークロック
**日付**: 2026-05-02
**分野**: cs
**タグ**: #分散システム #Lamport #論理時計 #ベクタークロック #因果関係

## 学んだこと

### 出発点 — "Time, Clocks, and the Ordering of Events" (1978)
Leslie Lamport の 1978 年論文は、分散システムにおける「時間」概念を物理時計から切り離した。動機は単純で深い：物理時計は完全に同期できず (相対性理論的にも、工学的にも)、「同時」「以前」を厳密に定義できない。ならば **観測可能な因果関係だけで「順序」を定義できないか**？

### Happens-before 関係 (→)
Lamport は半順序関係 → を以下で定義した：
1. 同一プロセス内で、a が b より先に実行されたなら a → b
2. a が「メッセージ送信」イベントで b が「対応する受信」イベントなら a → b
3. 推移律: a → b かつ b → c なら a → c

a → b でも b → a でもないとき、a と b は **concurrent (並行)** と呼ぶ。並行イベントは「順序が定義できない」のではなく「順序が観測不能なので定義する必要がない」。これが分散システム理論の基礎概念。

### Lamport タイムスタンプ (スカラー論理時計)
各プロセス P_i は単調増加カウンタ C_i を持つ。
- ローカルイベント時: C_i ← C_i + 1
- メッセージ送信時: メッセージに C_i を添付
- メッセージ受信時: C_i ← max(C_i, msg.timestamp) + 1

これにより、a → b ⇒ C(a) < C(b) が保証される（**clock condition**）。ただし逆は成り立たない: C(a) < C(b) でも a, b が concurrent な場合がある。つまり Lamport 時計は因果関係を **inject はするが reflect しない** (片方向のみ)。

### スカラー時計の限界
全順序が必要な場合 (例: 分散ミューテックスの優先順位決定) は、(C_i, i) の辞書式順序でタイブレークすれば全順序にできる (**Lamport 全順序**)。しかしこの全順序は人為的で、もとの因果関係を超えた順序を強制している。

実際の問題は、**「2 つのイベントが本当に concurrent か、それとも因果関係があるのか」を後から判定したい** こと。Lamport 時計ではそれができない。これがベクタークロックを生んだ。

### ベクタークロック (Mattern 1989, Fidge 1988)
各プロセス P_i は長さ N (プロセス数) のベクトル V_i を持つ。V_i[j] は「P_i から見た P_j の現在の論理時刻」。
- ローカルイベント時: V_i[i] ← V_i[i] + 1
- 送信時: ベクトルをメッセージに添付
- 受信時: 各 j について V_i[j] ← max(V_i[j], msg.V[j])、その後 V_i[i] ← V_i[i] + 1

比較規則：
- V(a) ≤ V(b) ⟺ ∀k. V(a)[k] ≤ V(b)[k]
- V(a) < V(b) ⟺ V(a) ≤ V(b) かつ V(a) ≠ V(b)
- V(a) と V(b) が比較不能 (どちらも ≤ でない) ⟺ a と b は concurrent

**a → b ⟺ V(a) < V(b)** という強い性質が成立する (両方向)。これがベクトル時計の本質。

### 計算複雑度のトレードオフ
- スカラー時計: O(1) サイズ、因果関係の特徴付けは弱い
- ベクトル時計: O(N) サイズ、因果関係を完全に特徴付ける
- 実装上の問題: N が動的に変わるシステム (mobile, IoT) ではベクトル長が膨張する

実用的には **Version Vector** (主に DynamoDB / Riak 等の eventual consistency DB で使われる) として動作。各レプリカが書き込みごとにベクトルを更新し、読み出し時に concurrent な書き込みを検出して siblings として返す (アプリが衝突解決を担う)。

### Dotted Version Vector (Almeida et al. 2014)
Riak が採用。クライアント数が増えると古典的バージョンベクトルが false concurrency を生む問題への対処。「サーバ ID で更新を整理し、クライアントが見たコンテキストを保持する」設計。古典 VV の理論的純粋性を犠牲にして実用性を高めた例。

### Hybrid Logical Clock (Kulkarni et al. 2014)
物理時計と論理時計のハイブリッド。CockroachDB, MongoDB が採用。物理時計をベースとしつつ、因果関係順序を保証する微小なオフセットを論理部分で持つ。NTP 同期された物理時計の利点 (人間可読、デバッグしやすい) と論理時計の正確性 (単調性、因果保証) を両立。

## 気づき・洞察

Lamport の偉大さは「物理時間という直感を捨てて、観測可能な因果関係だけで分散世界をモデル化した」哲学的飛躍にある。これは特殊相対性理論で「絶対同時」を捨てた革命と同型。**「分散システムでは絶対時間は存在せず、観測者ごとに異なる『過去の光円錐』しか定義できない」** というのが分散システムの根本的世界観。

ベクトル時計の比較不能性 (concurrent) は、欠陥ではなく **設計上の正直さ** である。「順序が決まらない」のではなく「順序を決める情報が不足している」を明示的に表現する。これに対して Lamport 全順序 (タイブレーク) は嘘を強いるので、応用 (CRDT 衝突解決等) では正しい意味で使えないことがある。

「O(N) サイズが嫌だから O(1) スカラーで済ませたい」という工学的圧力は常にあるが、**その代償として失う情報を理解せずに省略すると、見えないバグが入る**。Cassandra が初期に last-write-wins だけで運用され、ノード時計のずれで update が消えた事故は、論理時計を省略した代償。

## 他分野との接続

- **music (フーガ・対位法)**: 多声部音楽は「複数の独立した時間軸が並行する」構造。各声部はそれ自体の時間で進行し、特定の和声点 (cadence) で同期する。これは分散プロセスがメッセージで同期するモデルと同型。
- **anatomy (基底核 2026-04-25, 小脳 2026-04-24)**: 脳内の運動制御は複数の神経回路が並行動作し、特定タイミングで統合される。基底核ループと小脳ループは独立に時間進行するが、皮質で統合される。完全な「中枢時計」は存在せず、各回路は局所的な時刻感覚を持つ。
- **piano (ポリリズム 2026-03-26)**: 3 対 4 のポリリズムは 2 つの独立した拍子系が並行進行し、12 の最小公倍数で「同期」する。Lamport 時計と同じく、各リズムは独自時間軸で進み、合流点でのみ「順序」が定義される。
- **golf (Strokes Gained 2026-04-18)**: SG は「比較対象 (ベンチマーク)」を必要とする。ベクトル時計が「他の全プロセスとの相対時刻」で因果を定義するように、SG も「他選手の相対パフォーマンス」で価値を定義する。絶対値ではなく相対構造で意味が決まる点が共通。
