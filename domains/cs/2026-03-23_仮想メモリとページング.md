# 仮想メモリとページング — アドレス空間の抽象化と物理メモリ管理の設計論
**日付**: 2026-03-23
**分野**: コンピュータサイエンス
**タグ**: #仮想メモリ #ページング #TLB #ページテーブル #ページ置換 #CoW #ASLR #スラッシング

## 学んだこと

### 1. 歴史的動機 — なぜ仮想メモリが発明されたか

仮想メモリの起源は1962年、Manchester大学のTom Kilburnらが開発した**Atlas Computer**に遡る。Atlasは16,384ワードの高速コアメモリと96Kワードの磁気ドラムを持ち、両者の間のデータ転送をプログラマから隠蔽する**「one-level store」**という概念を導入した。

それ以前のプログラマは**オーバーレイ（overlay）**と呼ばれる手法で、プログラムの各部分を手動でメモリにロード/アンロードしていた。これは:
- プログラマの生産性を著しく低下させる（Kilburnは最大3倍の生産性向上を見込んだ）
- プログラムのサイズが物理メモリ量に制約される
- メモリ管理のバグが頻発する

Kilburnの洞察は、**ユーザープログラムのアドレスを仮想アドレスとし、仮想→物理の変換をハードウェアとOSの協調で自動化する**というものだった。2022年にIEEEマイルストーンとして認定されている。

**設計上の核心的判断**: メモリ管理の責任をプログラマからシステムに移す。これはCPUキャッシュの設計（2026-03-20エントリ）と同じパターン — ハードウェアの物理的制約をソフトウェアから隠蔽する抽象化層の導入。

### 2. アドレス変換の基本機構 — ページテーブルとTLB

#### ページングの基本構造

仮想アドレス空間を固定サイズの**ページ**（通常4KB）に、物理メモリを同サイズの**フレーム**に分割する。ページテーブルが仮想ページ番号（VPN）→物理フレーム番号（PFN）のマッピングを保持する。

```
仮想アドレス: [ VPN (上位ビット) | オフセット (下位12ビット) ]
                    ↓ ページテーブル参照
物理アドレス: [ PFN (上位ビット) | オフセット (そのまま) ]
```

各ページテーブルエントリ（PTE）には以下のメタデータが含まれる:
- **Present bit**: 物理メモリに存在するか（0ならページフォルト発生）
- **Read/Write bit**: 書き込み可能か
- **User/Supervisor bit**: ユーザーモードからアクセス可能か
- **Accessed bit**: 参照されたか（ページ置換アルゴリズムが使用）
- **Dirty bit**: 書き込まれたか（ページアウト時にディスク書き戻しが必要か判断）

**なぜ固定サイズのページか？** セグメンテーション（可変サイズ）は外部フラグメンテーションを引き起こす。固定サイズのページングは外部フラグメンテーションを完全に排除し、空きフレーム管理をビットマップや空きリストで単純化できる。内部フラグメンテーション（最後のページの未使用領域）は平均でページサイズの半分だが、4KBなら許容範囲。

#### TLB — アドレス変換のキャッシュ

毎回のメモリアクセスでページテーブルを参照すると、メモリアクセス回数が倍増する。これを解決するのが**TLB（Translation Lookaside Buffer）**。

TLBはCPU内のMMU（Memory Management Unit）に実装された、アドレス変換結果の専用キャッシュ。CPUキャッシュエントリ（2026-03-20）で学んだキャッシュ設計と同じ原理が適用されるが、いくつかの重要な違いがある:

| 特性 | CPUデータキャッシュ | TLB |
|------|---------------------|-----|
| エントリ数 | 数千～数万ライン | 64～2048エントリ |
| ミスペナルティ | 数十～数百サイクル | ページテーブルウォーク（数百サイクル） |
| 結合度 | 8-16 way | フルアソシアティブが多い |
| カバレッジ | エントリ数 × 64B | エントリ数 × ページサイズ |

TLBのフルアソシアティブ設計は、エントリ数が少ないため検索コストが許容でき、かつミスペナルティが非常に大きいためヒット率最大化が最優先だから。

**TLBカバレッジの問題**: 典型的なTLBが1024エントリ × 4KB = 4MBしかカバーできない。Metaの報告では64GBサーバーで**実行サイクルの約20%がTLBミス処理**に費やされている。これがhuge pages導入の動機。

#### ハードウェアページテーブルウォーク（x86-64の場合）

x86-64では、TLBミス時にハードウェアが自動的にページテーブルを走査する:

```
CR3レジスタ → PML4[bits 47:39] → PDPT[bits 38:30] → PD[bits 29:21] → PT[bits 20:12] → PFN + offset[bits 11:0]
```

各レベルで9ビットのインデックス（512エントリ × 8バイト = 4KB = 1ページ）。この設計は、ページテーブル自体がページ単位で管理できるという再帰的な美しさを持つ。

### 3. マルチレベルページテーブル — 疎なアドレス空間への対処

#### 単一レベルの問題

48ビット仮想アドレス空間、4KBページの場合:
- ページテーブルエントリ数: 2^36 = 約687億
- エントリあたり8バイトとして: **256GB のページテーブル**（プロセスごと！）

しかし実際のプロセスは仮想アドレス空間のごく一部しか使わない。テキスト領域は低アドレス、ヒープはその上、共有ライブラリが中間、スタックは最上位。間の広大な領域は未使用。

#### マルチレベルの解法

OSTEP (Arpaci-Dusseau) が明快に説明するアプローチ:
1. ページテーブルをページサイズの単位に分割する
2. 全エントリが無効なページテーブルページは**割り当てない**
3. 上位のページディレクトリが、どのページテーブルページが有効かを追跡

```
疎なアドレス空間でのメモリ使用量の比較:
  単一レベル: O(仮想アドレス空間サイズ / ページサイズ)  ← アドレス空間に比例
  マルチレベル: O(使用中のページ数)                      ← 実使用量に比例
```

**本質的なトレードオフ**: 空間効率（ページテーブルサイズ削減）と引き換えに、TLBミス時のアドレス変換コストが増加する。4レベルなら4回のメモリアクセスが必要。しかしTLBのヒット率が高い（典型的に99%+）ため、このトレードオフは成り立つ。

#### x86-64の4レベルページテーブル

| レベル | 名称 | インデックスビット | 管理単位 |
|--------|------|---------------------|----------|
| 4 | PML4 | bits 47:39 | 512GB領域 |
| 3 | PDPT | bits 38:30 | 1GB領域 |
| 2 | PD | bits 29:21 | 2MB領域 |
| 1 | PT | bits 20:12 | 4KB領域 |

PML4は1つで512エントリ = 512 × 512GB = 256TBの仮想アドレス空間をカバー（48ビット）。

#### 5レベルページテーブル（LA57）

Intel Ice Lakeで導入。PML5を追加し、仮想アドレスを48ビット→57ビットに拡張。アドレス可能な仮想メモリが256TiB→128PiBに。Linux kernel 4.14以降でサポート。

**なぜ必要か**: 物理メモリの大容量化（数TBのサーバー）と、メモリマップドI/Oの増加により48ビット（256TB）では不足するユースケースが出現。ただし、5レベル化はTLBミス時のウォークコストをさらに1段増やすペナルティがある。

### 4. 逆ページテーブル — 物理メモリ中心の設計

通常のページテーブルは仮想アドレス空間に比例したサイズだが、逆ページテーブルは**物理メモリのフレーム数に比例**する。物理フレームごとに1エントリで、「このフレームはどのプロセスのどの仮想ページに対応するか」を記録。

**採用例**: PowerPC, UltraSPARC, IA-64

**利点**: メモリ使用量が物理メモリサイズのみに依存し、仮想アドレス空間の広さに影響されない。64ビットアドレス空間でも4GB物理メモリなら約100万エントリで済む。

**欠点**:
- **検索がO(n)**: VPN→PFNの変換にハッシュテーブルが必要。ハッシュ衝突時はチェイニングで性能劣化
- **空間局所性の破壊**: ハッシュにより隣接する仮想ページのPTEが物理的に散在。ページテーブルウォーク自体のキャッシュ効率が悪化
- **共有メモリの複雑化**: 1フレームに1エントリなので、複数プロセスからの共有マッピングにチェイニングが必要

この設計は現代のx86-64では採用されていない。マルチレベルページテーブルの方が、TLBミス時のウォークの予測可能性とキャッシュ効率で優位。

### 5. ページフォルト処理

ページフォルトは、PTEのPresent bitが0の仮想アドレスへのアクセスで発生するCPU例外。3種類に分類される:

1. **Minor fault（ソフトフォルト）**: ページは物理メモリ上にあるがPTEが未設定。マッピングのみ更新
2. **Major fault（ハードフォルト）**: ページがディスクにあり、I/Oが必要。数ミリ秒のコスト
3. **Invalid fault**: 不正なアドレスへのアクセス。SIGSEGV送信

ページフォルト処理のフロー:
```
1. CPU例外発生 → OS のページフォルトハンドラに制御移行
2. フォルト原因の判定（アドレスは有効か、権限は正しいか）
3. 空きフレームの確保（なければページ置換アルゴリズムで犠牲ページを選択）
4. ディスクからページを読み込み（Major faultの場合）
5. PTE更新、TLB更新
6. フォルトを起こした命令を再実行
```

### 6. ページ置換アルゴリズム — 限られたフレームの最適利用

物理メモリが不足した時、どのページを追い出すかを決定するアルゴリズム。

#### Bélády の最適アルゴリズム（OPT）

**未来の全アクセスパターンを知っている**前提で、「次に使われるのが最も遠いページ」を追い出す。理論上の下限を与え、他のアルゴリズムの評価基準として使われる。

実装不可能だが重要な理由:
- 理論的最適解として他のアルゴリズムのベンチマーク
- **Bélády の異常（Bélády's anomaly）**: FIFOではフレーム数を増やすとページフォルトが増える場合がある。OPTとLRUはスタックアルゴリズムであり、この異常が起きないことが証明されている

#### LRU（Least Recently Used）

「最も長くアクセスされていないページを追い出す」。過去のアクセスパターンから未来を予測する — 時間的局所性の直接的な活用。

**理論的にはOPTに近い性能を出すが、厳密な実装は高コスト**:
- 全ページにタイムスタンプを記録 → メモリアクセスごとの更新が不可能
- スタックやカウンタベースの実装も、ハードウェアの臨界パスに載せるには重すぎる

#### Clock（Second-Chance）アルゴリズム

LRUの実用的近似。PTEのAccessed bit（参照ビット）を活用:

```
循環リストとしてページを管理（時計の針のイメージ）
1. 針の位置のページの参照ビットを確認
2. 参照ビット = 1 → 0にクリアして針を進める（セカンドチャンスを与える）
3. 参照ビット = 0 → このページを追い出す
```

**なぜClockがLRUの良い近似になるか**: 参照ビットは「最近アクセスされたかどうか」の1ビット情報だが、針の巡回周期が「最近」の時間窓を暗黙的に定義する。メモリ圧迫が高いほど針が速く回り、時間窓が短くなる — 適応的な振る舞い。

Linux kernel は LRU に基づく**二重リスト方式（Active/Inactive リスト）**を採用。Multi-Generation LRU（MGLRU, kernel 6.1）でさらに精緻化されている。

#### Working Set モデル（Denning, 1968）

Peter Denningが定式化。**ワーキングセット** W(t, Δ) は、時刻tから過去Δ時間内にアクセスされたページの集合。

核心的洞察: プログラムの実行には**局所性のフェーズ**がある。各フェーズで頻繁にアクセスするページ集合（ワーキングセット）が存在し、そのサイズは比較的安定している。ワーキングセットが物理メモリに収まれば高性能、収まらなければスラッシング。

### 7. スラッシング — 仮想メモリシステムの崩壊

**スラッシング**は、プロセスのワーキングセットの合計が物理メモリを超えた時に発生する正のフィードバックループ:

```
ワーキングセット > 物理メモリ
→ 頻繁なページフォルト → ディスクI/O待ちでCPU利用率低下
→ OSが「CPUが暇だ」と判断し多重度を上げる
→ さらにメモリ不足 → さらなるページフォルト → ...
```

Denning (1968) "Thrashing: Its Causes and Prevention" がこの現象を分析し、ワーキングセットモデルに基づく防止策を提案:
- 各プロセスのワーキングセットサイズを追跡
- 全プロセスのワーキングセット合計が物理メモリを超えたら、一部のプロセスをスワップアウト（多重度を下げる）
- **局所性の原理により、ワーキングセット管理は近似最適でスラッシング免疫を実現できる**

TCP輻輳崩壊（2026-03-21エントリ）との構造的類似: どちらも共有リソース（メモリ/帯域幅）の過剰使用が正のフィードバックループで崩壊に至る。解法も類似 — 自己の使用量を能動的に制御する分散メカニズム。

### 8. デマンドページングとプリページング

#### デマンドページング

プロセス起動時にページを一切ロードせず、アクセスされた時点で初めてページフォルトを介してロードする。

**利点**:
- プロセス起動が高速（全コードをロードする必要がない）
- 使われないページはメモリを消費しない
- 実行されない関数のコードページは永遠にロードされない可能性

**コスト**: 実行初期にページフォルトが集中（cold start problem）

#### プリページング

ページフォルト発生前に、必要になりそうなページを先読みする。CPUキャッシュのプリフェッチ（2026-03-20エントリ）と同じ「予測に基づく先読み」パターン。

ディスク上で連続するページを一括読み込みすることで、ランダムI/Oをシーケンシャルに変換。ただし予測が外れれば無駄なI/Oとメモリ消費。

### 9. Copy-on-Write（CoW）

`fork()` システムコールの最適化として極めて重要。子プロセスに親のアドレス空間を物理的にコピーする代わりに:

```
1. 親子で同一の物理フレームを共有
2. 両方のPTEを読み取り専用に設定
3. どちらかが書き込みを試みるとページフォルト発生
4. その時点で初めてページをコピーし、書き込みプロセスに専用のフレームを割り当て
```

**なぜ重要か**: Unix系OSでは `fork()` + `exec()` パターンが頻出。`exec()` はアドレス空間を完全に置き換えるため、`fork()` でのコピーは無駄になる。CoWにより、`exec()` が即座に呼ばれる場合のコピーコストはほぼゼロ。

仮想メモリの権限ビットを利用してトラップを発生させるという手法は、**ハードウェア機構の「目的外使用」**の好例。同様の手法はdirty bit（ページの変更追跡）でも使われる。

### 10. メモリマップドファイル

`mmap()` システムコールは、ファイルの内容を仮想アドレス空間に直接マッピングする:

```c
void *addr = mmap(NULL, length, PROT_READ, MAP_PRIVATE, fd, offset);
// addr を通じてファイル内容にメモリアクセスと同じインタフェースでアクセス
```

**仕組み**: デマンドページングと同じ機構を利用。マッピング時点では物理メモリを消費せず、アクセス時にページフォルト→ページキャッシュからフレームをマップ。

**利点**:
- `read()`/`write()` のシステムコールオーバーヘッドを回避（カーネル↔ユーザー間のデータコピーが不要）
- 複数プロセスが同一ファイルをmmapすると物理メモリを共有
- 実行バイナリと共有ライブラリは mmap でロードされる — これが共有ライブラリが「共有」される仕組み

**注意点**:
- minor page faultのコストは存在する
- 32ビットアーキテクチャではアドレス空間の制約（4GB）が問題

### 11. Huge Pages（スーパーページ）

TLBカバレッジ問題への直接的対策。ページサイズを大きくすることで、1つのTLBエントリでカバーする物理メモリ量を増やす。

x86-64での選択肢:
- **4KB**: デフォルト。細粒度制御、内部フラグメンテーション最小
- **2MB**: 512倍のカバレッジ。`MAP_HUGETLB` または Transparent Huge Pages (THP) で使用
- **1GB**: さらに大きいが、連続物理メモリの確保が困難

**トレードオフ**:
- TLBカバレッジ向上 ↔ 内部フラグメンテーション増加
- ページフォルト処理の粒度が粗くなる
- huge pageの分割・統合のオーバーヘッド（THP の場合）
- メモリ圧迫時にhuge pageの確保が困難（物理的に連続した領域が必要）

Linux THPはカーネルがバックグラウンドで透過的に4KBページを2MBに統合する。アプリケーション変更不要だが、khugepaged のCPUコストと予期しないレイテンシスパイクの問題がある。

### 12. ASLR — 仮想メモリによるセキュリティ

**Address Space Layout Randomization（ASLR）**は、仮想メモリの柔軟性をセキュリティに活用する技術。プロセスのコード、データ、ヒープ、スタック、共有ライブラリの配置アドレスをランダム化する。

**歴史**: Linux PaXプロジェクトが2001年に提案。OpenBSD 3.4 (2003)、Linux (2005)、Windows Vista (2006) で採用。

**なぜ有効か**: バッファオーバーフロー等の脆弱性を悪用するには、攻撃対象のアドレスを知る必要がある。ASLRにより攻撃者は正しいアドレスを推測しなければならず、探索空間が指数的に増大する。

**限界**: メモリリーク（情報漏洩）の脆弱性で1つのアドレスが漏れると、相対位置から他のアドレスを推測可能。DEP（Data Execution Prevention）、スタックカナリア、CFI（Control Flow Integrity）との多層防御が前提。

**仮想メモリの本質的な柔軟性**: 仮想アドレスと物理アドレスの分離により、プログラムの論理的動作を変えずにメモリ配置を自由に変更できる。当初セキュリティは設計目的ではなかったが、抽象化層が後から新たな用途を獲得した好例。

### 13. TLBシュートダウン — マルチコア時代の課題

マルチコアシステムでは各コアが独自のTLBを持つ。ページテーブルが変更された際（ページアンマップ、権限変更）、該当エントリをキャッシュしている全コアのTLBを無効化する必要がある。

**TLBシュートダウンの手順**:
1. ページテーブルを変更するコアがIPI（Inter-Processor Interrupt）を他コアに送信
2. 各コアが割り込みを受け、該当TLBエントリを無効化
3. 完了を通知

**コスト**: IPIレイテンシは1-5マイクロ秒/コア。対象コア数に比例してスケールしない。MESIのスヌーピング（2026-03-20エントリ）がキャッシュラインレベルで行うコヒーレンスを、TLBはソフトウェア（OS）が明示的にIPIで行わなければならない。

**ASID（Address Space Identifier）**: TLBエントリにプロセスIDを付与し、コンテキストスイッチ時のTLB全フラッシュを回避する。PCIDはx86-64でのASID実装。

## 気づき・洞察

### 仮想メモリは「メモリの仮想化」

仮想メモリの本質は、物理リソース（メモリ）の**仮想化（virtualization）**。各プロセスに「自分専用の広大なメモリ空間がある」という幻想を提供する。これはCPUの仮想化（タイムシェアリング）と同じパターン:

| リソース | 仮想化手法 | 抽象化 |
|----------|-----------|--------|
| CPU | タイムシェアリング | プロセス |
| メモリ | 仮想メモリ | アドレス空間 |
| ディスク | ファイルシステム | ファイル |

### キャッシュ設計の再帰的適用

TLBはキャッシュの一種であり、CPUキャッシュと同じ設計原則（局所性の活用、ミスペナルティとヒット率のトレードオフ）が適用される。しかしTLBは:
- エントリ数が少ないためフルアソシアティブが許容される
- ミスペナルティがデータキャッシュより大きい（ページテーブルウォーク）
- コヒーレンスがハードウェアではなくソフトウェア（IPIベースのシュートダウン）

### 「遅延評価」パターンの普遍性

デマンドページング、CoW、mmapの遅延マッピング — すべて「必要になるまでコストのかかる処理を遅延させる」戦略。プログラミングにおけるlazy evaluationと同じ原理がOS設計にも現れる。

## 他分野との接続

- **CPUキャッシュ（cs, 2026-03-20）**: TLBはキャッシュの特殊形態。キャッシュ設計で学んだ結合度・置換ポリシー・コヒーレンスの概念がすべて適用される。VIPTキャッシュ設計ではTLBとL1キャッシュのアクセスを並列化する工夫がある
- **B木（cs, 2026-03-19）**: ページサイズ（4KB）はディスクブロックサイズと一致させる設計。B木のノードサイズ最適化と同じ「転送単位への最適化」パターン
- **TCP輻輳制御（cs, 2026-03-21）**: スラッシングと輻輳崩壊は、共有リソースの過剰使用→正のフィードバックループ→システム崩壊という同一構造。防止策もフィードバック制御に基づく
- **解剖学・身体**: ワーキングセットモデルは「必要なものを必要な時に適切な量だけ」という最小限リソース管理。身体のエネルギー配分にも通じる考え方
- **ゴルフ（メンタル）**: コース上で「全ホールを同時に考えない」＝ワーキングセットを小さく保つ戦略との類比

## 次に深掘りしたいこと

- **VIPT（Virtually Indexed, Physically Tagged）キャッシュ**: TLBアクセスとL1キャッシュアクセスの並列化の仕組み。エイリアシング問題
- **NUMAと仮想メモリ**: ソケット間のメモリアクセスレイテンシ差をページ配置で最適化するnumactl, mbind
- **KSM（Kernel Same-page Merging）**: CoWの応用としての重複ページ統合（仮想化環境で有用）
- **IOMMU**: デバイスからのDMAに対する仮想メモリ保護
- **Persistent Memory（PMEM）とDAX**: mmapとNVDIMMの組み合わせによるバイパスページキャッシュ
- **Linux MGLRU（Multi-Generation LRU）**: kernel 6.1で導入された改良ページ置換の設計思想
