**分野**: コンピュータサイエンス

## 学んだこと

### 中心命題：理論は1968年に完成していたのに、世界は遅い実装を選んだ
この分野の最も衝撃的な事実は、**「正規表現は線形時間で確実に照合できる」という結果が Thompson の 1968 年 CACM 論文（"Regular Expression Search Algorithm"）で実用化されていた**にもかかわらず、Perl・PCRE・Python `re`・Java `java.util.regex`・JavaScript（V8 Irregexp）といった現代の主要エンジンの大半が、最悪指数時間に陥りうるバックトラッキング型を採用している、という点だ。Russ Cox の "Regular Expression Matching Can Be Simple And Fast"（2007, swtch.com）はこのギャップを暴いた決定的な一次資料で、本エントリの背骨をなす。

**数字が残酷**だ。病的なパターン `a?ⁿaⁿ`（`a?` を n 回 + `a` を n 回）に対し、n=29 で **Perl は 60 秒以上**、Thompson NFA シミュレーションは **20 マイクロ秒**——百万倍の差。n=100 では Perl は理論上「10¹⁵ 年以上」、NFA は 200 マイクロ秒未満。同じ「正規表現マッチング」という看板の下で、計算量クラスが指数 vs 線形に分かれている。

### 形式言語の階層：正規言語と有限オートマトンの等価性
基礎を押さえる。Kleene の定理（1956）が **「正規表現が表す言語 = 有限オートマトンが受理する言語 = 正規言語」** という三位一体を確立した。有限オートマトンには2種類ある：

- **DFA（決定性）**: 各状態・各入力文字に対し遷移先が一意。1文字読むごとに1状態。照合は入力長 n に対し **厳密に O(n)**、しかも1文字あたり定数仕事。だが状態数が爆発しうる。
- **NFA（非決定性）**: 同一入力に複数遷移を許し、ε遷移（入力を消費しない遷移）も持つ。「全ての可能性を同時に追う」モデル。Rabin-Scott（1959）の **部分集合構成法（subset construction）** で任意の NFA は等価な DFA に変換できる——ただし NFA の k 状態に対し DFA は最悪 **2^k 状態**（状態爆発）。

ここに最初のトレードオフがある。DFA は実行が速いが構築・保持が高くつき、NFA は構築が安い（正規表現サイズに線形）が素朴に実行すると遅くなりうる。**設計の核心は「この2極のどこに座るか」だ。**

### Thompson 構成法と「NFA シミュレーション」——指数を回避する鍵
正規表現 → NFA の変換が **Thompson 構成法**。各演算子（連接・選択 `|`・閉包 `*`）に対し ε遷移つきの小さな部品を再帰的に組み立てる。**正規表現の文字・メタ文字1個につき状態1個**しか作らないので、NFA のサイズは正規表現長に線形（重要：照合時間の線形性の前提）。Thompson の 1968 年の天才的工夫は、NFA 状態を**機械語の小片**として表現し、可能状態のリストを関数呼び出し命令の列として実行時に動的生成したこと——1960 年代のマシンでの JIT 的発想だった。

線形時間を生む決定的アイデアは **「NFA を多状態で同時シミュレートする」** こと。バックトラッキングが「1つの経路を試し、失敗したら戻ってやり直す」のに対し、Thompson 法は **現在到達可能な NFA 状態の集合を1つ持ち、入力を1文字読むごとに集合全体を一斉に進める**。状態数は高々 m（正規表現長）なので、各ステップの仕事は O(m)、入力 n 文字で **O(mn)**。

> **核心**: 効率の源泉は「どの経路で到達したか」を**捨てる**こと。到達可能な**状態の集合だけ**を覚え、経路の履歴は持たない。バックトラッキングは経路を覚えるから 2ⁿ 通りを辿る羽目になる。Thompson は集合を覚えるから経路の組合せ爆発を**情報として捨てている**。

これは事実上、部分集合構成で得られる DFA を**実行時に遅延構築（lazy/on-the-fly DFA）している**のと等価だ。状態集合をキャッシュすれば、訪れた DFA 状態だけを作るので、2^k の状態爆発を「実際に必要な分だけ」に抑えられる。これが grep の高速さの正体である。

### Glushkov 構成という別解
Thompson と並ぶもう一つの正規表現→NFA 変換が **Glushkov 構成（position automaton, 1961）**。各記号出現に一意の「位置」を割り当て、firstpos/lastpos/followpos 集合を計算して位置を状態とする。決定的な違いは **ε遷移を一切作らない**こと。状態数は「正規表現の記号数 + 1」ちょうど。NYU の Mohri らが示したように、**Thompson NFA から ε遷移を除去すると Glushkov NFA が得られる**——両者は同じ自動機の別の視点だ。ε-free であることはコンパイラ的に扱いやすく、ハードウェア NFA 実装や一部の高速マッチャ（dual Glushkov など）で好まれる。

### Unix の系譜：知っていたのに使わなかった歴史
皮肉な歴史がある。Thompson は QED エディタ（CTSS, 1960年代）に正規表現を持ち込み、それが ed・grep へ流れた。**ところが ed や初期 grep はバックトラッキングを使った**——Thompson 自身がより良いアルゴリズムを知っていたのに。Al Aho の **egrep**（Seventh Edition Unix, 1979）が初めて完全な正規表現構文を**事前計算した DFA**で実装。Eighth Edition（1985）では Rob Pike が sam エディタ用に部分マッチ追跡つきの効率的 NFA シミュレーションを書いたが、**この技法は注目されなかった**。代わりに Henry Spencer の 1980 年代のバックトラッキング再実装が広く採用され、Perl・PCRE・Python・Tcl の祖先になった。**速いアルゴリズムが「忘れられ」、遅いアルゴリズムが事実上の標準になった**——技術の経路依存性の教科書的事例。

### なぜバックトラッキングを選ぶのか：後方参照という麻薬
ここが最も深い論点だ。なぜ世界は遅い方を選んだのか？答えは**機能**——とりわけ**後方参照（backreference, `\1`）**だ。`(cat|dog)\1` は「最初のキャプチャ群が捕まえた**まさにその部分文字列**」の繰り返しを要求する。これは「マッチするか否か」だけでなく「何を捕まえたか」を覚えて後で文字列比較する必要があり、**正規言語の枠を超える**。

決定的な事実：**後方参照つき正規表現は `{wcw | w ∈ {a,b}*}` のような非正規言語を表せる**。ポンプの補題（pumping lemma）が証明する有限オートマトンの限界——`aⁿbⁿ` のような「数を数える」言語は正規言語ではない——を、後方参照は**易々と踏み越える**。つまり **「正規表現」と呼ばれているものは、もはや形式言語論の正規言語ではない**。名前が実体を裏切っている。

そして致命的な帰結：**Aho が示したように、後方参照つき正規表現のマッチング問題は NP 完全**（Perl 正規表現マッチングは NP 困難, perl.plover.com）。包含・等価・普遍性といった判定問題は**決定不能**。つまり後方参照を一般的に効率よく実装する方法は**誰も知らない**（おそらく存在しない）。NFA シミュレーションは「状態集合」という有限の情報しか持たないから、無限に多様な「捕まえた部分文字列」を表現できない——後方参照は構造的に多状態シミュレーションと相容れない。バックトラッキングは経路（=どの分岐でどう捕まえたか）を全部覚えるから後方参照を扱えるが、その代償が指数時間だ。

> **これが理論と実用のギャップの正体**: 「後方参照（と look-around）が欲しい」→「バックトラッキングしかない」→「最悪指数時間を受け入れる」。POSIX が後方参照を標準化したことで、実装はバックトラッキングに**ロックイン**された。機能要求が計算量クラスを決めてしまった。

### Catastrophic Backtracking と ReDoS — 実害としての指数時間
バックトラッキングが指数爆発する具体機序が **catastrophic backtracking**。`(a*)*` や `(a+)+$`、`(a|a)*$` のように **「曖昧な反復のネスト」** があると、エンジンは入力を分割する 2ⁿ 通りの方法をすべて試す。マッチ成功なら早く終わるが、**わずかに失敗する入力**（末尾だけ違う）に対して全分割を網羅し尽くす。

これを攻撃に使うのが **ReDoS（Regular Expression Denial of Service）**。攻撃者が病的入力を送り込めば、1リクエストで CPU を1コア占有させられる。脆弱な正規表現がリクエストパスにあれば、少数のリクエストでサービス全体を落とせる。

### Cloudflare 2019-07-02 障害——27分間、世界の一部が止まった
これが現実に起きた最も有名な事例（Cloudflare 公式 postmortem, John Graham-Cumming）。WAF の管理ルール（XSS 対策、simulate モードのはずだった）に含まれた正規表現：

```
(?:(?:\"|'|\]|\}|\\|\d|(?:nan|infinity|true|false|null|undefined|symbol|math)|\`|\-|\+)+[)]*;?((?:\s|-|~|!|{}|\|\||\+)*.*(?:.*=.*)))
```

病巣は末尾の **`.*.*=.*`**。`=` を含まない入力に対し、最初の `.*` が貪欲に全消費 → 失敗 → 1文字戻す → 2番目の `.*` が試す → … と全分割を試行する。Cloudflare の計測では、`x=x` は 23 ステップ、`x=xxxxxxxxxxxxxxxxxxxx`（x が20個）は **555 ステップ**——線形でない伸び。末尾を `;` 付きにした `.*.*=.*;` では同じ入力が **5,353 ステップ**に跳ね上がった。

これがグローバルにデプロイされ（カナリアも段階展開もなく全エッジへ一斉）、**CPU が全サーバでほぼ 100% に張り付き、トラフィックが 80% 落ちた**。13:42 UTC デプロイ、14:09 復旧——**27分間の障害**。Discord・Shopify・Notion など多数のサイトが巻き添え。当時のエンジンは **PCRE（バックトラッキング、暴走を防ぐ機構なし）**。Cloudflare の結論は明快：「パターンをより具体的に書き直す以外の真の解決策は、**このバックトラッキング機構を持つ正規表現エンジンから移行すること**」——**RE2 か Rust regex（共に実行時保証あり）**への移行を表明した。

### 線形時間保証エンジンの設計判断：RE2 と Rust regex
**RE2**（Google, Russ Cox 主導, 2006 開始, Google Code Search 由来）の設計思想は徹底している：**「バックトラッキングでしか解けない構文はサポートしない」**——だから **後方参照と look-around を意図的に削除した**。これにより**入力長に線形のマッチ時間を保証**する。

実装は精緻なハイブリッド：
- **最大4つの DFA**を持つ（前方の first-match / longest-match、後方の2つ＝unanchored 検索用）。
- DFA は**遅延構築**し、状態を LRU でキャッシュ。フル DFA は最悪 O(2^k) 空間だが、**メモリ予算を超えたら LRU で状態を捨て、必要なら NFA シミュレーションへフォールバック**。
- だが**この NFA は通常の NFA と違い、決して指数時間にならない**（= Thompson 多状態シミュレーション）。

結果として **「速度（DFA）と空間上限（NFA フォールバック）の両立、しかも全域で線形時間保証」** を達成。Go の `regexp` パッケージはこの哲学を標準ライブラリに組み込み、設計段階で後方参照を諦めることで「どんな入力でも安全」を言語の保証にした。Rust の `regex` クレート（RFC 0042）も同じ系譜——「正規言語に閉じる代わりに線形性を保証する」という明示的選択。

## 気づき・洞察

- **「名前が実体を超えてしまった」希有な事例**。形式言語論の「正規表現」は正規言語に対応するはずだった。だが実務の「正規表現」は後方参照・look-around で非正規言語まで表現し、Kleene の等価性定理が成り立たない代物になった。理論用語が実装の都合で意味を変質させられた——標準化（POSIX）が経路依存を固定化する力の例。

- **計算量クラスは「機能セット」が決める、という設計教訓**。後方参照という1機能を入れるか否かが、線形 vs NP困難という**計算量クラスそのものを切り替える**。これは [[計算可能性理論 — チューリングマシン・停止問題・Rice の定理]] で見た「決定不能を回避する＝入力（言語）を制限する工学」と完全に同型だ。RE2 が後方参照を捨てたのは、eBPF verifier が無制限ループを禁じて停止性を決定可能にしたのと、全関数型言語が構造的再帰だけ許してチューリング完全性を犠牲に停止保証を得たのと、**まったく同じ設計パターン**。「理論が不可能と言う ⇒ 入力を制限する」の典型。

- **「履歴を捨てることで速くなる」という反直観**。Thompson NFA が線形なのは、経路情報を捨てて状態集合だけ持つから。情報を**減らす**ことが計算量を減らす。これは [[Burrows-Wheeler 変換 (BWT) と FM-index]] の「逆順処理で指数探索が単調絞り込みになる」、HyperLogLog の「正確さを諦めて巨大スケールに到達」と同じ系譜——**正しい忘却が効率を生む**。

- **遅延 DFA 構築は「実行時の世代別キャッシュ」**。RE2 が DFA 状態を LRU でキャッシュし予算超過で捨てる構造は、[[ガベージコレクション]] の世代別管理や Kademlia の k-bucket LRU と発想が通底する。「全部を事前計算（2^k 状態）」と「毎回計算（NFA シミュレーション）」の間に「使った分だけ memoize」を挟む——RUM 空間（Read/Update/Memory）でのバランス点の選択。

- **削除こそ最大の設計成果、再び**。RE2 の本質は「後方参照を消したこと」、すなわち [[TLS 1.3 ハンドシェイクの設計哲学]] の「削除こそが最大の設計成果」と完全に同じ哲学。TLS 1.3 は危険な暗号スイートを削除して安全を得、RE2 は危険な構文を削除して線形性を得た。**「柔軟性は脆弱性の母」「制限が安全性を生む」**がここでも成立する。Cloudflare 障害は「柔軟な PCRE が DoS を許した」一方、Go/RE2 は「制限した設計が DoS を原理的に不可能にした」対比そのもの。

- **理論→実用の数十年遅延**。Thompson 1968 が線形アルゴリズムを実用化していたのに、主流が線形保証エンジン（RE2 2006-2009, Go 2009, Rust regex 2014-）へ動くまで **40年**かかった。これは [[ラムダ計算と型システムの基礎]] の線形型（Girard 1987 → Rust 2015）、ZKP（1985 → Zcash 2016）と同じ「基礎研究の数十年遅れの翻訳」パターン。**良いアルゴリズムが存在することと、それが採用されることは別問題**——技術選択は計算量だけでなく機能要求・歴史・標準化の経路依存で決まる。

## 他分野との接続

- **[[計算可能性理論 — チューリングマシン・停止問題・Rice の定理]]**: 正規言語は計算階層（チョムスキー階層）の最底辺。後方参照付き正規表現の包含・等価判定は**決定不能**——停止問題の親戚。「制限すれば判定可能、機能を足すと決定不能/NP困難」という階層の登り方が両分野で共通。
- **[[ラムダ計算と型システムの基礎 — STLC から System F、Curry-Howard へ]]**: 「表現力 vs 決定可能性」のトレードオフが同型。型推論可能性（HM）vs 表現力（System F は型推論不能）の対立は、線形時間（正規言語）vs 後方参照（NP困難）と同じ構図。
- **[[SSA形式とコンパイラ最適化]]**: NFA→DFA の部分集合構成も、SSA 構築も「再帰的構文木からのグラフ構築」。コンパイラのレキサ（字句解析）は正規表現→DFA そのもの（lex/flex は事前計算 DFA）。Thompson が NFA を機械語に直接コンパイルした発想は SSA/sea-of-nodes の JIT 思想の先駆。
- **[[Burrows-Wheeler 変換 (BWT) と FM-index]]**: 文字列照合の2大潮流——オートマトン（正規表現）vs 接尾辞構造（BWT）。FM-index の backward search も「履歴を持たず範囲だけ絞る」点で Thompson 多状態シミュレーションと哲学的に同型。
- **anatomy / golf との同型**: 「全可能性を試す（バックトラッキング）vs 状態集合で並行追跡（NFA）」は、把持力制御やコースマネジメントで見た「全数探索を避け良い近似で妥協する」工学パターンの再来。

## 次に深掘りしたいこと

- **Pike の VM アプローチ（Thompson VM）の実装詳細**: 正規表現を「バイトコード」(`char`/`split`/`jmp`/`match`)にコンパイルし、スレッド集合を進める仮想機械として実装する手法（Russ Cox regexp2 論文）。submatch 追跡（キャプチャ群）を線形時間で行う Pike VM の具体構造。
- **JavaScript V8 Irregexp と最近の non-backtracking 実験**: V8 は backtracking だが、ReDoS 対策として実験的な線形時間エンジン（experimental regexp flag）を導入。なぜ後方参照互換性を保ちつつ線形化が難しいか。
- **静的 ReDoS 検出**: 正規表現を静的解析して exponential/polynomial backtracking 脆弱性を検出する手法（NFA の「曖昧度 ambiguity」解析、IDA/EDA 判定、recheck・rxxr2 ツール）。substructural logic を使った解析（arxiv 1405.7058）。
- **後方参照の多項式時間部分クラス**: 一般には NP困難だが、特定の制限付き後方参照は多項式時間で解けるという最近の結果（arxiv 1903.05896, MFCS 2025）。memoization による backtracking の多項式時間化（Davis らの work）。
- **Antimirov 微分（partial derivatives）と Brzozowski 微分**: 正規表現を「微分」して直接 DFA を作る代数的アプローチ。状態最小性の観点での Glushkov/Antimirov/follow automaton の統一理論（Mohri らの NYU TR）。
- **ハードウェア正規表現マッチング**: FPGA/専用 ASIC（Intel Hyperscan, ネットワーク IDS の DPI）での NFA 並列実装。SIMD を使った Hyperscan の「文字列分解 + ビット並列」設計。
