# CPUパイプラインと分岐予測 — 投機的実行のマイクロアーキテクチャ
**日付**: 2026-04-19
**分野**: コンピュータサイエンス
**タグ**: #コンピュータアーキテクチャ #分岐予測 #投機実行 #セキュリティ #パイプライン #TAGE #Spectre

## 学んだこと

### 1. パイプライン進化 — なぜ深く、広くなったか
Patterson & Hennessy の古典的 5 段 RISC（IF/ID/EX/MEM/WB）は CPI=1 を目指した素朴な設計。Pentium 4 Prescott は 31 段（2004）まで深化し分岐ミスペナルティ暴騰で行き詰まり、現代の Golden Cove/Zen 5 は **14-19 段**程度の「バランス深度」+ **スーパースケーラ幅 6-8 uop/cycle** に収束した。深いパイプラインはクロックを上げるが分岐ミス時の全段フラッシュコストが二次関数的に増加するため、**深度 × 幅 × 予測精度**の3変数最適化が必要。

### 2. ハザード三分類
- **構造ハザード**: 資源競合（単一メモリポート）→ ハードウェア複製で解決（Harvard 型 L1 分離）
- **データハザード**: RAW/WAR/WAW。**フォワーディング**（EX→EX バイパス）で ALU-ALU 依存を1サイクル化。ロード-使用は L1 レイテンシ（4-5 cycle）がボトルネック
- **制御ハザード**: 分岐まで次命令が分からない。深いパイプラインでは毎分岐でストールすると CPI ≒ 3 まで退化。**これが分岐予測器の存在理由**

### 3. 分岐予測の進化系譜
- **2-bit saturating counter**（Smith 1981）: 強弱2状態で1度の逸脱で反転させない。精度 80-90%
- **2-level adaptive**（Yeh & Patt 1991）: Global History Register (GHR) で分岐履歴をインデックス化。PAg/GAg/gshare（Ohio State 由来）。精度 95%+
- **TAGE**（Seznec & Michaud 2006）: **幾何級数的履歴長**（L₀=0, L_i=α^i・L₀ 的に 5, 15, 44, 130…）を持つ複数テーブル + 部分タグマッチ。CBP2/CBP3 優勝、現代の事実上の標準
- **Perceptron**（Jiménez & Lin, HPCA 2001）: 履歴ビットを重み付き和で線形分類。**履歴長に線形資源**で長大履歴活用。Oracle SPARC T4、AMD Zen が採用

### 4. 現代プロセッサの分岐予測器
- **Intel Golden Cove** (Alder Lake P-core, 2021): **3 レベル BTB**（L1/L2/L3 階層、ミス時+1 cycle）、BTB 第1段 128 エントリ、**2 taken branch/cycle** throughput、uop queue 内ループ展開
- **AMD Zen 4**: 1-ahead TAGE、大容量 BTB
- **AMD Zen 5** (2024): **2-ahead TAGE**（Seznec 1996 "Multiple-block ahead" を工業実装化）、1サイクルで **2 分岐先読み**。Chips and Cheese 実測で Golden Cove を僅かに上回る予測精度
- **ARM Neoverse V2** (Graviton 4): **12K エントリ BTB**、8-table TAGE。サーバワークロード対応

### 5. Out-of-Order 実行 — Tomasulo 1967 が全ての源流
IBM System/360 Model 91 浮動小数点ユニット向けに Tomasulo が設計した動的スケジューリング：
- **Reservation Station**: 各演算ユニット前のバッファ。オペランド到着まで命令保持
- **Common Data Bus (CDB)**: 完了結果を全 RS へブロードキャスト
- **Register Renaming**: 論理レジスタ（x86: 16本）→ 物理レジスタ（Zen 4: 224本 int）の写像で WAR/WAW を構造的除去
- **Reorder Buffer (ROB)**: 命令完了を元順序で commit。**投機的状態のロールバック単位**。Zen 5: 448 エントリ、Golden Cove: 512 エントリ
- "in-order issue, out-of-order execute, in-order commit" の三段構造が **精密例外**（precise exception）と投機実行の両立を可能に

### 6. Spectre/Meltdown (2018年1月) — 投機が開けた情報漏洩チャネル
- **Meltdown (CVE-2017-5754)**: ユーザ空間から投機的にカーネルメモリ読出 → キャッシュ痕跡でサイドチャネル漏洩。Intel/Apple 固有、OS 側 **KPTI (Kernel Page Table Isolation)** で緩和 — syscall 毎に **5-30% 性能低下**
- **Spectre (CVE-2017-5753/5715)**: 攻撃者が **分岐予測器を訓練**し被害プロセス内で越境ロードを投機実行させる。Intel/AMD/ARM 全 OoO CPU が影響
- **緩和策のコスト**:
  - **Retpoline** (Google): 間接分岐を return trampoline に置換、BTB を使わせない。OLTP で 4-8%
  - **IBPB** (Indirect Branch Predictor Barrier): コンテキスト切替で予測器状態クリア
  - **STIBP** (Single Thread Indirect Branch Predictors): SMT 兄弟スレッド間の予測共有禁止
- **教訓**: **マイクロアーキテクチャ状態 ≠ アーキテクチャ状態**という ISA の虚構が崩壊。キャッシュ・予測器・TLB が観測可能で副作用を残す現実

### 7. 分岐ミスのコスト
現代 CPU で **15-20 cycle**（パイプライン深度 + フェッチ/デコード再投入）。IPC=4 なら 1 ミスで **60-80 命令分の機会損失**。分岐密度 20%、予測精度 95% でも 1000 命令中 50 ミス × 15 cycle = **750 cycle ロス**。これが予測器にトランジスタ数万を割く経済的根拠。

### 8. コード側の対策
- **Branch-free code**: `cmov`、算術で条件分岐を置換（`dst = cond ? a : b` → cmov）。ソート済み配列 vs 未ソートで有名な Stack Overflow 例（2倍差）
- **PGO (Profile-Guided Optimization)**: ホットパスの分岐方向をコンパイラに教示し BB レイアウトを最適化（fall-through を多用方向に）
- **`__builtin_expect` / `[[likely]]`/`[[unlikely]]`**: コンパイラの BB 配置と命令プリフェッチに影響。予測器の訓練そのものは変えない

### 9. 設計差分 — Neoverse vs BOOM
- **ARM Neoverse V2**: 12K BTB、幅 8 デコード、ROB 320、SMT なし。**高コア数サーバ指向**
- **RISC-V BOOM**: 2-level 予測（NLP=BTB + BPD=TAGE/GShare 選択可）。**教育・研究用オープン**で Chisel 記述、学術検証プラットフォーム
- **投機幅**: Golden Cove ~512 命令窓 vs BOOM 128 程度。**ROB/PRF サイズが投機深度上限**を決める

## 気づき・洞察

- **予測と投機の非対称性**: 正解時は「見えない高速化」、誤り時は **情報漏洩 + ロールバックコスト**。ベネフィットは性能、ハザードはセキュリティというトレードオフ軸の非対称
- **TAGE の幾何級数履歴**は「どの長さの履歴が効くか事前に分からない」問題への**マルチスケール対応**。FFT のバンド分解や CNN の受容野多層化と同じ思想
- **Spectre が暴いた抽象化漏れ**: ISA は「メモリ保護 + 逐次意味論」を約束するが、μarch は投機で先走る。性能のための抽象化違反が 30 年かけて脆弱性として露呈
- **2-ahead 予測** (Zen 5) の復活は、1996 年 Seznec 論文の30年越しの工業化。**帯域律速時代のアーキテクチャ**ではフェッチ幅そのものが予測精度を要求する

## 他分野との接続

- **キャッシュ（2026-03-20）**: 分岐予測器もキャッシュ構造。BTB は命令アドレス→ターゲットアドレスの連想配列、予測テーブルは履歴→方向のキャッシュ。**全てが「過去の観測で未来を予測する」投機機構**
- **仮想メモリ（2026-03-23）**: KPTI（Meltdown 対策）は TLB フラッシュコストを発生 → PCID/ASID 活用で緩和。投機実行脆弱性と仮想メモリ機構が直接干渉
- **TCP 輻輳制御（2026-03-21）**: 分岐予測とパケット送出予測は同構造 — 「過去履歴から未来を推測し、外れたらロールバック」。CUBIC の W_max 学習は TAGE の飽和カウンタと相似
- **GC（2026-04-18）**: Write Barrier と Memory Barrier は同じ発想 — **投機/並行処理中の状態変化を捕捉**する界面機構
- **暗号（2026-04-17）**: Curve25519 の constant-time 設計は**分岐除去そのもの**。サイドチャネル対策は分岐予測・キャッシュ挙動からの情報漏洩を塞ぐ
