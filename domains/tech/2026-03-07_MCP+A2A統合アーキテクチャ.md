# MCP+A2A統合アーキテクチャ
**日付**: 2026-03-07
**分野**: テクノロジー・開発
**タグ**: #MCP #A2A #マルチエージェント #アーキテクチャ #AgentGateway #OAuth

## 学んだこと

### 1. 2軸体制の具体像——垂直と水平の交差点

MCP（垂直: エージェント→ツール/データ）とA2A（水平: エージェント↔エージェント）の2軸体制は、2026年に入り「概念」から「実装パターン」へと移行した。核心は、一つのAIエージェントが**A2Aサーバーとして外部に能力を公開しつつ、内部的にはMCPクライアントとして複数のツールに接続する**という二重構造にある。

例えば、サプライチェーンの在庫管理エージェントを考える。このエージェントは：
- **A2Aサーバー側**: Agent Cardで「在庫照会」「発注提案」といったスキルを公開し、調達エージェントや需要予測エージェントからのタスク委任を受け付ける
- **MCP クライアント側**: BigQueryデータウェアハウス、ERPシステム、物流APIなどの各MCPサーバーに接続し、実データを取得・操作する

この構造により、エージェントの「対外的な能力の広告」（A2A）と「内部的な実行能力の獲得」（MCP）が完全に分離され、それぞれ独立にスケールできる。

### 2. Gateway/Routerパターン——AgentGateway

2026年のエンタープライズ実装で最も注目されるのが**AgentGateway**パターンである。Solo.ioが開発し、Linux Foundationに寄贈されたagentgatewayプロジェクトがその代表例だ。

AgentGatewayは、エージェントとLLM、ツール、他のエージェントの間に位置する**AIネイティブなデータプレーン**で、以下を提供する：

- **プロトコル統合**: A2AとMCPの両方を単一エンドポイントで扱える。複数のバックエンドタイプ（A2Aエージェント、MCPサーバー、REST/gRPC API）へのプロキシを一箇所で管理
- **セキュリティ層**: MCP/A2Aプロトコル自体が持たない認証・認可・監査をインフラレベルで注入。全ツール呼び出しをブローカーが検査・承認してから実行
- **オブザーバビリティ**: OpenTelemetry統合により、リクエスト-レスポンスの各ペアを評価可能な単位として追跡。エージェント間通信の全経路を可視化
- **マルチテナンシー**: ステートフルなセッション管理とテナント分離

これは、マイクロサービスアーキテクチャにおけるAPI Gateway / Service Meshの役割を、エージェンティックAIの世界に翻訳したものと考えてよい。Istioがサービス間通信にセキュリティとオブザーバビリティを注入したように、AgentGatewayがエージェント間通信に同じことを行う。

### 3. Agent Registry——発見と統治の中心

A2Aプロトコルの中核概念である**Agent Card**（JSONメタデータ文書）は、エージェントの能力・スキル・連絡先情報を機械可読に記述する。エンタープライズではこのAgent Cardを管理する**Agent Registry**が統治の要となる。

agentic-community/mcp-gateway-registryプロジェクトは、MCP Gateway & Registryとして以下を実現する：

- **統一登録フロー**: MCPサーバーとA2Aエージェントの両方を統一UIで登録
- **キュレーテッドレジストリ発見**: 管理者が承認したエージェント/ツールのみが発見可能。「エージェントのApp Store」だが、エンタープライズレベルのキュレーションとガバナンス付き
- **動的ツール発見**: OAuth認証付きでセキュアにツールを発見
- **A2Aエージェントの自己登録**: エージェントが自らレジストリに登録し、他のエージェントを発見・接続。自己管理型のエージェントエコシステムを形成

ここで重要なのは、Agent Registryが単なる名簿ではなく**ポリシーエンフォースメントポイント**として機能することだ。どのエージェントがどのスキルにアクセスできるかを制御し、ガバナンスの中枢となる。

### 4. セキュリティモデル——MCP OAuth 2.1とA2Aの非対称性

MCP+A2A統合で最も複雑なのがセキュリティ層の設計である。両プロトコルは根本的に異なるセキュリティアプローチを取る：

**MCP側**: OAuth 2.1 + PKCE（Proof Key for Code Exchange）を全クライアントに**義務化**。動的クライアント登録、スコープベースのアクセス制御。2026年時点でOAuth 2.1はMCPサーバーの新しいベースラインとなった。APIキーからOAuth 2.1への移行が進行中。

**A2A側**: 特定のフレームワークを義務化せず、Agent Cardで対応する認証スキームを**宣言**する方式。Bearer token、Basic Auth、APIキー、OAuth、相互TLS（mTLS）などをサポート。柔軟だが統一性に欠ける。

この非対称性をエンタープライズでどう統合するか。現時点のベストプラクティスは：

1. **AgentGateway層で統一認証を強制**: 個々のプロトコルのセキュリティに依存せず、ゲートウェイ層でKeycloak/Entra IDなどのIdPと統合し、全通信に認証・認可を適用
2. **Cedar言語によるきめ細かなポリシー**: RBAC（ロールベース）、ABAC（属性ベース）、ReBAC（関係ベース）を組み合わせたファイングレイン認可。Amazon Verified Permissions的なアプローチ
3. **エージェントアイデンティティの管理**: Microsoft Entra Agent ID、AWS AgentCore Policyなどが2026年に登場。エージェントに対して人間のユーザーと同様のアイデンティティ管理を適用する動き

### 5. オーケストレーション層の設計パターン

実際のエンタープライズ実装では、以下のレイヤード・アーキテクチャが主流となりつつある：

```
Layer 4: ビジネスロジック層
  └─ ドメイン固有のワークフロー定義

Layer 3: オーケストレーション層（A2A）
  └─ Orchestrator Agent → 専門エージェントへのタスク委任
  └─ Agent Card による動的な能力発見・ルーティング

Layer 2: ゲートウェイ/メッシュ層
  └─ AgentGateway / Agent Mesh
  └─ セキュリティ、オブザーバビリティ、レート制限

Layer 1: ツール接続層（MCP）
  └─ 各エージェントが個別のMCPサーバーに接続
  └─ データベース、API、ファイルシステム等
```

Google Cloud Vertex AIのAgent Engineでは、ADK（Agent Development Kit）+ MCP + A2Aの組み合わせで、ルートオーケストレーションエージェントが複数のサブエージェント（各々がMCPでツール接続）を協調させるリファレンスアーキテクチャを公開している。

InfoQが紹介するMLOps向けレイヤード戦略では、A2Aが上位層（エージェント間協調）、MCPが下位層（ツール接続）として明確に分離され、新しい能力の追加がコアの通信ロジックを変更せずに行えることを強調している。

### 6. 実践事例——サプライチェーンとServiceNow

**サプライチェーン**: Logistics Viewpoints（2026年2月）が報告した実践例では、オーケストレーションエージェント（Root）が追跡エージェント、配送業者エージェント、受注エージェントの3つのサブエージェントを統括。各サブエージェントはA2AのAgent Cardで能力を広告し、MCPサーバー経由でBigQueryやERPにアクセスする。部門横断の調達シナリオでは、在庫照会（MCP→ERP）とコンプライアンス審査（A2A→法務エージェント）が自然に連携する。

**ServiceNow**: 2026年3月時点でMCPとA2Aの両方をサポートし、エージェンティックワークフロー内での統合を実現。エンタープライズITSMにおけるマルチエージェント協調の先行事例。

**市場規模**: A2Aの支持企業は2025年4月のローンチ時50社以上から、2026年2月には100社を超えた。Gartnerは2026年にほぼすべてのビジネスアプリケーションにAIアシスタントが搭載されると予測。

### 7. 残る課題と危険信号

- **Agent Meshの未成熟**: AgentGatewayは単一ゲートウェイとしては機能するが、メッシュトポロジー（エージェント間の直接P2P通信の管理）はまだ発展途上
- **発見と命名の未解決問題**: Solo.ioが指摘するように、A2Aにはエージェントの発見・命名・解決（DNS的な仕組み）が欠けている。Agent Cardの配布・更新・失効の標準的なメカニズムが不十分
- **セキュリティの脆弱性**: HALOCKの分析では、エージェント間通信の暗号化、認証トークンの漏洩防止、マルチホップ委任チェーンの検証が依然として課題
- **ガバナンスの40%問題**: 2027年末までにエージェントAIプロジェクトの40%以上がキャンセルされるというGartner予測。ガバナンスの欠如が主因。統合アーキテクチャの技術的実装よりも、組織的なガバナンス設計が成否を分ける

## 気づき・洞察

**マイクロサービスの歴史が繰り返されている。** API Gateway → Service Mesh → Sidecar Proxy という進化をたどったマイクロサービスの世界と、AgentGateway → Agent Mesh → エージェントごとのプロキシ注入というエージェンティックAIの進化が驚くほど相似形である。Kubernetesがコンテナオーケストレーションの標準になったように、MCP+A2A+AgentGatewayの組み合わせがエージェントオーケストレーションの標準スタックになる可能性が高い。

**「セキュリティの非対称性」が統合の最大のボトルネック。** MCPのOAuth 2.1義務化とA2Aの宣言型セキュリティの間のギャップは、設計上意図的なものだが、エンタープライズ実装では常にゲートウェイ層での統一化が必要になる。これは「各プロトコルが解決する問題のスコープの違い」を体現しており、プロトコル設計の哲学的な差異が実装コストに直結する好例。

**「発見」がキラー問題。** DNSがインターネットの基盤となったように、エージェントの発見メカニズムがマルチエージェントシステムの基盤となる。現在のAgent Card + Registryは、いわば「ホストファイルによる名前解決」の段階であり、スケーラブルな分散発見メカニズムの確立が次の大きなマイルストーンとなる。

## 他分野との接続

- **哲学（分散と統一）**: MCP+A2Aの2軸体制は、個の自律性（各エージェントの専門性）と全体の調和（オーケストレーション）のバランスという、東洋哲学的な「一即多、多即一」の現代的表現とも言える
- **ゴルフ（コースマネジメント）**: エージェントオーケストレーションは、ゴルフのコースマネジメントに似ている。全体戦略（A2A: どのエージェントにどのタスクを委任するか）と個別ショット（MCP: 各エージェントがどのツールを使うか）の両方を最適化する必要がある
- **解剖学（神経系のアナロジー）**: MCP+A2Aアーキテクチャは神経系に似ている。MCPは末梢神経（感覚器官・筋肉へのダイレクト接続）、A2Aは中枢神経系（脳の各領域間の協調）、AgentGatewayは血液脳関門（セキュリティゲート）
- **音楽（オーケストラ）**: オーケストレーションエージェントは指揮者、各専門エージェントは楽器セクション、Agent Cardは楽譜（能力の宣言）、MCPは各演奏者の楽器（実行能力）
