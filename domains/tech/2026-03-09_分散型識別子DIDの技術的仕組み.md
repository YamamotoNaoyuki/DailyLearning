# 分散型識別子（DID）とエージェント認証
**日付**: 2026-03-09
**分野**: テクノロジー
**タグ**: #DID #VerifiableCredentials #AgentIdentity

## 学んだこと

### W3C DID Core仕様の現在地

分散型識別子（Decentralized Identifier, DID）は、中央管理機関に依存しない新しい種類の識別子だ。W3C DID v1.0は2022年7月にW3C Recommendationとなり、2026年2月にはv1.1の Working Draftが公開された。v1.1では、DIDドキュメント内の暗号学的素材と検証メソッドの表現方法が洗練され、AIエージェントの新興IDシステムへの対応が明記された。

DIDの基本構造は `did:method:method-specific-id` の3部構成。例えば `did:web:example.com:agents:assistant-01` は、webメソッドで解決され、example.comドメインの特定のエージェントを指す。

### DIDドキュメントの構造

DIDを解決すると、JSON-LD形式のDIDドキュメントが返る。このドキュメントには以下の要素が含まれる：

1. **verificationMethod**: 公開鍵と暗号アルゴリズムの宣言。Ed25519VerificationKey2020やJsonWebKey2020が主流
2. **authentication**: DID主体の本人確認に使える検証メソッドの参照
3. **assertionMethod**: Verifiable Credentialの発行に使う鍵
4. **service**: エンドポイントの宣言（A2AのAgent Cardに相当するメタデータを含められる）
5. **controller**: DIDドキュメントを更新できる主体のDID

v1.1の重要な変更点は、verificationMethodの表現がより柔軟になったことだ。マルチキー形式（`publicKeyMultibase`）が推奨され、異なるDIDメソッド間の相互運用性が向上した。

### 主要なDIDメソッド

**did:web** — 最も実用的。既存のウェブインフラ（HTTPS + DNSドメイン）を利用。`did:web:example.com` は `https://example.com/.well-known/did.json` に解決される。A2Aの `.well-known/agent.json` と同じRFC 8615パターン。導入障壁が最も低いが、ドメイン管理者への信頼が必要（＝完全な分散ではない）。

**did:key** — 公開鍵そのものがDID。レジストリ不要、即座に生成可能。`did:key:z6Mkh...` のように鍵のマルチベース表現がそのままIDになる。エフェメラルなエージェントに最適だが、鍵のローテーションができない（使い捨て前提）。

**did:ion** — Bitcoinのレイヤー2上に構築。Sidetreeプロトコルを使用。Microsoftが主導。完全な分散と鍵ローテーションを両立するが、アンカリングに時間がかかる（数十分〜数時間）。

**did:ethr** — Ethereumアドレスから直接DIDを生成。スマートコントラクトでDIDドキュメントを管理。Web3エコシステムとの統合に強い。

### Verifiable Credentials（VC）とVerifiable Presentations（VP）

VCは「発行者が主体について何かを証明する」デジタル証明書だ。DIDで識別された発行者が、DIDで識別された主体について、暗号署名付きの主張を行う。

エージェント認証の文脈では：
- **発行者**: プラットフォーム（Anthropic, OpenAI等）がDIDを持つ
- **主体**: AIエージェントがDIDを持つ
- **VC内容**: 「このエージェントはAnthropicによってデプロイされ、以下の能力を持つ」
- **検証者**: 他のエージェントがVCの署名を検証し、能力と信頼性を確認

VPは「主体が検証者に対してVCを提示する」プロトコル。選択的開示（selective disclosure）により、VC全体ではなく必要な属性のみを証明できる。ZKP（Zero-Knowledge Proof）ベースのVPなら、「このエージェントはX能力を持つ」を証明しつつ、発行者や他の属性を秘匿できる。

### エージェントIDの実装

**Microsoft Entra Agent ID**: Azure AD（現Entra ID）をAIエージェントに拡張。マネージドID + VCでエージェントの能力と権限を証明。did:webベースでエンタープライズ内の発見と認証を統合。

**AWS AgentCore**: 2025年発表のエージェントランタイム。IAMベースのエージェント認証。DID標準との直接統合は未発表だが、VCに準じたポリシードキュメントで能力を宣言。

### Agent Discoveryとの接続

前回学んだAgent Discovery問題とDIDは以下のように接続する：

1. **A2A Agent Card + DID**: Agent CardのURL（`.well-known/agent.json`）とDIDドキュメント（`.well-known/did.json`）を同一ドメインに配置。Agent Cardが「何ができるか」、DIDドキュメントが「誰がそれを証明するか」を担当
2. **ANS + DID**: OWASPのAgent Name ServiceがPKIベースの命名を提供し、DIDが分散型の検証基盤を提供。ANSNameの能力記述をVCとして発行すれば、信頼チェーンが暗号的に検証可能に
3. **DNS-AID + did:web**: IETF DraftのDNS-AIDがSVCBレコードでエージェントの場所を教え、did:webがそのドメインのDIDドキュメントで認証情報を提供

## 気づき・洞察

**DIDはURLの再発明ではなく、URLに「信頼のレイヤー」を追加する試みだ。** HTTPのURLは「場所」を示すが「誰がそこにいるか」「信頼できるか」を示さない。DIDは識別子に暗号学的な検証可能性を組み込むことで、この欠落を埋める。Agent Discovery問題の本質は「エージェントを見つけること」ではなく「見つけたエージェントを信頼してよいか判断すること」であり、DID + VCはまさにその判断のインフラだ。

**did:webの「不完全な分散」が最も実用的である理由。** 完全な分散（did:ion, did:ethr）は理想的だが、ブロックチェーンのアンカリング時間やガスコストが障壁になる。did:webは既存のDNSとHTTPSに依存する「不完全」な分散だが、A2Aの.well-knownパターンと同居できる点で、最も早くAgent Discoveryに統合される可能性が高い。

**v1.1がCandidate Recommendationへの進行を2026年4月以降と明示したことは、DIDがまだ発展途上であることを示す。** エージェント認証の標準化は、DID v1.1の安定化を待たずに各プラットフォームが独自実装で先行する状況が続くだろう。

## 他分野との接続

- **哲学**: 九鬼周造の「偶然性の問題」でいう離接的偶然——「この特定のDIDが指すエージェントが、この特定の能力を持つ」という事実は偶然的であり、VCによる暗号学的証明は、その偶然性を必然性に変換する試みだ。西田の「絶対無」の場所が全ての存在の根拠であるように、DIDの信頼基盤は全てのエージェント関係の根拠を提供する
- **音楽**: メシアンのアイソリズム——17音と29音の素数周期——のように、DIDとVCは異なる周期（識別と証明）を独立に回しながら、重ね合わせで「信頼」という創発的属性を生む
- **ゴルフ**: プレショットルーティンにおけるコミットメント——「このクラブ、この方向で打つ」と決めたら迷わない——と同様に、VCによるエージェント認証は「このエージェントを信頼する」という判断のコミットメントを暗号的に裏付ける
