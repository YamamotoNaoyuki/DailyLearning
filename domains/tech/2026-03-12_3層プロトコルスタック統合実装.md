# MCP+A2A+WebMCP 3層プロトコルスタック統合実装パターン
**日付**: 2026-03-12
**分野**: テクノロジー

## 学んだこと

### 1. エージェントの二重役割：A2Aサーバー + MCPクライアント

2026年初頭、「1つのエージェントがA2Aサーバーとして外部に能力を公開しつつ、内部的にはMCPクライアントとして複数ツールに接続する」という二重構造が、概念から実装パターンへ移行している。

**具体的な実装構造:**
- **A2Aサーバー側**: Agent Card（`.well-known/agent-card.json`）でスキルを機械可読に公開。タスク委任をHTTP + JSON-RPC + SSE経由で受付
- **MCPクライアント側**: 内部的にJSON-RPCでBigQuery、ERP、物流API等の各MCPサーバーに接続
- **分離の利点**: 「対外的な能力の広告」（A2A）と「内部的な実行能力の獲得」（MCP）が完全に分離され、独立にスケール可能

**フレームワークの対応状況:**
- **LangGraph v0.2**（2026年1月）: A2AとMCPを「first-class protocol targets」としてサポート
- **Google ADK**: A2A・MCP両方のfirst-classサポート。Vertex AI Agent Engine上のリファレンスアーキテクチャを公開
- **Salesforce Agentforce**: MCPクライアントとしてMuleSoft/Heroku/サードパーティMCPサーバーに接続し、A2Aで他エージェントと通信

### 2. AgentGateway——Rustで構築された統合データプレーン

AgentGatewayはSolo.io発、Linux Foundation寄贈のAIネイティブなデータプレーン。MCP/A2Aの両方を単一エンドポイントで扱える。

**アーキテクチャ:**
- **モジュラー設計**: LLMルーティング、MCPハンドリング、ポリシー適用が独立動作しつつ共通基盤を共有
- **プロトコル対応**: A2A通信、MCPサーバー（tools/prompts/resources）、従来のHTTP/gRPC
- **SessionManager**: MCPをステートフルプロトコルとして扱い、セッション生成と維持を管理。ただし**ローカルインプロセスストア**のため、水平スケール時にセッションアフィニティの課題あり

**セキュリティ実装:**
- MCP Authorization仕様準拠のOAuth実装（Auth0等と統合）
- **CELベース認可**: Common Expression Languageで、JWTフィールドとMCP属性のファイングレイン制御
- プロトコルバージョンネゴシエーション・フォールバックを組み込み済み

### 3. エンタープライズ実装事例

**ServiceNow（AI Agent Fabric）:**
- Zurich Patch 4でMCPサーバー + A2Aプロトコルの両方をサポート
- OAuth必須。エンタープライズITSMにおけるマルチエージェント協調の先行事例

**Salesforce（Agentforce + MuleSoft）:**
- 3系統のMCPサーバーに接続: MuleSoft MCP Server、Heroku MCP Server、サードパーティ
- A2Aでは住宅ローン審査の事例を公開: Credit Check Agent、Document Signature Agent、Regulatory Compliance Agentが相互発見・連携
- **MuleSoft Agent Fabric**: エージェントエコシステム全体の発見・オーケストレーション・ガバナンス・可観測性を統合

**Microsoft（Agent Factory）:**
- Azure上でSAP、ServiceNow、Salesforceとのエージェンティック統合を公式サポート

### 4. 3層セキュリティモデルの非対称性と統合戦略

| レイヤー | セキュリティモデル | 特徴 |
|---------|-----------------|------|
| **MCP** | OAuth 2.1 + PKCE **義務化** | 動的クライアント登録、スコープベースアクセス制御 |
| **A2A** | **宣言型** | Bearer token/APIキー/OAuth/mTLS等を柔軟にサポート。統一性に欠ける |
| **WebMCP** | **ブラウザネイティブ同意境界** | Same-Origin Policy、CSP、HTTPS必須。3段階リスク分類 |

**統合戦略（2026年ベストプラクティス）:**
1. **ゲートウェイ層での統一認証強制**: AgentGateway + Keycloak/Entra ID
2. **CEL言語によるファイングレイン認可**: RBAC + ABAC + ReBACの組み合わせ
3. **エージェントアイデンティティ管理**: Microsoft Entra Agent ID、AWS AgentCore Policyが2026年に登場
4. **「致命的トライアド」対策**: 銀行+メール+ショッピングの複数タブ同時アクセスシナリオに対し、ドメインレベル隔離 + ハッシュ検証 + `SubmitEvent.agentInvoked`フラグで防御

### 5. 開発者ワークフロー：3プロトコル対応エージェントの構築

**推奨される段階的アプローチ:**
1. **Step 1 (MCPクライアント)**: MCP SDK（Python/TS/Java/Kotlin/C#/Swift）で外部ツール接続。月間DL 9,700万超
2. **Step 2 (A2Aサーバー)**: Agent Cardを定義しスキルを公開。LangGraph v0.2 or Google ADKでプロトコルハンドリング
3. **Step 3 (WebMCPツール公開)**: `navigator.modelContext.registerTool()`でブラウザ経由ツール公開。既存HTMLフォームへの属性追加で段階的対応も可能
4. **Step 4 (AgentGateway統合)**: 本番環境でセキュリティ・オブザーバビリティ・ガバナンスを注入

### 6. パフォーマンス

**MCP Serverベンチマーク（390万リクエストテスト）:**
- Java / Go: 事実上同一のスループット。最も安定
- Python: 高性能層の18%しか達成できず
- Rust、Quarkus、GraalVM native等を含む15実装の比較あり

**WebMCPのトークン効率:**
- 従来のスクリーンショット→Vision Model解析方式と比較して**89%改善**
- Webサイトが構造化ツールを直接公開することで、Vision Modelの推論コストが不要に

### 7. Agent Registryによる統一的発見——併存と収斂

| レジストリ | 方式 | 特徴 |
|-----------|------|------|
| A2A Agent Card | 分散型 | 受動的発見（URL既知前提） |
| MCP Registry (GitHub mcp.json) | 中央集権 | シンプル |
| Solo.io agentregistry | 3層構造 | Registry + ANS + Gateway |
| Kong MCP Registry | AAIF準拠 | Konnect上で一元管理 |
| mcp-gateway-registry | OAuth統合 | MCP+A2A統一登録 |
| AGNTCY Agent Directory | 完全分散 | IPFS DHT + Sigstore署名 |

**収斂方向:** 短期=.well-known+プラットフォーム内、中期=DNS標準化+能力ベース発見、長期=分散/中央ハイブリッド

### 8. 既知の課題とエッジケース

1. **Agent Meshの未成熟**: 単一ゲートウェイは機能するが、P2Pメッシュは発展途上
2. **SessionManagerのスケーラビリティ**: ローカルストアのためマルチインスタンスで問題
3. **「数兆のエフェメラルエージェント」問題**: 現在のDNS/IP/CAスタックは大量の短命エージェントを想定していない
4. **プロトコル間エラー伝播**: MCP層のツール呼出エラーがA2A層のタスク状態にどう反映されるか未標準化
5. **プロンプトインジェクション**: WebMCPのツール説明文経由の新攻撃面
6. **ガバナンスの40%問題**: 2027年末までにエージェントAIプロジェクトの40%以上がガバナンス欠如でキャンセル予測

## 気づき・洞察

3層統合の最大の教訓は「**セキュリティモデルの非対称性が統合の最大のボトルネット**」であること。MCP（OAuth 2.1義務化）、A2A（宣言型）、WebMCP（ブラウザネイティブ）という3つの異なるセキュリティ哲学を統一するには、AgentGatewayのようなプロキシ層が不可欠。これは「マイクロサービス時代のIstio」に相当する。

AgentGatewayのSessionManagerがローカルストアという制約は興味深い。ステートフルプロトコル（MCP）とステートレスプロトコル（A2A）の共存は、HTTPとWebSocketの共存問題の再来。歴史は繰り返す。
