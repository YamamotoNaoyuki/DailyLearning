# Mixture of Experts（MoE）と大規模LLMのスパース化

**日付**: 2026-05-12
**分野**: テクノロジー
**タグ**: #MoE #LLM #スパースモデル #DeepSeek #推論効率 #スケーリング則

## 学んだこと

### なぜ密モデル（dense）のスケーリングが行き詰まったか

Transformer の各層は「Attention（トークン間の混合）」と「FFN（トークンごとの非線形変換）」からなり、パラメータの大半は FFN にある。dense モデルでは**すべてのトークンが全 FFN パラメータを通過する**ため、パラメータ数 N に対して 1 トークンあたりの FLOPs も O(N) で増える。Chinchilla 則（学習トークン数はパラメータ数に比例させるのが計算最適）に従うと、性能を上げるには学習・推論コストが二乗的に膨らむ。

MoE の発想は **「容量（capacity）と計算量（compute）を分離する」**こと。FFN を E 個の小さな「エキスパート」に分割し、各トークンはルーター（gating network）が選んだ k 個（典型的に k=1〜8）だけを通る。総パラメータ数は E 倍に増やせるが、1 トークンあたりの計算量は k/E 倍で済む。これを **sparse activation**（条件付き計算 conditional computation）と呼ぶ。

### MoE 層の構造

1. **ルーター**: 入力ベクトル x に対し、各エキスパートとの親和度スコア s_i = x·w_i を計算し、softmax で正規化。上位 k 個を選ぶ（top-k routing）。
2. **エキスパート**: 選ばれた k 個の FFN にそれぞれ x を通す。
3. **重み付き和**: 出力 = Σ (gate_i × Expert_i(x))。gate は softmax 確率（または正規化したもの）。

ルーティングは**離散選択**なので微分不可能。GShard/Switch Transformer 系は「選ばれたエキスパートの gate 値だけが勾配を受け取る」近似で対処する（選ばれなかったエキスパートには勾配が流れない＝ルーターの学習が遅い、という問題が残る）。

### 最大の難所：負荷不均衡（routing collapse）

ルーターを素朴に学習させると、**一部のエキスパートばかりが選ばれる**悪循環に陥る（よく選ばれる→よく学習される→さらに良くなる→さらに選ばれる）。残りのエキスパートは「死んで」しまい、実効的な容量が激減する。さらに、各エキスパートには**容量上限（capacity factor）**があり、溢れたトークンは処理されず素通り（token dropping）になる——分散学習で各エキスパートを別 GPU に置くため、バッファサイズを固定する必要があるからだ。

対策の系譜：
- **補助損失（auxiliary load-balancing loss）**: 「各エキスパートへのトークン割当の分散」をペナルティ項として加える（Switch Transformer）。ただし主タスクの損失と綱引きになり、性能を犠牲にする。
- **Expert Choice routing**: トークンがエキスパートを選ぶのではなく、エキスパートが自分の容量分のトークンを選ぶ（逆向き）。負荷は構造的に均等になるが、あるトークンが 0 個のエキスパートにしか割り当てられない可能性がある。
- **補助損失フリー（auxiliary-loss-free, DeepSeek-V3, 2024末）**: 各エキスパートのスコアに**学習可能なバイアス項**を足し、混雑しているエキスパートのバイアスを下げ、空いているエキスパートのバイアスを上げる動的調整を行う。バイアスは top-k 選択にだけ使い、最終的な gate 重みには使わない。主損失を汚さずに均衡を取る——現在の事実上の標準。

### 2024–2026 の主要 MoE モデルと設計選択

- **Mixtral 8×7B / 8×22B（Mistral, 2023末–2024）**: エキスパート 8 個・top-2。「8×7B」だが共有部分があるため総パラメータは ~47B、活性は ~13B。オープンウェイト MoE の普及点。
- **DeepSeek-V3（2024-12, 671B 総 / 37B 活性）**: 「**fine-grained experts**」——エキスパートを細かく分割（256 ルーテッドエキスパート、各層 top-8）し、知識をより細分化して専門化を促す。加えて**共有エキスパート（shared expert）**1 個を常時全トークンに適用し、共通知識（文法・常識）を担わせて他のエキスパートを専門化に専念させる。補助損失フリー均衡。Multi-head Latent Attention（KV キャッシュ圧縮）と組み合わせ、14.8T トークンを **278 万 H800 GPU 時間**という破格の効率で学習。
- **Qwen3-235B-A22B / Llama 4 Scout・Maverick / DBRX / Grok**: いずれも MoE。2025–2026 のフロンティアはほぼ全部 MoE で、「dense は研究用・エッジ用、フロンティアは sparse」という棲み分けが定着した。GPT-4 系も MoE と広く推測されている。

### 推論時のコストモデル：compute は安いが memory は高い

MoE の落とし穴は **VRAM**。671B のパラメータは（37B しか使わなくても）全部メモリに載せておかねばならない（次のトークンでどのエキスパートが要るか分からない）。だから MoE は「FLOPs を節約してメモリと帯域を消費する」トレードオフ。さらに分散推論では **expert parallelism**——エキスパートを GPU 群に分散し、トークンを正しい GPU へ all-to-all 通信で送る——が必要で、ネットワークが律速になりやすい。バッチ内のトークンがエキスパート間で偏ると一部 GPU が遊ぶ（負荷不均衡が推論時にも効く）。vLLM/SGLang などのサービングスタックは expert-parallel と continuous batching を組み合わせてこれを緩和する。

## 気づき・洞察

MoE は**「専門家の委員会」より「ルーティングテーブル」と捉える**方が正確だ。各エキスパートが「数学担当」「コード担当」のように人間的に解釈できる役割を持つわけではない——可視化研究では、エキスパートはむしろ**トークンの表層的特徴（句読点、特定の語、構文位置）**で分かれることが多い。「fine-grained + shared expert」設計が効くのも、「専門家を増やす」というより「ルックアップの粒度を細かくしつつ、共通項を括り出してキャッシュする」工学だと考えると腑に落ちる。これはデータベースの**インデックス設計**——細かく分割すれば検索は速いが管理コストが増える、ホットスポットを避けるためにシャーディングを工夫する——とほぼ同型の問題で、実際 fine-grained MoE の「256 個に分けて top-8」は「シャード数を増やしてレプリカで負荷分散」の発想そのものだ。

もう一つの洞察：MoE は **Chinchilla 則を「曲げる」のではなく別の軸に逃がしている**。dense では「compute = データ × パラメータ」が一本の制約だったのを、MoE は「学習 compute は活性パラメータで決まるが、モデルの知識容量は総パラメータで決まる」と二軸に分解した。だから MoE の真の制約は「推論時のメモリ予算」に移った——Sigstore の回（鍵管理問題を ID 管理問題に変換した）と同じく、**制約を消すのではなく、より扱いやすい場所に移動させる**のがシステム設計の常套手段だと再確認できる。

## 他分野との接続

- **cs（今日のエントリ・計算可能性理論）**: ルーティングは離散選択＝本来 NP 的に難しい組合せ最適化。だが「学習で近似的に均衡させる」ことで実用化した。eBPF の verifier が「決定不能な停止性判定を、bounded loop に制限して決定可能にした」のと同型——**理論的に難しい問題を、入力を制約することで実用圏に落とす**設計判断。
- **cs（過去エントリ・MESI / 一貫性モデル）**: expert parallelism の all-to-all 通信と負荷不均衡は、分散システムの古典問題（パーティショニング、ホットシャード、コーディネーション・コスト）そのもの。MoE はニューラルネットの皮を被った分散システムだ。
- **解剖学（基底核・小脳の機能分化）**: 脳も「全領域が全タスクを処理する」のではなく、領域ごとに役割分担しつつ、視床や基底核が「どの回路を活性化するか」を選択（ゲーティング）する。MoE のルーターは基底核の「行動選択」機能の人工版に近い。
- **music（ブラームス4番・オーケストレーション）**: オーケストラも「全楽器が常時鳴る」のではなく、指揮者（ルーター）が場面ごとに楽器群（エキスパート）を選んで前面に出す。総奏（tutti）は shared expert に相当する。
- **other（ゲーム理論）**: 補助損失フリー均衡は、各エキスパートの「混雑度に応じてバイアスを動的調整」する——これは**混雑料金（congestion pricing）**＝Pigou 税の発想で、ナッシュ均衡を社会的最適に近づけるメカニズムデザインそのもの。

## 次に深掘りしたいこと

- **MoE の理論**: なぜ fine-grained が効くのか、エキスパート数 E と活性数 k の最適比はスケーリング則でどう決まるか（"MoE scaling laws" の研究）
- **モダリティ拡張**: vision MoE、Soft MoE（離散ルーティングを連続化）、MoE と Mixture-of-Depths（層をスキップする）の組み合わせ
- **量子化と MoE**: 「使われないエキスパートは積極的に量子化/オフロードする」per-expert quantization、SSD オフロード推論
- **upcycling**: dense モデルを後から MoE に「分割」して継続学習する手法（Mixtral がこれと噂される）の実態
