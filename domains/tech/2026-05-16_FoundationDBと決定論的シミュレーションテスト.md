# FoundationDB と決定論的シミュレーションテスト — 分散DBの信頼性工学
**日付**: 2026-05-16
**分野**: tech
**タグ**: #FoundationDB #DST #分散システム #信頼性工学 #fault-injection

## 学んだこと

FoundationDB（FDB）は Apple が買収後にオープンソース化した分散トランザクショナル KV ストア。Snowflake、iCloud、CockroachDB の元プロトタイプなど、産業界で「最も信頼できる分散 DB」とまで評される。その品質を支えているのが **Deterministic Simulation Testing (DST)** という独特のテスト方法論で、近年 Antithesis、TigerBeetle、WarpStream、Polar Signals などが模倣・発展させている。

**DST の核となる発想は「I/O・時計・乱数・ネットワークなど非決定性の源を全て抽象化し、シミュレータの単一プロセス内で疑似時間を進めることで、分散システム全体を決定論的に再現する」**。FDB の場合：

- **`Flow` 言語** — 独自の C++ 拡張で、`ACTOR`/`Future`/`Promise` というアクター系プリミティブをコルーチンに変換する。実装はそのまま `IAsyncFile`、`INetwork`、`IRandom` という抽象境界を通して I/O を呼ぶ。
- **実装の双対化** — 同じバイナリ内に「実 I/O サブシステム」と「シミュレーション I/O サブシステム」を持ち、起動オプションで切り替える。シミュレーション側は単一プロセス内に数十〜数百のプロセス役を立ち上げ、メモリ内のメッセージキューだけでネットワークを実装する。
- **離散イベント駆動** — シミュレータは「次に到達する将来時刻」へジャンプするタイムウィール。実時間は流れない。これにより 1 秒のシミュレーションが数 ms で完了し、数年分の運用を一晩で焼ける。
- **乱数のシード化** — 全ての乱択（タスクスケジューリング、メッセージ並べ替え、障害注入位置）は単一のシードから派生。バグを発見したテストは、シードさえ覚えておけば 1 ビットも違わず再現できる。

**Buggify** という名物機能：本番コードの随所に `if (BUGGIFY) { do_bad_thing(); }` というマクロが埋め込まれており、シミュレーションモード時のみランダムに `true` を返す。これによって「本来 100 万回に 1 度しか起きない奇妙な状態」（タイムアウト直前、バッファあと 1 バイト、リーダー選挙の真っ最中、など）を高頻度で叩ける。本番では完全に死に、コードパスを汚さない。

障害注入の粒度も極めて細かい：プロセスクラッシュ、マシン再起動、ラック障害、データセンター丸ごとの分断、ディスクの bit flip、ファイル切り詰め、メッセージの重複・並べ替え・遅延・喪失、クロックスキュー… これらを Poisson 過程で同時多発させる「Nightmare 多発障害シナリオ」を実装している。

**実績**：FDB チームは「ディスクに 1 バイトも書き込む前」に 18 ヶ月かけてシミュレータを作り込み、累計 **1 兆 CPU 時間** のシミュレーションを回している。Kyle Kingsbury（Jepsen 作者）は「FDB は Jepsen より厳しいテストを既に通っているから自分が試す意味がない」と書いた——分散 DB 界では伝説のコメント。

なぜ業界の主流にならなかったか：**実装に Flow が必要で、後付けが非常に難しい**。後発の TigerBeetle は最初から Zig で書き直し、WarpStream は Go で似た構造を作った。Antithesis は逆のアプローチで、ハイパーバイザレベルで（コード書き換えなしに）決定論を実現する独自仮想化プラットフォームとして商業化した。

## 気づき・洞察

DST の本質は「**現実を抽象化の壁で囲い、その内側を計算上の宇宙にする**」こと。普通のテストは「コードを実物の I/O に投げて何が起きるか観察する」。DST は逆で「I/O こそが仮想化された側で、コードは『現実』として走る」。**因果関係の向きが逆転している**。これが一度腑に落ちると、ユニットテストとも統合テストともカオスエンジニアリングとも違う、第四の象限にあるテスト技法だと分かる。

決定論を捨てるとなぜ駄目か：分散システムのバグは「並行性 × 障害 × タイミング」の組み合わせ爆発で潜む。再現できないバグは存在しないも同然——直したつもりがまた出る、直し方が分からない、リグレッション検出ができない。DST はこのループを断ち切る。**「再現性」こそが品質の上限を決める**という洞察は、フィードバック制御理論における「観測可能性」が制御可能性の上限を決めるのと同じ構造をしている。

なぜ Buggify が天才的か：通常のフォールトインジェクションは「外部から落とす」。Buggify は **コード内部の状態遷移エッジに障害をリンク**しておく。例えば「コミットログ書き込み直後、ACK 送信直前」という極めて狭い窓を狙える。これはバグの「自然分布」を歪めて、レアな経路の発生確率を意図的に持ち上げる手法で、**統計的にエッジケースをサンプリングする**操作と捉えると美しい。

一方、限界も鮮明。**シミュレータが抽象化していない部分はテストできない**——FDB は OS のスレッドスケジューラや CPU の memory reordering を抽象化していないので、それ起因のバグは捕まらない。実 I/O との壁の内側だけが「証明されている世界」。だから FDB も本番では従来のカナリアリリースや段階展開を併用する。**形式手法（[[TLA+と形式手法]] のような）が「全状態空間を網羅」を目指すのに対し、DST は「巨大だが有限の状態空間を圧倒的高速で消化する」アプローチ**で、両者は補完関係。

## 他分野との接続

- **[[cs/形式手法とTLA+]]**: TLA+ は仕様レベルで状態空間を網羅探索する。DST は実装レベルで（疑似乱数で）サンプリング探索する。同じ「分散システムの正しさ」問題に対する **演繹的アプローチと実験的アプローチ** の二刀流。AWS は両方を使っており、TLA+ で設計し DST 風シミュレーションで実装を検証している。
- **[[anatomy/小脳と運動学習の内部モデル]]**: 小脳は身体運動の **内部モデル（forward model）** を持ち、運動指令と感覚入力の予測誤差で学習する。DST のシミュレータも実システムの「内部モデル」で、実 I/O との予測誤差（バグ）を最小化する方向で実装が修正されていく。**身体と分散システムは「予測モデルでの内部試走」を共有する**。
- **[[piano/メンタルプラクティスと運動イメージ]]**: ピアニストが暗譜後にメンタルプラクティスで脳内演奏するのも、自分の身体・楽器・ホールを「シミュレータ」として走らせる行為。実演奏という非決定的本番の前に、決定論的な脳内シミュレーションで失敗パターンを潰す。
- **[[other/ゲーム理論_ナッシュ均衡]]**: マルチエージェントシステム（FDB のプロセス群もそう）の振る舞いは戦略的相互作用に近い。Buggify は「全プレイヤーが最悪の手を打ったらどうなるか」の網羅探索とも読める。

## 次に深掘りしたいこと

- Antithesis の実装：彼らはハイパーバイザレベルで決定論を実現するというが、TLB shootdown や RDRAND など本物のハードウェア非決定性をどう抑え込んでいるか
- TigerBeetle のテストハーネス（vortex test runner）が FDB の Flow なしで Zig でどこまで再現できているか
- DST と Lineage-driven Fault Injection (LDFI) との関係：後者は障害を「過去ログの因果関係」で計画する。両者を組み合わせれば、ランダムでなく「論理的に意味のある障害」だけを注入できるか
