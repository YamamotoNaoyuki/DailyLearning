# Nix と再現可能ビルド — 関数型パッケージ管理の設計哲学
**日付**: 2026-04-18
**分野**: tech
**タグ**: #Nix #NixOS #ReproducibleBuilds #FunctionalPackageManagement #Flakes #DevOps

## 学んだこと

### 関数型パッケージ管理という発想の転換
従来のパッケージマネージャ（apt, yum, brew 等）は **グローバルな副作用** を前提とする——`/usr/bin` に上書きインストールし、依存関係が暗黙に解決され、同じコマンドが日をまたいで違う結果を返す。Nix はこのモデルを根本から破る。すべてのパッケージは `/nix/store/<hash>-<name>-<version>/` という**コンテンツアドレスドなパス**にビルドされ、グローバル状態は一切変更しない。パッケージのビルド入力（ソース、依存関係、ビルド命令、環境変数、Nixのバージョン）をすべてハッシュ化した結果がパス名そのものになる——これが「関数型」の意味で、入力が同じなら出力パスも同じ。

具体例: `hello-2.12.1` を glibc 2.38 でビルドすると `/nix/store/xvjp...hello-2.12.1` に配置される。glibc を 2.39 に上げると、ハッシュが変わるので **全く別のパス** に再ビルドされる。古いバージョンはそのまま残り、両方を並行使用可能。これが「ロールバックが自明」「複数バージョン共存」の技術的基盤。

### Nix Expression Language と派生（Derivation）
ビルド命令は Nix Expression Language（純粋関数型・遅延評価）で記述する。`stdenv.mkDerivation { name = "hello"; src = ...; buildInputs = [...]; }` の形で「何をどうビルドするか」を宣言する。この式を評価すると `.drv` ファイル（派生）が生成される——派生は「ビルドの完全な記述」で、そこからストアパスが一意に決定される。

### Sandboxed Build と Purity
Nix のビルドは **sandboxed** で実行される。ネットワーク遮断、`/nix/store` 以外のファイル不可視、時刻固定、HOME ディレクトリ隔離。これによりビルドプロセスが「宣言された入力だけ」に依存することが強制される。`configure; make; make install` 系のソフトウェアが「たまたまシステムに入っているライブラリを暗黙に拾う」事故を構造的に防ぐ。

### 再現可能ビルドの実態
NixOS エコシステムは 2026年時点で **80,000 パッケージ以上で 90% 超の再現性**を達成している。残り 10% の非再現性の原因は：タイムスタンプ埋め込み、並列ビルドの非決定性、乱数種、ビルドパス埋め込み、コンパイラの非決定的最適化など。reproducible.nixos.org はこれを継続監視し、`stripNondeterminism` 等のツールで除去する。

### Flakes — モダンな Nix 開発ワークフロー
Flakes (2021年導入、2023年以降デフォルト化進行) は「プロジェクトの完全記述」を `flake.nix` + `flake.lock` で提供する。`flake.lock` は依存のハッシュを固定し、`go.sum` や `Cargo.lock` と同じ役割を果たす。`nix develop` で完全に隔離された開発シェル、`nix build` で決定的ビルド、`nix run github:user/repo` で任意のリポジトリのビルドを即座に実行。

### 分散検証への進化 — LILA プロジェクト
FOSDEM 2026 で発表された LILA（Lila: Decentralized Build Reproducibility Monitoring）は arXiv 2601.20662 に投稿されている。従来の r13y.com のような中央集権的な再現性検証ではなく、**複数の独立したビルダーがそれぞれ独立にビルドし、ハッシュが一致するかを分散で検証**する。ビザンチン対耐性を持つ再現性保証への進化。

## 気づき・洞察

- **「純粋関数」という抽象が OS 管理に適用できる衝撃**: Haskell の純粋関数（同じ入力なら同じ出力、副作用なし）という数学的概念を、パッケージ管理という泥臭い実世界の問題に当てはめた発想が革命的。入力=(ソース+依存+環境)、出力=(ストアパス) という関数として捉えると、副作用まみれの問題が数学的に扱える。
- **サプライチェーン攻撃への構造的防御**: SolarWinds や xz-utils 型の攻撃は「ビルド環境の汚染」を突く。Nix の sandboxed build と入力の完全ハッシュ化は、ビルド過程に不透明な要素を許さない。2026年時点で SBOM（Software Bill of Materials）の自動生成との統合が進む。
- **「間違いが上書きされない」という信頼性**: Ubuntu の `apt upgrade` で壊れたら戻せない、という日常的ストレスを、Nix は generation 管理で根本解決する。`nix-env --rollback` で瞬時に前の状態に戻る——これは「状態 vs 履歴」のイベントソーシング哲学と同型。
- **学習曲線の壁**: Nix の強力さと引き換えに、Nix Expression Language の独特な文法、エラーメッセージの難解さ、ドキュメントの断片性が普及を妨げてきた。2025-2026 で Determinate Systems などが UX 改善に本腰。

## 他分野との接続

- **イベントソーシング（tech 2026-03-23）との同型性**: Nix の generation 管理は「状態の履歴を全て保持し、現在状態は履歴から再構成」という構造が同型。`nix-env --rollback` は projection の再構築に相当。
- **CRDT（tech 2026-03-21）との対比**: CRDT は「複数ノードでの状態収束」を数学的保証で解く。Nix は「同じビルド入力からの一意出力」を数学的保証で解く。どちらも **制約を受け入れて冪等性を得る** 戦略。
- **Passkeys（tech 2026-04-17）の哲学的共通点**: 「秘密の記憶」「システム状態の記憶」という人間/マシンの不確実な記憶を放棄し、暗号的に検証可能な決定的構造に置き換える思想が通底する。
- **関孝和の和算（other 2026-04-17）との遠い共鳴**: 和算は数学を実用問題（測量、暦）に厳密な道具として適用した。Nix は関数型プログラミングをシステム管理に厳密に適用した。「抽象を実用に持ち込む」系譜。
- **イベントソーシングと Nix の「補償イベント」**: Nix では「削除」は新 generation への移行であり、古い generation は残る。イベントソーシングで「削除イベント」が補償イベントとして記録されるのと同じ——**どちらも append-only な真実の源泉を作る**。
