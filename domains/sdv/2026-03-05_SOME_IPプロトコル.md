# SOME/IPプロトコル——サービス指向車載通信の基盤
**日付**: 2026-03-05
**分野**: SDV
**タグ**: #SOME/IP #AUTOSAR #サービス指向 #Automotive_Ethernet #vsomeip

## 学んだこと

### SOME/IPの位置づけ

SOME/IP（Scalable service-Oriented MiddlewarE over IP）は、2011年にBMWが開発し、後にAUTOSARの標準プロトコルとなった車載ミドルウェアだ。CAN busが「信号ベース」の通信であるのに対し、SOME/IPは「サービスベース」の通信を実現する。

前回学んだCAN busは、センサー値のような小さなデータ（最大8バイト/64バイト）を定期的にブロードキャストする通信に適している。一方、SOME/IPは、カメラ映像、ソフトウェアアップデート、複雑な診断データなど、大きなペイロードを「必要な時に」やり取りするための通信に設計されている。

### CAN vs SOME/IP——パラダイムの転換

| 特性 | CAN bus | SOME/IP |
|------|---------|---------|
| 通信モデル | 信号ベース（PDU） | サービスベース（RPC/イベント） |
| トランスポート | CAN物理層 | TCP/UDP over Ethernet |
| データサイズ | 8B（CAN）/64B（CAN FD） | 理論上制限なし |
| 帯域幅 | 1Mbps（CAN）/8Mbps（CAN FD） | 100Mbps～10Gbps |
| 発見 | 固定的（DBC定義） | 動的（Service Discovery） |
| スケーラビリティ | ECU数に依存して悪化 | IPネットワークで柔軟にスケール |

### メッセージフォーマット

SOME/IPメッセージは固定16バイトのヘッダー + 可変長ペイロードで構成：

```
[Service ID: 16bit][Method ID: 16bit]
[Length: 32bit]
[Client ID: 16bit][Session ID: 16bit]
[Protocol Version: 8bit][Interface Version: 8bit][Message Type: 8bit][Return Code: 8bit]
[Payload: variable]
```

- **Service ID + Method ID**: どのサービスのどのメソッドを呼ぶか
- **Client ID + Session ID**: リクエスト-レスポンスの対応付け
- **Message Type**: REQUEST(0x00), REQUEST_NO_RETURN(0x01), NOTIFICATION(0x02), RESPONSE(0x80), ERROR(0x81)

### 4つの通信パターン

**1. Request/Response（RPC）**
クライアントがリクエストを送信、サーバーがレスポンスを返す。例: 診断ツールがECUにDTCを要求

**2. Fire & Forget**
レスポンスを期待しないリクエスト。例: コマンド送信

**3. Event Notification**
サーバーが状態変化をクライアントに通知。Publish/Subscribe型。例: 車速が閾値を超えたら通知

**4. Field（Getter/Setter/Notifier）**
プロパティへのアクセス。Get（値の取得）、Set（値の設定）、Notify（変更時の通知）

### Service Discovery（SD）——動的サービス発見

SOME/IP-SDは、サービスの存在とアクセス方法をネットワーク上で動的に発見するプロトコル。

- **Offer Service**: サーバーが「このサービスを提供しています」をマルチキャストで告知
- **Find Service**: クライアントが「このサービスを探しています」をマルチキャスト
- **Subscribe Eventgroup**: クライアントがイベントグループへの購読を要求
- **Subscribe Eventgroup Ack**: サーバーが購読を承認

SDはmDNS/DNS-SDに似た仕組みだが、車載環境に最適化されている。SDにより、ECUの追加・交換が「プラグ&プレイ」的に可能になる。

### vsomeip——オープンソース実装

COVESA（Connected Vehicle Systems Alliance）がGitHubで公開している参照実装。C++で記述。

- Eclipse Kuksaとの統合: `kuksa-someip-provider` がvsomeipを介してSOME/IPデータをKuksa Databrokerに転送
- SOME/IPの生データ → VSS（Vehicle Signal Specification）への変換ブリッジとして機能
- Adaptive AUTOSAR環境でのara::com通信の内部トランスポートとしてvsomeipが使用されるケースが多い

### Eclipse SDVエコシステムとの接続

前回学んだEclipse Kuksaは、SOME/IPから来るデータをVSSに変換して管理する。通信スタックとしてのSOME/IP → データモデルとしてのVSS → データブローカーとしてのKuksa——この3層がEclipse SDVの車内通信アーキテクチャの骨格だ。

Eclipse uProtocolは、このSOME/IPを含む複数のトランスポート（Zenoh, MQTT, SOME/IP）を統一APIで抽象化するレイヤーだ。

## 気づき・洞察

CAN → SOME/IPの転換は、まさにテクノロジー分野で学んだパラダイムシフトと同型だ。REST APIが「メソッド呼び出し」を抽象化したように、SOME/IPは車載通信を「サービス呼び出し」に抽象化した。信号レベルの低レベル通信から、サービスレベルの高レベル通信へ——これはCからHTTPへの進化に等しい。

Service Discoveryの仕組みは、A2AプロトコルのAgent Cardと驚くほど類似している。「自分が何を提供できるか」を宣言し（Offer/Agent Card）、「必要なサービスを探す」（Find/Discovery）。車載通信とAIエージェント通信が同じ設計パターンに収斂しているのは偶然ではない——分散システムの本質的なニーズが同じだからだ。

## 他分野との接続

- **テクノロジー・A2A**: SOME/IP-SDのOffer/Find ≒ A2AのAgent Card + Discovery。Subscribe Eventgroup ≒ A2AのTask Subscription。プロトコル設計のパターンが領域を超えて共通
- **解剖学**: SOME/IPの4つの通信パターンは神経系の通信に類似。RPC＝体性神経系（意図的な指令と応答）、Event Notification＝自律神経系（閾値超過で自動通知）
- **音楽**: オーケストラの指揮者→奏者通信はRequest/Response、室内楽のアイコンタクトはEvent Notification。バルトークの弦楽四重奏はまさにSOME/IP的な4エージェントのサービス指向通信
- **パン**: サワードウの微生物間コミュニケーション——乳酸菌が酵母に「環境pH変化」をイベント通知する。Publish/Subscribeの生物学的実装
