# ゾーンアーキテクチャへの移行
**日付**: 2026-03-07
**分野**: SDV（Software Defined Vehicle）
**タグ**: #ゾーンアーキテクチャ #ドメインECU #ゾーンECU #HPC #Eclipse_SDV

## 学んだこと

### ドメインアーキテクチャの限界——なぜ移行が不可避になったか

ドメインアーキテクチャは、車両の機能をパワートレイン、ボディ、シャシー、インフォテインメント、ADASといったドメインごとに分割し、各ドメインに専用のECUを配置する構成だった。この設計は機能の独立性を保ちやすく、Tier1サプライヤーがドメイン単位で責任を持てるという組織的なメリットもあった。しかし、車両の高機能化に伴いECU数は爆発的に増加し、現代の高級車では100を超えるECUが搭載されている。問題は単にECUの数ではない。ドメインをまたがる機能——たとえばADASがパワートレインとシャシーの両方を制御する場面——が増えるにつれ、ドメイン間の通信経路は複雑化し、CAN/LINゲートウェイの多段中継による遅延が無視できなくなった。

さらに深刻なのがワイヤーハーネスの問題だ。各ECUからセンサー・アクチュエータへの配線は機能ドメインの論理に従って車両全体に引き回される。物理的に近い位置にあるセンサーが、論理的に異なるドメインに属しているだけで別々のECUまで長距離配線されるという非効率が常態化していた。現代の高級車のワイヤーハーネス総延長は約3kmに達し、車体重量の中で銅線は無視できないコスト・重量要因となっている。EVにおいては重量が航続距離に直結するため、この問題はさらに深刻だ。

### ゾーンアーキテクチャの基本構造——物理配置に基づくECU再編

ゾーンアーキテクチャは、この「機能ドメインによる論理分割」を「車両の物理的な位置による空間分割」に置き換える。車両をフロント左、フロント右、リア左、リア右（あるいはフロント、左、右、リアなど）のゾーンに分割し、各ゾーンにゾーンECU（Zone Controller / Zone Gateway）を配置する。ゾーンECUはそのゾーン内のすべてのセンサー・アクチュエータを管理し、上位のHPC（High-Performance Computer）と高速Ethernetバックボーンで接続される。

この構成のポイントは3つある。

**第一に、配線の物理的最適化。** センサーやアクチュエータは、機能ドメインに関係なく、物理的に最も近いゾーンECUに接続される。ドアロック、ウィンドウモーター、スピーカー、サイドミラーが同じゾーンECUに集約される。これにより配線長が劇的に短縮される。Teslaは Model S の約3kmのワイヤーハーネスを Model 3 で約1.5kmに半減させ、将来的には100mまで削減する構想を持つ。VWのSSP（Scalable Systems Platform）でもワイヤーハーネス40%削減、ECU数50%以上削減を見込んでいる。

**第二に、HPC（High-Performance Computer）への機能集約。** ゾーンアーキテクチャにおいて、ゾーンECUは本質的にはI/Oゲートウェイであり、「手足」に過ぎない。「脳」はHPCが担う。従来、各ドメインECUに分散していたアプリケーションロジックがHPCに集約されることで、ドメインをまたがる機能（統合ボディ制御、ADAS連携など）がシンプルなソフトウェア実装になる。HPCは車両に1〜3台搭載され、それぞれが数百TOPS級のSoCを持ち、LinuxやAUTOSAR Adaptiveを走らせる。

**第三に、ソフトウェアとハードウェアの分離。** ゾーンECUが標準化されたI/Oインターフェースを提供し、HPCが汎用計算プラットフォームとなることで、ソフトウェアがハードウェアの物理配置から独立する。これがSDVの本質——車両のライフサイクル中にソフトウェアを更新・追加できるアーキテクチャ——を実現する基盤となる。

### ゾーンECUの内部構造と技術要件

ゾーンECUは「単純なI/Oハブ」と説明されることがあるが、実際の技術要件はかなり高い。

**Mixed-Criticality（混合安全度）の処理。** 一つのゾーンECUが、ASIL-D相当のブレーキ制御信号と、QM（品質管理レベル）のアンビエントライティングを同時に扱う可能性がある。これにはハイパーバイザー（例：Elektrobit EB corbos Hypervisor、ASIL-B認証済みのType-1マイクロカーネルベース）による空間的・時間的分離が必要になる。AUTOSAR ClassicのリアルタイムタスクとAUTOSAR Adaptiveのサービスを一つのSoC上で共存させるアーキテクチャが現実的な解となっている。

**通信ゲートウェイ機能。** ゾーンECU内部では、レガシーのCAN/LIN接続（末端センサー・アクチュエータ向け）とEthernetバックボーン（HPC向け）を変換する。CAN-to-Ethernetゲートウェイとしての役割は、既存の車両ネットワークからの移行期において極めて重要だ。将来的にはゾーン内のラストマイルも10BASE-T1Sに置き換わる可能性がある。

**10BASE-T1Sの役割。** 10BASE-T1SはIEEE 802.3-2022に含まれるEthernet PHY規格で、マルチドロップ（バス型）トポロジーをサポートする。これはCAN busのバス型配線のメリット（1本のケーブルに複数ノード接続）をEthernetで実現するものだ。PLCA（Physical Layer Collision Avoidance）によりCSMA/CDの衝突問題を回避し、決定的な遅延特性を持つ。ゾーンECUとゾーン内センサー群の接続において、CAN/LINの代替として期待されている。Microchipは2025年にLAN866xファミリの10BASE-T1Sエンドポイントデバイスを発表し、現代自動車と共同で車載ネットワークへの適用を推進している。

**Ethernetバックボーン。** ゾーンECU間およびゾーンECUとHPC間は、1000BASE-T1（1Gbps）ないし将来的にはMulti-Gig Ethernet（2.5G/5G/10G-BASE-T1）で接続される。TSN（Time-Sensitive Networking）による確定的遅延保証と組み合わせることで、リアルタイム制御信号とベストエフォートのインフォテインメントデータを同一ネットワーク上で共存させる。

### OEM各社の動向——理論から量産へ

**Tesla：先駆者の5年以上のリード。** Teslaは業界で最初にゾーンアーキテクチャに近い構成を量産車に導入した。Model 3（2017年）では、左ボディコントローラー、右ボディコントローラー、フロントトランクコントローラーといったゾーン型のECU配置を採用し、Central Computing Module（CCM）に自動運転・インフォテインメント機能を集約した。S&P Global Mobilityの分析によれば、TeslaのE/Eアーキテクチャは他OEMに対して5年以上の先行優位を持つ。特筆すべきは、Teslaがモジュラーワイヤリングシステムの特許を取得していることだ。従来のワイヤーハーネスの概念自体を廃し、ゾーンハブが周辺デバイスを直接制御するアーキテクチャにより、将来的にはCAN bus自体を不要にする構想を持つ。

**Volkswagen Group：SSP（Scalable Systems Platform）。** VWグループは2026年にSSPプラットフォームの冬季テストを開始し、2027年のID.Every1からシリーズ生産に入る。SSPは完全なゾーンアーキテクチャを採用し、グループ全体（VW、Audi、Porsche、Skoda等）で最大3,000万台の基盤となる。800V技術、LFP/LMO/NMCの複数バッテリケミストリー対応と併せ、E/E統合による大幅なコスト削減を狙う。VWはRivianとの提携を通じてゾーン型SDVアーキテクチャの開発を加速させており、RivianのSoCとソフトウェアプラットフォームの知見を取り込んでいる。

**その他のOEM。** BMW、Mercedes-Benzも次世代プラットフォームでゾーンアーキテクチャへの移行を表明しているが、既存車種のヴァリアント対応やサプライチェーン再編の規模から、完全移行は段階的に進む。多くのOEMは「クロスオーバーアーキテクチャ」——ドメイン構成を一部維持しつつゾーンの配線最適化を取り入れるハイブリッド——を中間ステップとして採用している。

### Eclipse SDVプロジェクトとの関連——ゾーンアーキテクチャのソフトウェア基盤

ゾーンアーキテクチャにおいて、Eclipse SDVプロジェクト群は以下のように位置づけられる。

**Eclipse Ankaios——ゾーン横断のワークロード管理。** Ankaiosのサーバー/エージェントモデルは、まさにゾーンアーキテクチャのために設計されている。HPCにAnkaiosサーバーが稼働し、各ゾーンECU（あるいはHPC内の各VMパーティション）にAnkaiosエージェントが配置される。コンテナ化されたワークロードがどのノードで実行されるかを宣言的に管理し、ノード障害時の再配置も可能にする。gRPCベースのサーバー-エージェント間通信は、Ethernetバックボーン上で自然に動作する。v1.0.0 GAでこのマルチノード管理が安定し、量産レベルでの採用が現実味を帯びている。

**Eclipse Kuksa + SOME/IP——ゾーンをまたぐデータ統合。** Kuksa Databrokerはゾーンの物理的境界を意識しない統一的なデータアクセスを提供する。VSSの名前空間（`Vehicle.Chassis.Axle.Row1.Wheel.Left.Brake.*`）は物理的な位置情報を含んでおり、ゾーンECUが収集したローカルデータがKuksaを通じてHPC上のアプリケーションから透過的にアクセスできる。SOME/IPのService Discoveryはゾーン構成の動的変更にも対応する。

**Eclipse Leda——ゾーンECU向けOS基盤の可能性。** Leda（Yocto/Pokeyベース）は当初HPC向けのLinuxディストリビューションとして開発されたが、ゾーンECUが高性能化するにつれ、ゾーン側でもLinuxベースのOSを走らせるシナリオが出てくる。特にハイパーバイザー上でAUTOSAR ClassicとLinuxを共存させる構成において、Linux側のベースイメージとしてLedaが活用できる。

**Eclipse uProtocol——異種通信の統合。** ゾーンアーキテクチャでは、ゾーン内のCAN/LIN、ゾーン間のEthernet/SOME/IP、クラウドとのMQTT/HTTPSなど、複数の通信プロトコルが混在する。uProtocolのトランスポート非依存レイヤーは、この異種環境を統一的なAPIで抽象化する役割を果たす。

### AUTOSAR Adaptiveとの関係

ゾーンアーキテクチャにおけるAUTOSAR Adaptiveの役割は、主にHPC側のアプリケーションプラットフォームとして位置づけられる。ara::comによるサービス指向通信、ara::execによる実行管理、ara::diagによる診断が、HPCで動くADASや統合ボディ制御のフレームワークとなる。一方、ゾーンECU側では依然としてAUTOSAR Classicが支配的であり、MCUベースのリアルタイム制御には引き続きClassicのRTE（Runtime Environment）が使われる。

ここで注目すべきは、AUTOSAR AdaptiveとEclipse SDVの関係性の変化だ。初期にはAUTOSAR Adaptiveが唯一のHPC向けプラットフォームと見なされていたが、Eclipse SDVプロジェクト群（Ankaios、Kuksa、Velocitas）がLinuxベースの代替スタックを形成しつつある。量産車では両者が共存する——AUTOSAR Adaptiveが安全性認証が必要な制御系を担い、Eclipse SDVがクラウド連携やOTA更新、サービスアプリケーション層を担うハイブリッド構成が主流になると予想される。

## 気づき・洞察

- ゾーンアーキテクチャの本質は「物理と論理の分離」にある。従来は物理配線が論理的なドメイン構造に従属していたが、ゾーンでは物理配線は空間的近接性に従い、論理的な機能統合はHPCのソフトウェアが担う。これはまさにクラウドコンピューティングでサーバーの物理配置とサービスの論理配置を分離したのと同じパラダイムシフトだ
- Teslaの「ワイヤーハーネス100m構想」は、末端デバイスにインテリジェンスを持たせる（Smart Actuator ECU）ことで初めて実現する。これはIoTのエッジコンピューティングと同じ発想——中央集約と末端分散のバランス点が最適解
- 10BASE-T1Sは「CANの精神をEthernetで実現する」技術であり、ゾーン内のラストマイル通信において、CANからの移行パスとして極めて合理的。バス型トポロジー + PLCAによる衝突回避は、CANエンジニアにとって馴染みやすい
- Ankaiosのサーバー/エージェントモデルは、ゾーンアーキテクチャの「HPC + 複数ゾーンECU」トポロジーに対してイソモーフィック（同型）。アーキテクチャ的必然性がある

## 他分野との接続

- **テクノロジー（tech）**: Kubernetes/コンテナオーケストレーションの車載版がAnkaios。マイクロサービスアーキテクチャの車載適用がゾーンアーキテクチャ + HPC
- **解剖学（anatomy）**: 人体の神経系は脊髄（ゾーンゲートウェイ）が末梢神経（センサー/アクチュエータ）を集約し、脳（HPC）に情報を送る構造。ゾーンアーキテクチャは人体の神経系アーキテクチャに驚くほど似ている
- **哲学（philosophy）**: 「形式は機能に従う」——ドメインアーキテクチャは機能の論理に従い、ゾーンアーキテクチャは物理の法則に従う。どちらが「正しい」かではなく、システムの複雑度が閾値を超えたときにパラダイムが転換する
- **音楽（music）**: オーケストラのセクション配置（弦・管・打楽器のドメイン型）から、空間的音響配置（サラウンドのゾーン型）への移行に類似。物理的配置が体験を変える
