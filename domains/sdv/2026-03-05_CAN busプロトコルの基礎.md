# CAN busプロトコルの基礎——車のデジタル神経系を理解する
**日付**: 2026-03-05
**分野**: SDV（Software Defined Vehicle）
**タグ**: #CANbus #Bosch #差動信号 #アービトレーション #CANFD #OBD-II

## 学んだこと

### 誕生と歴史

1983年にドイツのRobert Bosch GmbHが開発を開始。1986年にSAEカンファレンスで正式発表。1987年にIntelとPhilipsが最初のコントローラチップを生産。今日では事実上全ての自動車に搭載され、車両内の**デジタル神経系**として機能している。

### 物理層——差動信号の美しさ

CAN busは2本のツイストペアケーブル（**CAN-H**と**CAN-L**）を使う差動信号方式。

- **ドミナント（論理0）**: CAN-H≈3.75V、CAN-L≈1.25V（電圧差あり）
- **リセッシブ（論理1）**: CAN-H≈CAN-L≈2.5V（同電位）

外部ノイズは両線に同等に影響するため、差分を取ることでノイズ耐性が極めて高い。バスの両端には**120Ωの終端抵抗**が必要。バスはANDゲートのように動作——いずれかのノードがドミナント（0）を送信すると、全ノードがドミナントを読む。

### 通信速度とケーブル長のトレードオフ

| ボーレート | 最大ケーブル長 |
|---|---|
| 125 kbps | 500m |
| 250 kbps | 250m |
| 500 kbps | 100m |
| 1 Mbps | 40m |

### データフレーム構造

7つのフィールドで構成：SOF（1ビット）→ アービトレーション（ID + RTR）→ コントロール（DLC、データ長0-8バイト）→ データ（最大8バイト）→ CRC（15ビット）→ ACK → EOF（7ビット）。

### アービトレーション——非破壊的な優先度制御

複数ノードが同時送信すると、IDの数値が**小さいほど優先度が高い**。各ノードは自分が送信したビットとバス上のビットを比較し、リセッシブ（1）を送ったのにドミナント（0）を読んだ場合、より高優先度のメッセージがあると判断して送信を中止しリスナーに切り替わる。**勝者のデータは損失なし**。敗者は次の機会に再送信。

### CAN 2.0A vs 2.0B vs CAN FD

| 項目 | CAN 2.0A | CAN 2.0B | CAN FD |
|---|---|---|---|
| ID長 | 11ビット | 29ビット | 11/29ビット |
| データ長 | 最大8バイト | 最大8バイト | 最大**64バイト** |
| データ速度 | 最大1Mbps | 最大1Mbps | 最大**8Mbps** |

CAN FD（2012年、Bosch開発）はアービトレーション部分を従来速度で行い、データ部分のみ高速に切り替える**BRS（Bit Rate Switch）**を搭載。後方互換性を維持しつつ性能を大幅向上。

### OBD-IIとの関係

OBD-II（On-Board Diagnostics II）はCANの**上位層プロトコル**。「CAN busは電話、OBD-IIは話す言語」。16ピンコネクタのピン6（CAN-H）とピン14（CAN-L）を使用。

### Eclipse SDV / KUKSA との接続

**KUKSA DBC Feeder**がCAN busの生データを読み取り、DBC（Database CAN）ファイルでシグナルを解釈し、VSS（Vehicle Signal Specification）に変換する。これがハードウェアとソフトウェアの「翻訳層」だ。

## 気づき・洞察

CAN busのアービトレーションは**非破壊的**——衝突が起きてもデータが失われない。これはEthernetのCSMA/CD（衝突検知→再送信）とは根本的に異なる。車のような安全が最優先の環境では、「衝突してもデータが生き残る」設計が不可欠だ。

差動信号の2本のケーブル（CAN-H/CAN-L）は、前回学んだ前腕の2本の骨（橈骨/尺骨）と構造的に同型だ。2つの要素の「差」が情報（信号/回転）を伝え、外部の干渉（ノイズ/衝撃）に対する耐性を確保する。

CAN busの8バイト制限→CAN FDの64バイト拡張は、SDVの進化における最初のボトルネック。しかしEclipse SDVではCAN busの上にKuksa Databrokerが乗り、さらにその上にuProtocolが乗る。物理層の制約はソフトウェア層で抽象化される——これがSDVの本質だ。

## 他分野との接続

- **解剖学（前腕）**: 差動信号の2本線と前腕の2本の骨の構造的類似。「差」が情報を伝える設計原理
- **テック（MCP）**: CAN busがECU間の通信標準なら、MCPはAIツール間の通信標準。どちらも「N×M→N+M」のインターフェース標準化
- **音楽（オーケストレーション）**: CAN busの複数ノードが1本のバスを共有＝複数の楽器が1つの空間を共有。アービトレーション＝指揮者による音量バランス調整
- **パン（酵母と乳酸菌）**: サワードウの共生関係＝CAN busの協調的メッセージング。限られたリソース（バス帯域/栄養素）を複数のアクターが共有
