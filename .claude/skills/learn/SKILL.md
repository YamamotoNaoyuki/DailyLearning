# 🌀 影分身・日次学習スキル

全分野（music, piano, golf, anatomy, house-hunting, deutsch, other）から**まだ学習していない新しいテーマ**を1つずつ選び、調査→記録→サマリー更新→統合レポート作成→HTML 化→コミット & プッシュ（GitHub Pages 公開）までを一気に実行する。

廃止済みドメイン（`domains/_archive/` 配下）: `tech` / `cs` / `bread` / `philosophy` / `sdv`。これらは学習対象から外れているので、Step 1〜4 で参照・更新しない。

## 追加指示
$ARGUMENTS

## 実行ステップ

### Step 1: カバレッジ台帳のロード（既習トピックの把握）
各 `domains/<分野>/_ledger.jsonl`（機械可読・1エントリ1行）と `domains/<分野>/_frontier.md`（未開拓軸の待ち行列）を読む。
- **`_ledger.jsonl` がこのドメインの既習トピックの唯一の照合先**。各行は `{date, title, slug, keywords[], depth, sources_tier, ...}`。60〜70行と軽量なので**全行読む**（800行ある `_summary.md` を選定のために精読する必要はない）。
- 台帳が存在しない/古いと感じたら、先に `python3 scripts/build_ledger.py <分野>` で再構築してから読む。
- **この台帳に載っているトピックはすべて既習**。以降のステップで「未着手／分野初／空白だった／初めて扱う」等と書くことを**固く禁ずる**（履歴未参照による事実誤認の宣言の禁止。過去の事故の主因）。

### Step 2: 新テーマの選定（フロンティア優先 → 重複ゲート → 深掘り判定）
各分野から1つずつ新トピックを決める。必ず次の 2-1 → 2-2 →（必要時）2-3 の順で通す。

#### 2-1. 候補生成（フロンティア最優先）
`_frontier.md` の待ち行列の**先頭から未消化（`[ ]`）の軸を最優先**で候補にする。フロンティアが枯渇しているドメインに限り、既習トピック集合から**意味的に最も遠い**（台帳 keywords との一致語が最小の）新テーマをモデルが生成する。「面白そう」だけで過去の定番に戻らない。

#### 2-2. 重複ゲート（台帳照合）
候補のキーワード集合を `_ledger.jsonl` の全 keywords と突き合わせる。
- **キーワードが3語以上一致、または同一トピッククラスタ（ハンディキャップ/ペダリング/肩/暗譜/タコ/地図投影/修繕積立金 等）に該当 → 「既習ヒット」**。
- ヒットしない候補があればそれを採用する。
- **クールダウン**: 同一トピッククラスタは**最低8週間（約56日）**あける。台帳に直近8週間以内の同クラスタ・エントリがあれば、たとえ深掘り条件を満たしても今日は選ばない（重複の時間的集中の防止）。

#### 2-3. 深掘り判定（再カバーを許す唯一の例外）
全候補が既習ヒットし、どうしても再訪する場合は、**次の3条件をすべて満たすときのみ**許可する。満たせなければ 2-1 のフロンティア候補へ差し戻す。
1. 先行エントリを本文冒頭で `[[タイトル]]` と**名指し参照**し、「何を新たに足すか」を1文で宣言する
2. **新レイヤーを最低1つ確約**する——一次文献／実証データ／数理定式化／反例・批判／別ドメインへの適用 のいずれか
3. そのトピックに既に「最深版」が存在しないこと（肩4本目・暗譜4本目・ペダリング5本目のような薄い積み増しは却下）

> デフォルトは「遠いテーマ」。**ユーザーが $ARGUMENTS で「ドメインXは昨日のテーマと関連したものを」等と指定した場合のみ**、該当ドメインは関連性重視で選定してよい（ただし 2-2 の重複ゲートとクールダウンは必ず通す）。具体的で掘り下げがいのあるトピックにする（広すぎるテーマは避ける）。

#### `other` 分野の特別ルール（セレンディピティ枠）
`other` は他の分野に当てはまらない**完全にランダムな分野・トピック**を選ぶ。
- まず分野をランダムに決める（例: 天文学、料理化学、言語学、心理学、映画、建築、数学、生物学、経済学、歴史、地政学、写真、園芸、陶芸、香水、ワイン、将棋、格闘技、海洋学、火山学…何でもよい）
- 次にその分野の中から具体的で面白いトピックを1つ選ぶ
- 過去の `other` エントリと同じ分野を避け、毎回異なるジャンルから選ぶこと
- 目的は「新しい興味の種を見つけること」。既存分野との意外な接続が見つかれば最高
- もし特定のトピックが継続的に興味を引くようになったら、新しい専門分野として独立させることを提案する

### Step 3: 調査と記録
各分野について、WebSearch 等で最新情報を調査し、`domains/<分野>/YYYY-MM-DD_タイトル.md` を作成する。
- **冪等性チェック**: 同一ドメイン・同一日（YYYY-MM-DD）に既存エントリファイルがあれば、そのドメインの新規作成を**停止**する（同日二重作成の防止。過去に anatomy で発生した事故の対策）。

#### ソース信頼性チェック
WebSearch の結果は**信頼性の高いソースのみ**から情報を採用すること。以下の基準で判定する：

**優先すべきソース（Tier 1）**:
- 学術論文・査読付きジャーナル（PubMed, Nature, Science, arXiv, Google Scholar 等）
- 公式ドキュメント・仕様書（RFC, W3C, 各プロジェクト公式サイト）
- 政府機関・公的機関のサイト（国交省、厚労省、FDA、WHO 等）
- 大学・研究機関の公開資料
- 業界の権威あるメディア（分野別: Grove Music Online for music, PGA.com for golf, JOSPT/PubMed for anatomy 等）

**許容するソース（Tier 2）**:
- 専門家が署名付きで書いた記事（著者の資格・所属が明記されている）
- 著名出版社の書籍・記事（O'Reilly, Cambridge UP, Springer 等）
- 評判の良い専門ブログ（長期運営、引用あり、著者が専門家）
- Wikipedia（概要の出発点としてのみ。詳細は引用元を追跡する）

**避けるべきソース（除外）**:
- 著者不明・匿名のブログ記事
- SEO目的のコンテンツファーム（薄い内容を大量生産しているサイト）
- AI生成が疑われる低品質な記事（内容が一般的すぎる、具体性がない）
- 明らかに古い情報（技術分野は2年以上前、法律分野は法改正前）
- 商品販売を主目的とするアフィリエイトサイト

**チェックプロセス**:
1. 検索結果の各ソースについて、ドメイン・著者・発行日を確認する
2. Tier 1 のソースを優先的に採用する
3. Tier 2 のソースは、Tier 1 と矛盾しない場合に補助的に使用する
4. 除外対象に該当するソースからの情報は使わない
5. 複数の独立したソースで裏付けが取れる情報を優先する（クロスチェック）
6. 各エントリの末尾に、主要な参考ソースを記載する

エントリはプロジェクトのテンプレート（`_templates/entry.md`）に従い、以下を必ず含む：
- 学んだこと（具体的で深い内容）
- 気づき・洞察（自分なりの解釈）
- 他分野との接続（横断的な視点）
- 次に深掘りしたいこと（→ Step 4 で `_frontier.md` の補充に使う）

また本文冒頭（タグ行の直後）に**新規性メタ行**を1行必須とする：
`> **この記録の新規性**: ___（何が過去エントリと違うのか）／ **関連既習**: [[既習タイトル]]（無ければ「なし」）`
2-3 の深掘り再訪の場合は、ここで先行エントリを名指しし、足した新レイヤー（一次文献/実証/数理/反例/別ドメイン適用）を明記する。

### Step 4: サマリー・台帳・フロンティアの更新
各ドメインについて、次の3ファイルを必ず更新する（重複の再発防止はこの台帳・フロンティア更新が要）。

**(1) `_summary.md`**（従来通り・人間用の累積知識）
- エントリ数を加算 / 最終更新日を今日に
- 新しく学んだ知識を「蓄積された知識」に追加（過去のものは消さない）
- 「未解決の疑問」を更新

**(2) `_ledger.jsonl`**（機械照合用・選定の唯一の照合先）
- 新エントリの1行を追記する。`python3 scripts/build_ledger.py <分野>` を実行すれば、本文のタグ行から keywords を自動抽出して追記される（**推奨。これを必ず走らせる**）。
- 追記後、その行の `depth`（`intro` か `deepening`）と `sources_tier`（`tier1`/`tier2`/`blog`）を調査の実態に合わせて記入する。

**(3) `_frontier.md`**（未開拓軸の待ち行列）
- 今日消化した軸を `[x]` にして「消化済み」へ移す
- 本文「次に深掘りしたいこと」から生まれた**新しい未開拓軸を待ち行列の末尾に補充**する（待ち行列を枯らさない）

### Step 5: 統合レポート作成
`_integrations/YYYY-MM-DD_デイリーレポート.md` を作成する。
- レポートタイトルは `# デイリーレポート YYYY-MM-DD`（絵文字や「影分身」は付けない）
- 冒頭のトピック一覧テーブルに各エントリへの GitHub リンクを付ける
  - リンク形式: `https://github.com/YamamotoNaoyuki/DailyLearning/blob/main/domains/<分野>/ファイル名.md`
- 各セクション見出しにも `[[詳細]](リンク)` を付ける
- トピック一覧テーブルの直後に **「今日のテーマ概要」** セクションを設け、各分野のテーマについて2〜3行のアブスト（何を学び、何が面白かったか）を記載する。箇条書きで、分野名を太字にする
- **蓄積サマリー**セクションをテーマ概要の後に配置し、以下を含める：
  - 各分野のエントリ数と総エントリ数
  - 学習開始日（最初のエントリの日付）からの経過日数
  - 各分野でカバーしてきた主要領域の概要（例: musicなら「バロック→古典派→ロマン派→現代」のように学習の軌跡を簡潔に）
  - deutsch分野は蓄積フレーズ数も記載
- 全分野を横断する「今日の横断的パターン」セクションで深い洞察を提示する
- 「意外な接続点」「統合的洞察」「次のアクション」セクションを続ける（「次のアクション」は `### music` のように分野キーごとの小見出しにする）
- 末尾に `*Generated by 知識統合システム*` を付ける
- **このレポートは次の Step で HTML に機械変換される**。セクション見出し（「今日のトピック一覧」「今日のテーマ概要」「蓄積サマリー」「今日の横断的パターン」「意外な接続点」「統合的洞察」「次のアクション」）と、トピック一覧の Markdown テーブル、テーマ概要の `- **分野**: …` 形式は崩さないこと

### Step 6: レポートを HTML 化（自己完結ファイル生成）
```
python3 scripts/build_report_html.py "$DATE"
```
これで `docs/reports/YYYY-MM-DD.html`（統合レポート＋全10ドメインの全文を1ファイルに埋め込んだ自己完結 HTML。各ドメインはクリックで開く折りたたみカード）と `docs/index.html`（レポート一覧）が生成・更新される。`$DATE` は Step 全体で使っている `date +%Y-%m-%d` の値。ローカルの `domains/` 配下の .md は一切変更しない（ドメイン別 Markdown のまま）。
なお `data/x/<DATE>.json`（`scripts/fetch_x.py` が `daily_run.sh` の中で当日朝に取得する、気になる人のXの直近投稿）が存在すれば、`build_report_html.py` がレポート上部に「今日のX」セクションとして自動で埋め込む。存在しなければそのセクションは出ないだけ。`data/x/` は `.gitignore` 済み（埋め込み済みの HTML だけが公開される）。

### Step 7: Git コミット & プッシュ
新しいエントリ、更新したサマリー、**更新した `_ledger.jsonl`・`_frontier.md`**、統合レポート（.md）、および `docs/`（生成された HTML）を git add → commit → push する。
コミットメッセージ例: `🌀 日次学習 YYYY-MM-DD: テーマ1, テーマ2, ...`
push 後、レポートは GitHub Pages で公開される：`https://yamamotonaoyuki.github.io/DailyLearning/reports/YYYY-MM-DD.html`（一覧: `https://yamamotonaoyuki.github.io/DailyLearning/`）。この URL をユーザーに共有する。

## 注意事項
- **ネタ被り防止が最優先の運用ルール**: テーマ選定は必ず Step 1（台帳ロード）→ Step 2（フロンティア優先・重複ゲート・8週間クールダウン・深掘り3条件）の順で通す。`_ledger.jsonl` に載るトピックを「未着手/分野初/空白」と書くのは禁止。記録後は Step 4 で `build_ledger.py` を走らせて台帳を更新し、`_frontier.md` の待ち行列を消化・補充する。再訪は「新レイヤー（一次文献/実証/数理/反例/別ドメイン適用）を足す深掘り」のときだけ許される。
- 今日の日付は `date +%Y-%m-%d` コマンドで取得すること（システムの currentDate やコンテキスト上の日付ではなく、実行時のシステム日付を使う）。エントリファイル名・レポートファイル名・レポートタイトル・HTML 生成スクリプトの引数・コミットメッセージすべてにこの日付を使用する
- `music` 分野はクラシック音楽の鑑賞・分析・歴史に特化すること
- `piano` 分野はピアノ演奏の身体技法・奏法・練習法に特化すること（music分野との重複を避ける）
- `golf` 分野はコースマネジメント、クラブ選択、戦略、ルール、メンタルゲーム、プロツアーの分析など「ゴルフというゲーム」の理解に焦点を当てること。身体の使い方・バイオメカニクスは解剖学分野に統合済み
- `anatomy` 分野は日常動作・ピアノ演奏・ゴルフスイングなどの身体最適化を解剖学的に扱うこと。golf/piano分野と連携する横断的な視点を重視する
- `deutsch` 分野はtelc B1取得済みのドイツ語学習者向けの毎日フレーズ学習に特化すること。毎回異なるテーマ（雑談・職場・意見・感情・不満・予定・食事・ニュース・買い物・旅行・健康・趣味・天気・お金・引っ越し・近所付き合い等）を選び、「教科書的でない、ネイティブが実際に使う」フレーズを4つ生成する。各フレーズには以下を含めること：
  - 日本語訳
  - 単語リスト（主要単語の意味）
  - 使う場面（具体的なシチュエーション）
  - フォーマル度（formal / neutral / umgangssprachlich / slang の4段階）
  - キーワード解説（格変化・前置詞の注記あり、文法ポイント）
  - 関連表現（同じ意味の別の言い方、バリエーション）
- `deutsch` 分野のエントリ形式は他分野と異なる。テンプレートの「学んだこと」セクションにフレーズ4つを記載し、「気づき・洞察」に文法的な気づきやドイツ語特有の表現パターン、「他分野との接続」にドイツ文化・哲学・音楽等との接点を記載する
- `deutsch` 分野のサマリーには、蓄積されたフレーズ数、カバー済みテーマ、重要文法パターンを記録する
- 各エントリは読み応えのある深さにすること（箇条書きの羅列ではなく、ストーリーのある内容）
- 他分野との接続は表面的なこじつけではなく、本質的な共通点を見出すこと
